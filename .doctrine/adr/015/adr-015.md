# ADR-015: Multi-dimensional priority scoring

## Context

Doctrine's `survey` and `next` surfaces rank work by actionability and a raw
inbound-reference count named `consequence`. That count is structurally honest but
semantically weak: an item referenced by five low-value ideas outranks, or ties,
one that gates a smaller number of high-value slices. IMP-118 asks for priority to
reflect the authored value, estimate, and risk facets already present in the
corpus, plus operator-tunable coefficients.

This decision is project-wide rather than slice-local. Once adopted, `score`
becomes a reusable ordering primitive for priority surfaces, explain output, and
future graph views. The policy therefore belongs in an ADR; SL-133 implements it.

Adjacent doctrine constrains the shape:

- ADR-001 requires risk parsing to be available below command-tier callers; the
  implementation must not make a leaf or engine module import `backlog`.
- ADR-004 forbids storing reverse/derived relation data in authored entity files;
  priority consequence is derived from the outbound relation graph at read time.
- The pure/imperative split requires scoring to be pure over scanned authored data,
  config, and an assembled graph; disk access remains at the scan/config seams.

## Decision

We will replace raw inbound-reference consequence with a multi-dimensional
floating-point priority **score**:

```text
base(N)        = value_dim(N) + risk_dim(N)
consequence(N) = leverage(N) + optionality(N)
score(N)       = base(N) + consequence(N)
```

### 1. Base dimensions

`value_dim` is the authored value adjusted by the global value coefficient, kind
and tag coefficients, divided by the estimate midpoint:

```text
value_dim = (coefficients.value × value × kind_weight × Σ tag_coefficients) / estimate_midpoint
```

`risk_dim` is the authored risk exposure adjusted by a risk coefficient:

```text
risk_dim = exposure × risk_coeff
```

Absent authored data collapses to an identity/floor rather than an error:

- absent value ⇒ `value_dim = 0`;
- absent estimate ⇒ `estimate_midpoint = 1.0`;
- absent risk facet ⇒ `risk_dim = 0`;
- absent tags / tag storage not yet shipped ⇒ tag contribution is the identity.

The risk model is the existing likelihood × impact exposure model; this ADR does
not create a new risk vocabulary.

### 2. Two-pass scoring model

Scoring has two pure phases around the existing graph build:

1. **Base pre-pass** — compute each node's base score from its own authored facets,
   kind, and priority config. This phase has no graph access and is safe to use as
   the graph mint tiebreaker.
2. **Consequence post-pass** — after the graph is assembled, compute consequence
   from graph edges and already-computed base scores.

The graph's structural mint order must not use graph-derived consequence. Mint uses
`base` only; display surfaces may sort by the final `score` after the graph exists.

### 3. Consequence classes

Consequence has two edge classes with different semantics.

**`needs` leverage is recursive.** If `A needs B`, then completing `B` unlocks `A`;
therefore `B` accrues a coefficient-weighted share of `A`'s base value and the
value recursively unlocked by `A`. `dep_coeff` is a recursive retention factor in
`[0, 1]`, where `0` (and any value clamped from `≤ 0`) explicitly DISABLES leverage;
above `0`, along a single path depth decays as `dep_coeff^k`.

Leverage is intentionally a finite path-sum over the finite needs graph. In a
reconvergent diamond, the shared downstream node contributes through each path.
`dep_coeff <= 1` gives per-path retention; it does not bound total magnitude under
fan-out. Large path-sums are visible through explain output rather than silently
reinterpreted as deduplicated reachability.

A diagnosed `needs` cycle is malformed authored data. The scoring read must still
terminate: cycle components are condensed for scoring, intra-component needs edges
contribute zero, and the cycle remains surfaced as an authoring error.

**Reference/lineage optionality is one-hop.** Work/lineage references contribute a
flat optionality share from direct referencers only. These overlays are
cyclic-capable and associative, so recursive accumulation over them is not part of
this policy. `ref_coeff` is a flat non-negative weight.

**`after` / seq is structural, not a score weight.** Sequence edges order work; they
are not a third consequence dimension and have no coefficient. If later evidence
shows structural sequencing is insufficient, a future ADR or amendment may add a
separate seq-weight policy.

### 4. Config contract

Doctrine owns a `[priority]` config section with these policy-level roles:

```toml
[priority.coefficients]
# value, risk

[priority.kind_weights]
# kind-name -> multiplier

[priority.tag_coefficients]
# tag -> multiplier

[priority.consequence]
# dep_coeff, ref_coeff
```

Unknown keys are tolerated for forward compatibility. Numeric default values,
example weights, maximum clamp constants, and the exact malformed-value recovery
mechanics are implementation-owned. The durable domains are policy:

- value/risk/kind/tag coefficients do not invert priority;
- `dep_coeff` is a recursive retention factor in `[0, 1]`, with `0` (and any clamped
  `≤ 0`) the explicit disable sentinel;
- `ref_coeff` is flat and non-negative;
- seq has no coefficient.

Coefficient and tag tuning is an expected, supported UX for shaping priority order
to a particular purpose. Changing global value-vs-risk weights, kind weights, tag
coefficients, or the tags authored on items changes the derived score immediately;
it is configuration/data authoring, not a code change and not an ADR amendment. The
system therefore promises cheap retuning plus explainability, not a single stable
priority sequence independent of the active config and tags.

### 5. Sort and explain contract

`score` is first-class node data for priority surfaces.

- `survey` orders by `actionability → score desc → canonical-id asc`.
- `next` shows the actionable frontier. Wherever surviving seq precedence is silent,
  it orders by `score desc → canonical-id asc`; along surviving seq chains, structure
  wins. The implementation must not replay seq edges that cordage evicted to make the
  structural order acyclic.
- `explain` exposes the score breakdown: base, value dimension, risk dimension,
  recursive leverage, one-hop optionality, and total.

No score or reverse consequence is authored to disk. Scores are derived in memory
from authored facets, config, and outbound relations.

Rejected alternatives:

- **Keep raw inbound count.** Simple but blind to value, estimate, and risk — the
  defect IMP-118 exists to fix.
- **One-hop consequence for all edge classes.** Safe, but misses deep blockers whose
  value appears beyond one dependent.
- **Recursive consequence for all relation overlays.** Treats cyclic-capable lineage
  and associative references as causal dependency chains; termination and semantics are
  wrong.
- **Use score in graph mint order.** Creates feedback: graph-derived data would affect
  the structural order of the graph that produces it.
- **Make seq a score clamp/weight now.** More complex and tie-prone; current policy is
  structural precedence first, weighted ordering only where structure is silent.

## Consequences

### Positive

- Priority surfaces rank by the value and risk of what an item unlocks, not by a raw
  edge count.
- Deep blockers become visible through recursive needs leverage.
- Score is explainable: large numbers decompose into base, leverage, and optionality.
- The mint/display split prevents scoring feedback loops while still making display
  order score-aware.
- Config and tag edits can deliberately retune ordering for a purpose without code
  changes, while durable coefficient domains keep the policy coherent.

### Negative

- Scores are floating-point values; implementation must enforce deterministic ordering
  and keep non-finite values away from sort keys and render output.
- Recursive path-sum leverage can grow sharply in wide reconvergent graphs. This is
  accepted semantics, but it can surprise operators until `explain` is consulted.
- `next` needs an explicit frontier sort over surviving seq constraints; plain cordage
  `order_key` is insufficient because level precedes the mint tiebreaker.
- Golden output churn is expected: `survey`, `next`, and `explain` change from inbound
  count to score columns/breakdowns.

### Neutral

- Numeric defaults are not governance. They may change as the corpus teaches better
  weights, without amending this ADR.
- Priority order is purpose-sensitive: changing coefficients or tags is expected to
  change `survey` / `next` ordering.
- Tag coefficients are part of the formula now, but tag storage can arrive later; until
  then, the tag term contributes its identity.
- No per-item priority override is introduced. Overrides, cost/maintainability
  dimensions, and score history remain future work.

## Verification

- Base-score tests cover value-only, risk-only, combined, absent facets, estimate
  identity, kind-weight defaults, and tag identity.
- Consequence tests prove direction and class semantics: needs leverage uses dependent
  edges recursively; reference optionality uses only the configured work/lineage label
  subset one hop; bookkeeping labels contribute zero.
- Recursive leverage tests cover a chain, a reconvergent diamond, and a diagnosed needs
  cycle with condensation and finite output.
- Sort tests prove: graph mint uses base only; `survey` uses score; `next` orders
  seq-incomparable actionable items by score while preserving surviving seq precedence
  and ignoring evicted seq edges.
- Determinism tests prove total ordering for equal scores and no NaN/∞ reaches mint,
  surface sorting, JSON, or human render.
- Explain tests assert the full `{base, value_dim, risk_dim, leverage, optionality,
  total}` breakdown so retuned rankings remain diagnosable.
- ADR-001 layering gate stays green: risk types live below command-tier consumers and
  priority config/scoring do not import command modules upward.
- ADR-004 is preserved: no authored reverse/consequence fields are added; consequence
  remains derived from outbound relations.
