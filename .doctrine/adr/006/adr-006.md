# ADR-006: Worktree posture: policy-agnostic framework, orchestrator-sole-writer dispatch

## Context

Doctrine is a *framework* consumed by other projects, not only its own repo.
Parallel agent execution (`/dispatch`) needs git working-tree isolation so
concurrent workers do not stomp each other. Three forces constrain the design:

1. **Framework neutrality.** Branch/worktree policy is the consuming project's
   choice. Doctrine must work equally for a solo developer on trunk with trivial
   merges and a team on branches with PR review and real conflicts. Doctrine
   therefore owns the *merge/fork behaviour of its own artifacts*, not the
   project's git policy. When in doubt, framework utility outranks the build
   repo's local convenience.

2. **Doctrine's storage tiers already pre-adapt.** Authored (committed) /
   runtime (gitignored, disposable) / derived. Runtime phase state being
   gitignored makes it per-worktree by construction; no central registry or
   counter exists, so authored entities are disjoint across slices.

3. **The hazards are concurrency, not storage.** Sequential `SL-NNN` ids can
   collide across concurrent creation; memory git-anchors can point at doomed
   branches; `notes.md` is shared-mutable *within* a slice; gitignored runtime
   state is invisible across worktrees.

Prior art: the spec-driver `dispatch` skill (orchestrator-sole-writer remit) and
the superpowers worktree skills (detect-don't-fight via `GIT_DIR != GIT_COMMON`).
Neither solves doctrine's tier-aware concerns; both inform the mechanics.

## Decision

- **D1 — Policy-agnostic posture.** We will detect isolation
  (`GIT_DIR != GIT_COMMON`, with the submodule guard) rather than prescribe it.
  Defaults work out of the box trunk-based; one or two alternates
  (worktree-per-slice, worktree-per-batch) are cheap; deeper change is by forking
  editable skills/templates. No configuration Rube Goldberg.

- **D2 — Worker-sole-writer invariant (load-bearing).** Under parallel dispatch,
  worktree workers mutate **source code only**. *All* doctrine-mediated writes —
  authored tiers, runtime phase state, memory, id minting, commits — funnel
  through the **orchestrator on trunk**. Workers may **read** doctrine state
  freely (it is in the fork) but **write** none of it; they return structured
  reports. This single contract collapses the notes-collision,
  cross-worktree-progress-invisibility, global-tier-coordination, and
  memory-anchoring hazards into "the orchestrator writes on trunk."
  - **D2a — Enforcement, in scope.** A CLI worker-mode guard (`DOCTRINE_WORKER=1`)
    hard-refuses every doctrine-mediated authored write (`memory record`,
    `slice/adr/spec new`, status transitions, doctrine-driven commits). Doctrine
    owns its CLI, so this is enforceable. It covers exactly the writes that mint
    ids and anchor memory.
  - **D2b — Residual gap, deferred.** Raw-tree confinement (a worker
    hand-editing a file or running bare `git commit`) is not CLI-stoppable, and
    the harness does **not** confine workers to their worktree (observed: Claude
    Code creates the worktree but lets the agent write to main). This is a known
    live risk, deferred to sandbox/harness work (see ADR-B), not papered over.

- **D3 — Minting is a trunk-side act.** Ids are allocated against the trunk ref
  (auto-detected `origin/HEAD` then `main`/`master`, overridable) *before* a
  worktree forks. Cross-branch offline collisions are caught by detect+reseat in
  `validate` plus a reseat verb.

- **D4 — Tier merge-safety invariant.** No shared-mutable authored file across
  the unit of isolation; runtime state stays gitignored/per-worktree; the
  `phases` symlink is relative. This invariant is *defended*, not assumed — any
  future central index reintroduces conflict.

- **D5 — Worktree creation is in-scope for skills.** `/dispatch` (mandatory
  isolation) and `/execute` (optional). Doctrine delegates the *mechanism* to the
  harness / native tools (never reinventing `git worktree`) and adds only two
  guards: **commit-before-spawn** (a fork sees only committed HEAD) and the
  **branch-point check** (HEAD pre/post-spawn mismatch → re-dispatch).

- **D6 — Orchestrator pre-distills worker context.** Workers receive a
  self-contained prompt (policy digest, design excerpts, pre-fetched memories,
  task spec, mandatory verification command). Workers do not read boot/governance
  or run `/boot`, so "governance frozen at fork" and "hooks-not-inherited" are
  moot for workers.

- **D7 — Funnel discipline.** The orchestrator persists incrementally (per
  batch), in strict order **merge → verify → commit → record knowledge**. Trunk
  is the durable store; orchestrator context is disposable; crash/overflow
  recovery is rebuild-from-trunk plus `git worktree list`. Knowledge always
  trails confirmed code.

- **D8 — Team placement.** The orchestrator may run on a delta-branch rather than
  main; ids are minted against trunk pre-branch; globals "land on main at merge"
  (convention, not mechanism) — a memory anchored to a delta-branch commit keeps
  its sha through a clean merge.

The single-tree, turn-based coordination regime (design↔review, audit, consult)
is the orthogonal primitive; it is specified in ADR-007. The shared boundary rule
both ADRs enforce: **the unit of isolation determines the primitive — never a
live shared ledger across worktrees.**

## Consequences

### Positive

- Collapses four concurrency hazards at once via the funnel (D2).
- The solo trunk-based path is untouched (no worktree required); teams get
  isolation without doctrine dictating branch flow (D1).
- The funnel enables memory dedup — the orchestrator sees all worker reports
  before recording.
- Because all durable state lives on trunk, the orchestrator's context is
  disposable: crash ≡ handover ≡ resume-from-trunk (D7).

### Negative

- The worker-sole-writer invariant has **no harness enforcement** (D2b); until
  raw-tree confinement is solved it rests on the CLI guard (D2a) plus the prompt
  contract.
- Genuine code coupling across parallel batches can still merge-conflict; policy
  is report, never auto-resolve.
- A new CLI worker-mode and trunk-ref/reseat machinery are required.

### Neutral

- Project-local concerns (bubblewrap jail, `CARGO_TARGET_DIR`,
  install-vs-installed templates, cargo bin races) are out of scope and belong to
  ADR-B.
- Two distinct worktree-creation paths exist (dispatch via harness isolation;
  solo `/execute` via native tools) and are named, not unified.

## Verification

- `validate` flags duplicate ids (detect half of D3); a reseat verb resolves them.
- The CLI worker-mode guard (D2a) is covered by tests asserting that
  doctrine-mediated writes refuse under `DOCTRINE_WORKER=1`.
- The branch-point check (D5) is an orchestrator-side assertion: HEAD captured
  pre-spawn must equal the worktree HEAD post-spawn.
- The tier merge-safety invariant (D4) is checked by the absence of any
  shared-mutable authored file and by the `phases` symlink being relative.
