# ADR-006: Worktree posture: policy-agnostic framework, orchestrator-sole-writer dispatch

## Context

Doctrine is a *framework* consumed by other projects, not only its own repo.
Parallel agent execution (`/dispatch`) needs git working-tree isolation so
concurrent workers do not stomp each other. Three forces constrain the design:

1. **Framework neutrality.** Branch/worktree policy is the consuming project's
   choice. Doctrine must work equally for a solo developer on trunk with trivial
   merges and a team on branches with PR review and real conflicts. Doctrine
   therefore owns the *merge/fork behaviour of its own artifacts*, not the
   project's git policy. When in doubt, framework utility outranks the build
   repo's local convenience.

2. **Doctrine's storage tiers already pre-adapt.** Authored (committed) /
   runtime (gitignored, disposable) / derived. Runtime phase state being
   gitignored makes it per-worktree by construction; no central registry or
   counter exists, so authored entities are disjoint across slices.

3. **The hazards are concurrency, not storage.** Sequential `SL-NNN` ids can
   collide across concurrent creation; memory git-anchors can point at doomed
   branches; `notes.md` is shared-mutable *within* a slice; gitignored runtime
   state is invisible across worktrees.

Prior art: the spec-driver `dispatch` skill (orchestrator-sole-writer remit) and
the superpowers `using-git-worktrees` skill (detect-don't-fight via
`GIT_DIR != GIT_COMMON`, plus the native→git-fallback→in-place creation ladder and
regenerate-then-baseline setup). The Claude Code `WorktreeCreate`/`WorktreeRemove`
hooks fed by a `.worktreeinclude` allowlist supply native fork provisioning. None
solve doctrine's tier-aware concerns; all inform the mechanics (D5/D9).

## Decision

- **D1 — Policy-agnostic posture.** We will detect isolation
  (`GIT_DIR != GIT_COMMON`, with the submodule guard) rather than prescribe it.
  Defaults work out of the box trunk-based; one or two alternates
  (worktree-per-slice, worktree-per-batch) are cheap; deeper change is by forking
  editable skills/templates. No configuration Rube Goldberg.

- **D2 — Worker-sole-writer invariant (load-bearing).** Under parallel dispatch,
  worktree workers mutate **source code only**. *All* doctrine-mediated writes —
  authored tiers, runtime phase state, memory, id minting, commits — funnel
  through the **orchestrator on the coordination branch** (see D8: usually trunk
  in solo mode, a delta branch in team mode). Workers may **read** doctrine state
  freely — its authored and provisioned tiers are in the fork (the
  coordination/runtime tier is withheld, D9) — but **write** none of it. A worker returns a
  **structured report plus a source delta** — a branch/worktree diff or patch —
  not merely a description of changes; it never commits doctrine state. This
  single contract collapses the notes-collision,
  cross-worktree-progress-invisibility, global-tier-coordination, and
  memory-anchoring hazards into "the orchestrator writes on the coordination
  branch."
  - **D2a — Enforcement, in scope.** A CLI worker-mode guard hard-refuses every
    doctrine-mediated authored write (`memory record`, `slice/adr/spec new`, status
    transitions, doctrine-driven commits) — plus the `Orchestrator` verb class
    (`fork`/`import`/`land`/`gc`, which mutate git refs/dirs). Doctrine owns its CLI,
    so this is enforceable. It covers exactly the writes that mint ids and anchor
    memory.
    - **Signal (amended SL-056 G2; re-amended SL-056 PHASE-05): disk-marker-primary,
      positive worker signal.** The worker signal is **`worker_mode =
      (is_linked_worktree && marker_present) OR env DOCTRINE_WORKER`**; the disk marker
      is the harness-agnostic primary, the env leg a codex/pi optimisation (claude has
      no env channel). The marker is the **positive** worker identity: **marker present
      in a linked worktree ⇒ refuse** (a confirmed worker); **marker absent ⇒ NOT
      worker-mode** — a solo `/execute` or the orchestrator may write (D6a: *mode, not
      location, decides*). The legit orchestrator runs at the coordination root
      (`!is_linked_worktree`); a solo agent in a worktree writes its own doctrine state
      directly. The sole marker-minting verb (`marker --stamp-subagent`) is exempt **by
      verb identity** — it writes the marker into its own linked-worktree `cwd`.
      - **The fence against an unstamped claude worker is the funnel + the jail, not a
        fail-closed CLI floor.** A worker whose SubagentStart stamp hook failed runs
        `marker_present=false` and is therefore *not* refused at the write seam. Its
        blast radius is bounded elsewhere: the **R-5 import belt** rejects any
        `.doctrine/` path in the `B..S` delta (an unstamped worker writing into its own
        fork is caught at import, report+halt); the **env worker-on-main catch**; the
        **pre-distilled prompt**; and — decisively for the build repo — the
        **bubblewrap jail with no push access** (ADR-008), which caps the worst case at
        *lost unpushed progress*. Stamp-failure is kept **non-silent** by an
        **orchestrator post-spawn marker check that aborts an unstamped fork** (IMP-052;
        the orchestrator *can* abort where the read-only SubagentStart hook cannot —
        ADR-011 D6), not by refusing the write.
      - **Why the G2 fail-closed clause was reverted (SL-056 PHASE-05, owner-locked
        VH).** G2 made marker-absent fail-CLOSED. That (a) contradicted **D6a**
        verbatim — D6a lets a solo writer write in a worktree; fail-closed refused it —
        and (b) killed solo-`/execute`-in-a-worktree direct-writing. Re-weighed against
        the *actual* risk — `P(SubagentStart hook failure) ≈ 0` (the hook blocks; a
        miss needs a crash / broken binary / disk error mid-stamp) **×** *jail-bounded*
        harm (no push ⇒ worst case is lost local work) — the floor bought negligible
        security over the funnel while costing a real workflow and a live D2a/D6a
        contradiction. The marker is the positive signal; the funnel + jail + IMP-052
        are the fence. (Codex-reviewed adversarially; see `worker-mode-floor-decision.md`
        §6.)
  - **D2b — Residual gap, deferred.** Raw-tree confinement (a worker
    hand-editing a file or running bare `git commit`) is not CLI-stoppable, and
    the harness does **not** confine workers to their worktree (observed: Claude
    Code creates the worktree but lets the agent write to main). This is a known
    live risk, deferred to sandbox/harness work (see ADR-008), not papered over.

- **D3 — Minting is a trunk-side act.** Ids are allocated against the configured
  **trunk ref** (the id baseline; auto-detected `origin/HEAD` then `main`/
  `master`, overridable) *before* a worktree forks. Cross-branch offline
  collisions are caught by detect+reseat in `validate` plus a reseat verb.

- **D4 — Tier merge-safety invariant.** No shared-mutable authored file across
  the unit of isolation; runtime state stays gitignored/per-worktree; the
  `phases` symlink is relative. This invariant is *defended*, not assumed — any
  future central index reintroduces conflict.

- **D5 — Worktree creation is in-scope for skills.** `/dispatch` (mandatory
  isolation) and `/execute` (optional). Doctrine *prefers* the harness's native
  mechanism and reinvents `git worktree` only at the fallback rung when no native
  tool exists (the creation ladder, D9 — battle-tested by superpowers
  `using-git-worktrees`). It adds two guards: **commit-before-spawn** (a fork sees
  only committed HEAD) and the **branch-point check** (HEAD pre/post-spawn mismatch
  → re-dispatch).

- **D6 — Orchestrator pre-distills worker context.** Workers receive a
  self-contained prompt (policy digest, design excerpts, pre-fetched memories,
  task spec, mandatory verification command). Workers do not read boot/governance
  or run `/boot`, so "governance frozen at fork" and "hooks-not-inherited" are
  moot for workers.

- **D6a — Worker vs solo `/execute`.** A *worker* is a thin code-mutator under an
  orchestrator: worker-mode is ON (D2a), it returns a source delta, and all its
  doctrine-mediated writes funnel. Solo `/execute` is a *full agent and its own
  orchestrator*: worker-mode is OFF, and it writes doctrine state directly (its
  own slice status, notes, phase, memory). The worktree is merely *where* each
  runs; the **mode, not the location, decides who may write.**

- **D7 — Funnel discipline.** The orchestrator persists incrementally (per
  batch), in strict order **import delta → verify → commit → record knowledge**:
  it imports each worker's source delta, verifies it, commits from the
  coordination branch, then records derived knowledge (memory, AC evidence,
  notes). The coordination branch is the durable store; orchestrator context is
  disposable; crash/overflow recovery is rebuild-from-coordination-branch plus
  `git worktree list`. Knowledge always trails confirmed code.

- **D8 — Coordination branch and placement.** The orchestrator runs on a
  **coordination branch** — usually trunk in solo mode, a delta branch in team
  mode. Ids are still minted against the configured **trunk ref** (the id
  baseline, D3) before a worktree forks; globals "land on main at merge"
  (convention, not mechanism). A memory anchored to a coordination-branch commit
  keeps its sha through a *non-squash* merge (see the squash caveat in
  Consequences).

- **D9 — Fork provisioning and the creation ladder.** Worktree creation follows a
  **preference ladder**: detect existing isolation (D1); else prefer the harness's
  native worktree tool (Claude Code: the `WorktreeCreate`/`WorktreeRemove` hooks);
  else fall back to `git worktree add`; else, on sandbox denial, degrade to
  **work-in-place, solo mode, no funnel** (the already-blessed D1/D6a trunk path).
  A fresh fork carries only committed files, so before dispatch the orchestrator
  **provisions** it along two axes: **regenerate** derivable prerequisites from
  committed sources (`cargo build`, `npm install`) and **copy** only the
  *irreducible* gitignored prerequisites (secrets, local config) via a
  **project-owned allowlist** (`.worktreeinclude`, gitignore syntax — D1
  policy-agnostic; Claude Code feeds it through the `WorktreeCreate` hook).
  **Invariant (guards D2):** the allowlist **excludes the coordination/runtime
  tier** (`.doctrine/state/`, the `phases` symlink, `handover.md`, memory caches)
  by construction — their *absence* in the fork is what makes worker-sole-writer
  free; a copied phase sheet would be invisibly mutable. D6's pre-distilled prompt
  substitutes for the withheld coordination state; provisioning substitutes for the
  absent execution env. Finally the orchestrator **baseline-verifies** the fork
  builds and tests green *before* handing the worker its task — an unbuildable fork
  is fixed in provisioning, never dispatched. (Doctrine's jail instantiates the
  regenerate axis with a per-worktree `CARGO_TARGET_DIR` — ADR-008 D-B1.)

  - **D9 amendment (SL-056 G2) — the claude rung is SubagentStart-stamp, and its
    baseline-verify is a weaker class.** The O3 spike (SL-056 PHASE-02) found the
    deployed Claude Code `WorktreeCreate` payload carries no `agent_type`,
    `worktree_path`, or base — so the create-replacing `create-fork` hook (fail-closed:
    provision+stamp *inside* creation) is **not buildable**, and is deferred until the
    payload grows those fields. The claude rung is therefore: **Claude performs its own
    default worktree creation** at `Agent isolation:worktree`; a **matcher-scoped,
    sync-blocking `SubagentStart` hook** (`matcher: dispatch-worker`) then provisions
    (the `.worktreeinclude` axis above) and stamps the marker into the payload `cwd`.
    Because SubagentStart blocks the worker until it exits, provision+stamp complete
    before the worker's first command **on hook success**.
    - **The pre-dispatch baseline-verify guarantee does NOT hold for claude (accepted
      weaker class).** Claude creates the worktree *at spawn*; there is no pre-spawn
      orchestrator moment, and SubagentStart is a read-only event that cannot abort the
      worker on a failed baseline. So a claude worker **may start on an unverified
      fork**; an unbuildable fork is caught **late, at the funnel's `import → verify`
      step** (cost: a wasted worker run — multiplied by batch width under the
      one-landing-per-base regime, SL-056 §7c). This is a deliberate altitude
      concession, named here, not papered over; codex/pi keep the pre-dispatch gate
      (the orchestrator owns `fork`, so it verifies before handoff).

The single-tree, turn-based coordination regime (design↔review, audit, consult)
is the orthogonal primitive; it is specified in ADR-007. The shared boundary rule
both ADRs enforce: **the unit of isolation determines the primitive — never a
live shared ledger across worktrees.**

## Consequences

### Positive

- Collapses four concurrency hazards at once via the funnel (D2).
- The solo trunk-based path is untouched (no worktree required); teams get
  isolation without doctrine dictating branch flow (D1).
- The funnel enables memory dedup — the orchestrator sees all worker reports
  before recording.
- Because all durable state lives on the coordination branch, the orchestrator's
  context is disposable: crash ≡ handover ≡ resume-from-coordination-branch (D7).

### Negative

- The worker-sole-writer invariant has **no harness enforcement** (D2b); until
  raw-tree confinement is solved it rests on the CLI guard (D2a) plus the prompt
  contract.
- Genuine code coupling across parallel batches can still merge-conflict; policy
  is report, never auto-resolve.
- **Solo `/execute` inside a worktree** writes memory directly (no funnel to
  reroute it, D6a); a memory anchored to that branch's sha is **orphaned by a
  squash-merge** — the content survives, but SL-008 staleness fires. Plain
  (non-squash) merges are unaffected.
- A new CLI worker-mode and trunk-ref/reseat machinery are required.

### Neutral

- Project-local concerns (bubblewrap jail, `CARGO_TARGET_DIR`,
  install-vs-installed templates, cargo bin races) are out of scope and belong to
  ADR-008.
- Two distinct worktree-creation paths exist (dispatch via harness isolation;
  solo `/execute` via native tools) and are named, not unified.

## Verification

- `validate` flags duplicate ids (detect half of D3); a reseat verb resolves them.
- The CLI worker-mode guard (D2a) is covered by tests asserting that
  doctrine-mediated writes refuse under `DOCTRINE_WORKER=1`.
- The branch-point check (D5) is an orchestrator-side assertion: HEAD captured
  pre-spawn must equal the worktree HEAD post-spawn.
- The tier merge-safety invariant (D4) is checked by the absence of any
  shared-mutable authored file and by the `phases` symlink being relative.
- `doctrine memory record` detects worktree context (`GIT_DIR != GIT_COMMON`) and
  warns when recording on a branch (squash-orphan risk), nudging record-on-trunk.
- Fork provisioning (D9): the project allowlist (`.worktreeinclude`) is asserted to
  **exclude** the coordination/runtime globs (`.doctrine/state/`, `phases`,
  `handover.md`, memory caches); and a baseline build+test must pass in the fork
  before dispatch (an unbuildable fork blocks dispatch, not the worker).

## Open

- **Anchor stability (deferred seam).** If squash-orphaning of solo-in-worktree
  memory (Negative) proves common, move the memory anchor from the volatile
  branch sha to a stable ref (merge-base or content-hash). Touches SL-007/008;
  deferred until evidence warrants.
- **Convention (unenforced for solo):** record durable memory on the coordination
  branch / trunk, not in a transient worktree branch — the same rule the dispatch
  funnel enforces for workers.
