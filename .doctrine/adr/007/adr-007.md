# ADR-007: Adversarial review as a first-class kind with turn-based ledger coordination

## Context

Doctrine needs **adversarial review at every lifecycle stage** — scope, design,
plan, phase-plan, phase implementation (code review and drift audit), slice-level
code review, audit, and reconciliation-to-spec. Today these are ad hoc:
`audit.md` is hand-made (a known scaffold gap), inquisition is informal, and code
review is unstructured.

Multi-agent coordination here is a **different regime** from dispatch (ADR-006).
Dispatch is *spatial* parallelism across isolated worktrees, solved by the
orchestrator funnel. Review is *temporal* alternation between a **raiser** (finds)
and a **responder** (dispositions), in a **single working tree**, over a shared
structured ledger. These are two coordination primitives, one per concurrency
axis; ADR-006 defines the first, this ADR the second.

In a single tree there is no git merge to catch a concurrent write, so a naive
shared ledger suffers silent last-write-wins clobber — worse than a conflict. The
coordination state (whose turn) must be an explicit, mediated token, and
concurrent invocations must be serialized by a real lock (D-C4 / D-C4a).

## Decision

- **D-C0 — Review is a first-class kind.** We will add a `review` kind with
  prefix **`RV`** (`RV-NNN`), not a slice-attached artifact. One generic ledger
  mechanism, parameterized by `facet`, reviews any subject (a slice and facet, a
  phase, or a spec) via the outbound edge `RV-NNN ──reviews──▶ ref`. Every review
  skill (`/inquisition`, `/audit`, `/code-review`, reconciliation) instantiates
  this kind. This closes the audit-scaffold gap structurally and is DRY across the
  lifecycle. (Multi-artefact **drift** is *not* an RV facet — see D-C11.)

- **D-C1 — Baton in runtime state.** The `await` token lives in runtime state
  (gitignored, per-worktree), never as an authored field. Rationale: the storage
  rule (transient coordination is not authored), clean git history (handoffs are
  invisible to git), and an auto-enforced single-tree boundary (see D-C7).

  *Clarification (SL-040): for the pilot, review verbs operate on the **main/parent
  tree**, where parent ≡ worktree and D-C7's single-tree boundary holds directly.
  Fork-invoked review (the locus would resolve to a fork that cannot see the parent
  baton — `WITHHELD` excludes `.doctrine/state/**` from forks) is **IMP-024** and
  not yet supported: review verbs are guarded to refuse a fork-resolved root.
  Reconciling "per-worktree" with a parent-locus addressing rule is deferred to
  IMP-024.*

- **D-C2 — Authored ledger is truth; baton is a regenerable cache.** `await` is a
  pure function of finding statuses; the runtime baton caches it and holds
  non-derivable bookkeeping. On loss it is recomputed from the authored ledger.

- **D-C3 — Ordering.** A turn writes authored content **first** and flips the
  baton **last**. A crash mid-turn leaves an un-flipped, resumable turn; authored
  writes are idempotent (findings carry immutable ids).

- **D-C4 — CLI-mediated turn-taking.** The CLI knows the closed status enum,
  computes `await`, refuses out-of-turn writes (role asserted via `--as`), and
  makes the two-file (authored-then-baton) write atomic within one invocation.

  *Clarification (SL-040): "out-of-turn" is enforced **per finding** via the
  transition graph (each edge single-owner, asserted by `--as`). `await` (D-C8) is
  a derived **summary** for status display and handoff routing — not an independent
  gate. A raiser may `raise` a new finding even while `await=responder`
  (append-only, raiser-owned); the single-owner edges (`dispose`/`verify`/`contest`/
  `withdraw`) are gated by the per-finding predicate.*

- **D-C4a — Runtime lock / CAS.** Every review mutation acquires an exclusive
  per-review runtime lock *before* it reads `await`, and releases it only after
  the authored write, derived recomputation, and baton write all complete. If the
  authored ledger changed since the read, the command aborts and asks the caller
  to re-run. The baton is the **turn** protocol; the lock is the **concurrency**
  primitive — neither substitutes for the other. ("Atomic within one invocation"
  (D-C4) does not by itself serialize *concurrent* invocations; D-C4a does.)

- **D-C5 — Schema and finding mutability.** *Finding identity* is append-only:
  findings are never deleted or renumbered, and `id` / `title` / `detail` /
  original `severity` are raiser-owned and fixed. *Lifecycle fields* mutate
  through CLI-mediated transitions — the responder owns `disposition` /
  `response`; the raiser owns the verify / contest / withdraw transitions. Field
  ownership keeps keys disjoint. Severity is reviewer-idiomatic:
  `blocker | major | minor | nit`.

- **D-C6 — Prose companion.** `review-nnn.md` carries a `## Brief` seeded at
  creation for the raiser (suggested pre-reading, context bootstrap, lines of
  attack) and an optional `## Synthesis` (the raiser's human-readable overall
  commentary).

- **D-C7 — Single-tree boundary.** The review loop is turn-based in one working
  tree; there is never a live shared ledger across worktrees. The runtime baton
  is per-worktree, so it auto-enforces this; authored-ledger git conflicts are the
  backstop. Cross-worktree coordination uses the ADR-006 funnel. This is the
  shared boundary rule with ADR-006.

- **D-C8 — Derived status function.** The review's overall status derives from
  finding statuses together with `await`, never hand-edited:

  ```text
  empty ledger                  → status = active, await = raiser   (raiser goes first)
  any open/answered/contested   → status = active, await = derived role
  all verified/withdrawn        → status = done,   await = none
  ```

  where the awaited role is: any `open`/`contested` ⇒ responder; else any
  `answered` ⇒ raiser. The empty ledger is `active` (not vacuously `done`), so an
  implementation can never mistake "no findings yet" for completion. This makes
  the kind born free of the status-rollup divergence SL-009 surfaces for slices.

- **D-C9 — Lifecycle teeth (two distinct gates).**
  - *(a) Review completion.* A review is `done` only when **every** finding is
    terminal (`verified` or `withdrawn`), regardless of severity.
  - *(b) Target closure.* `/close` on the subject refuses while any **`blocker`**
    finding in an active review targeting that subject is unresolved.

  Review-done is the strict per-review rule; severity gates whether the *target*
  may close. One status rule no longer does two jobs.

- **D-C10 — Warm-cache: uniform, self-scaling reviewer-context cache.** Every RV
  carries a runtime-tier cache of the reviewer's *learned* model of the subject —
  a `domain_map` (area → purpose → explored paths), plus invariants and
  risk areas — populated on a `prime` step and reused across review rounds. It is
  a **distillation** the reviewer authors, not the inference-layer prompt/token
  cache: the prompt cache is ephemeral (single context, minutes) and dies across
  the cross-round, cross-agent handoff that defines an adversarial review, whereas
  this cache is durable and drift-keyed and survives it. (Prior art: spec-driver
  `supekku:workflow.review-index@v1`, schema description "reviewer bootstrap cache
  for amortizing review setup cost".)

  It is **facet-agnostic and self-scaling**, not gated to a code-bound subset:
  every facet runs a multi-round loop that can emit changes needing re-check, and
  most facets explore code/artefacts to some depth (design at a high level; plan /
  phase-plan heavily unless no code exists; scope sometimes; reconciliation across
  code *and* adjacent artefacts). A thin review caches a thin map; a code-heavy one
  caches a rich one. One uniform mechanism, consistent with D-C8's one-mechanism
  ethos.

  **Staleness keys on content-hashes of the explored path set** (the `domain_map`
  paths), *not* bare `{phase, head}`. A bare git-head key is blind to drift that
  does not move HEAD — gitignored or uncommitted artefacts (e.g. `notes.md`,
  `inquisition.md`) — exactly the surface several facets read; content-hashing
  covers code, uncommitted, and gitignored paths uniformly. `{phase, head}` may
  survive only as a coarse fast-path. *Design consideration (deferred to slice
  design, not resolved here):* the cache is per-worktree runtime state and
  worktree provisioning copies some gitignored tiers while excluding others
  (ADR-006 coordination-tier exclusion), so the explored path set — and which
  paths are present and hashable — differs between the main tree and a fork; the
  content-hash staleness model must be worktree-aware.

  Lives beside the baton (runtime, gitignored, regenerable — D-C1/D-C2 tier).

- **D-C11 — Single-ref subject; multi-artefact drift is a distinct kind.** The
  `[target].ref` (single canonical id + optional `phase`) is sufficient for every
  RV facet, because each is a **single-subject** review: one subject, findings
  about it. Multi-artefact **drift reconciliation** — one entry spanning N typed
  sources / affected artefacts (prior art: spec-driver's separate `drift/` domain,
  "tracking divergence between normative and observed truth", `DriftEntry` with
  `sources[]` / `affected_artifacts[]`) — is structurally a *different* model and
  is **not** an RV facet. It is carved out as a distinct future **Drift Ledger**
  kind (backlogged). This keeps the single-ref thesis true by construction and
  avoids an ad-hoc multi-ref addressing grammar in RV.

### Schema shape

```toml
# review-NNN.toml — authored. Findings append-only; await/status derived, not stored.
[review]
facet     = "design"   # closed enum: scope|design|plan|phase-plan|implementation|code-review|reconciliation
raiser    = "reviewer"
responder = "author"

[target]                # outbound relation (ADR-004): RV-NNN ──reviews──▶ ref
ref   = "SL-024"        # single canonical id of the subject (reconciliation: the spec, e.g. PRD-010)
phase = "PHASE-03"      # optional scoped qualifier — phase-scoped facets only

[[finding]]
id          = "F-1"        # raiser-owned, fixed; bare doc-local id, append-only
status      = "verified"   # open|answered|contested|verified|withdrawn
severity    = "major"      # raiser-owned, fixed
title       = "..."        # raiser-owned, fixed
detail      = "..."        # raiser-owned, fixed
disposition = "fixed"      # responder-owned, mutable
response    = "..."        # responder-owned, mutable
```

`review-NNN.md`: `## Brief` (seeded for the raiser) + optional `## Synthesis`.
The single `ref` (plus optional `phase`) avoids multiple nullable addressing
fields that could be invalid in combination.

Transition graph (each edge is single-owner; status function in D-C8):

```
  (none) --raise[raiser]--> open --dispose[responder]--> answered --verify[raiser]--> verified*
                              ^                              |
                              +------ contest[raiser] -------+
   contested --dispose[responder]--> answered
   open|answered --withdraw[raiser]--> withdrawn*
```

## Consequences

### Positive

- One mechanism for all adversarial review across the lifecycle; closes the
  audit-scaffold gap with lifecycle teeth (D-C0, D-C9).
- Orthogonal to and composes with ADR-006: two coordination primitives (funnel =
  spatial, ledger = temporal) sharing one boundary rule (D-C7).
- Derived status (D-C8) means the new kind never accrues the slice
  status-divergence problem.
- CLI turn-mediation plus the per-review lock (D-C4/D-C4a) eliminates the silent
  lost-update inherent to a single-tree shared file.

### Negative

- `--as <role>` is self-asserted — cooperative, not a security boundary (the same
  model as ADR-006 D2a). It stops accidental out-of-turn writes, not malicious
  ones.
- A new kind requires install wiring (manifest dir plus gitignore negation), id
  allocation, the outbound relation edge, render/`show`, the `review` verb family,
  the runtime baton + lock/CAS, and the D-C10 warm-cache (`prime` + content-hash
  staleness). Non-trivial build.
- The close-gate (D-C9b) needs a **reverse** lookup over the outbound `reviews`
  edge ("active reviews targeting this ref with an unresolved `blocker`"). No
  general reverse-relation index exists today (relations are outbound-only,
  ADR-004); the gate is a corpus scan over RV `[target].ref` — feasible, but built,
  not reused.

### Neutral

- The single-tree constraint means a raiser cannot be fanned across worktrees
  within one review; a very large review is a funnel job feeding findings into the
  ledger — a future seam, noted not solved.
- Handoff chatter (a "why I handed back" note) stays in the runtime baton;
  anything durable is promoted to a finding/disposition in the authored ledger.

## Verification

- `review status` recomputes `await`/status from finding statuses and rebuilds the
  baton — covered by tests asserting the status function (D-C8) is total over the
  status enum, including the empty-ledger and all-terminal cases.
- The CLI turn guard (D-C4) is covered by tests asserting out-of-turn writes
  refuse and that each verb writes authored-first then baton-last.
- The per-review lock/CAS (D-C4a) is covered by a concurrent-invocation test:
  one writer wins; the other aborts-and-retries (no clobber).
- The close-gate (D-C9b) is covered by `/close` refusing while any `blocker` is
  unresolved; review completion (D-C9a) by `done` requiring all findings terminal.
- The single-tree boundary (D-C7) is verified by the baton residing in
  per-worktree runtime state.

## Resolved (from review)

- **Facet vocabulary:** **closed** enum (above); extend only by ADR. Uncontrolled
  facets would fragment status/render behaviour.
- **Relation verb:** **`reviews`** (reads `RV-007 reviews SL-024`).
- **Round capping:** **uncapped**; add observability (`rounds` in the runtime
  baton, per-finding contest tracking) rather than a policy limit. Human
  escalation can come later.
- **Non-slice targets:** one generic **`ref`** grammar now (single canonical id +
  optional `phase`); reconciliation targets the spec via `ref`. No ad hoc
  addressing mini-language.

### Resolved (round 2 — IMP-001 preflight, 2026-06-11)

Reconciling the prior art in `scratch/rv-review-prior-art.md` against this ADR
before acceptance surfaced three amendments (added as D-C10/D-C11 above):

- **Warm-cache restored as build intent (D-C10).** The original draft dropped
  spec-driver's warm-cache (a `## Brief` author-once prose replaced it). On
  review this was prior-art blindness, not a reasoned cut: the cache amortizes
  the cross-round, cross-agent re-exploration cost that defines an adversarial
  loop, which the ephemeral prompt cache cannot. Restored as uniform,
  self-scaling, content-hash-keyed (not `{phase, head}`).
- **`drift` dropped from the facet enum (D-C11).** Multi-artefact drift is
  structurally a different model (N sources / affected artefacts per entry) and
  becomes a distinct future **Drift Ledger** kind. The single-`ref` thesis holds
  for all remaining RV facets by construction.
- **Reverse-relation dependency named** (Consequences/Negative) — the D-C9b
  close-gate is a corpus scan; no reverse index exists yet.
