# ADR-007: Adversarial review as a first-class kind with turn-based ledger coordination

## Context

Doctrine needs **adversarial review at every lifecycle stage** — scope, design,
plan, phase-plan, phase implementation (code review and drift audit), slice-level
code review, audit, and reconciliation-to-spec. Today these are ad hoc:
`audit.md` is hand-made (a known scaffold gap), inquisition is informal, and code
review is unstructured.

Multi-agent coordination here is a **different regime** from dispatch (ADR-006).
Dispatch is *spatial* parallelism across isolated worktrees, solved by the
orchestrator funnel. Review is *temporal* alternation between a **raiser** (finds)
and a **responder** (dispositions), in a **single working tree**, over a shared
structured ledger. These are two coordination primitives, one per concurrency
axis; ADR-006 defines the first, this ADR the second.

In a single tree there is no git merge to catch a concurrent write, so a naive
shared ledger suffers silent last-write-wins clobber — worse than a conflict. The
coordination state (whose turn) must therefore be an explicit, mediated token.

## Decision

- **D-C0 — Review is a first-class kind.** We will add a `review` kind with
  prefix **`RV`** (`RV-NNN`), not a slice-attached artifact. One generic ledger
  mechanism, parameterized by `facet`, targets any subject (a slice and facet, a
  phase, or a spec). Every review skill (`/inquisition`, `/audit`,
  `/code-review`, drift, reconciliation) instantiates this kind. This closes the
  audit-scaffold gap structurally and is DRY across the lifecycle.

- **D-C1 — Baton in runtime state.** The `await` token lives in runtime state
  (gitignored, per-worktree), never as an authored field. Rationale: the storage
  rule (transient coordination is not authored), clean git history (handoffs are
  invisible to git), and an auto-enforced single-tree boundary (see D-C7).

- **D-C2 — Authored ledger is truth; baton is a regenerable cache.** `await` is a
  pure function of finding statuses; the runtime baton caches it and holds
  non-derivable bookkeeping. On loss it is recomputed from the authored ledger.

- **D-C3 — Ordering.** A turn writes authored content **first** and flips the
  baton **last**. A crash mid-turn leaves an un-flipped, resumable turn; authored
  writes are idempotent (findings carry immutable ids).

- **D-C4 — CLI-mediated turn-taking.** The CLI knows the closed status enum,
  computes `await`, refuses out-of-turn writes (role asserted via `--as`), and
  makes the two-file (authored-then-baton) write atomic within one invocation.

- **D-C5 — Schema.** Findings are append-only with immutable bare ids (`F-N`).
  Field ownership keeps keys disjoint: the raiser owns `severity`/`title`/
  `detail`; the responder owns `disposition`/`response`. The `status` enum drives
  the turn. Severity is reviewer-idiomatic: `blocker | major | minor | nit`, with
  `blocker` as the close-gate.

- **D-C6 — Prose companion.** `review-nnn.md` carries a `## Brief` seeded at
  creation for the raiser (suggested pre-reading, context bootstrap, lines of
  attack) and an optional `## Synthesis` (the raiser's human-readable overall
  commentary).

- **D-C7 — Single-tree boundary.** The review loop is turn-based in one working
  tree; there is never a live shared ledger across worktrees. The runtime baton
  is per-worktree, so it auto-enforces this; authored-ledger git conflicts are the
  backstop. Cross-worktree coordination uses the ADR-006 funnel. This is the
  shared boundary rule with ADR-006.

- **D-C8 — Derived entity status.** The review's overall status (`active` /
  `done`) derives from the turn function rather than being hand-edited, so the
  kind is born free of the status-rollup divergence SL-009 surfaces for slices.

- **D-C9 — Lifecycle teeth.** `done` requires no `open`/`answered`/`contested`
  finding; `/close` calls `review status` and refuses unresolved `blocker`
  findings.

### Schema shape

```toml
# review-NNN.toml — authored. Findings append-only; await derived, not stored.
[review]
facet     = "design"   # scope|design|plan|phase-plan|implementation|code-review|drift|reconciliation
raiser    = "reviewer"
responder = "author"

[target]                # outbound relation (ADR-004): review ──targets──▶ subject
subject = "SL-024"
phase   = "PHASE-03"    # when facet is phase-scoped
spec    = "PRD-010"     # when facet = reconciliation

[[finding]]
id = "F-1"; status = "verified"; severity = "major"
title = "..."; detail = "..."            # raiser-owned
disposition = "fixed"; response = "..."  # responder-owned
```

`review-NNN.md`: `## Brief` (seeded for the raiser) + optional `## Synthesis`.

Turn function over finding statuses:

```
any finding open OR contested  → await = responder
else any finding answered      → await = raiser
else (all verified/withdrawn)  → done
empty ledger                   → await = raiser   (raiser goes first)
```

Transition graph (each edge is single-owner):

```
  (none) --raise[raiser]--> open --dispose[responder]--> answered --verify[raiser]--> verified*
                              ^                              |
                              +------ contest[raiser] -------+
   contested --dispose[responder]--> answered
   open|answered --withdraw[raiser]--> withdrawn*
```

## Consequences

### Positive

- One mechanism for all adversarial review across the lifecycle; closes the
  audit-scaffold gap with lifecycle teeth (D-C0, D-C9).
- Orthogonal to and composes with ADR-006: two coordination primitives (funnel =
  spatial, ledger = temporal) sharing one boundary rule (D-C7).
- Derived status (D-C8) means the new kind never accrues the slice
  status-divergence problem.
- CLI turn-mediation eliminates the silent lost-update inherent to a single-tree
  shared file (D-C4).

### Negative

- `--as <role>` is self-asserted — cooperative, not a security boundary (the same
  model as ADR-006 D2a). It stops accidental out-of-turn writes, not malicious
  ones.
- A new kind requires install wiring (manifest dir plus gitignore negation), id
  allocation, the outbound relation edge, render/`show`, and the `review` verb
  family. Non-trivial build.

### Neutral

- The single-tree constraint means a raiser cannot be fanned across worktrees
  within one review; a very large review is a funnel job feeding findings into the
  ledger — a future seam, noted not solved.
- Handoff chatter (a "why I handed back" note) stays in the runtime baton;
  anything durable is promoted to a finding/disposition in the authored ledger.

## Verification

- `review status` recomputes `await` from finding statuses and rebuilds the baton
  — covered by tests asserting the turn function is total over the status enum.
- The CLI turn guard (D-C4) is covered by tests asserting out-of-turn writes
  refuse and that each verb writes authored-first then baton-last.
- The close-gate (D-C9) is covered by `/close` refusing while any `blocker` is
  unresolved.
- The single-tree boundary (D-C7) is verified by the baton residing in
  per-worktree runtime state.
