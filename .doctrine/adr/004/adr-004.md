# ADR-004: Relations stored outbound-only; reciprocity is derived

## Context

Doctrine's entities reference each other: a spec members its requirements, a tech
spec interacts with another spec, a knowledge record shapes a slice or a risk, an
ADR supersedes an ADR. Two product specs reach the same fork independently and
record it as the same open question:

- **PRD-009 OQ-004** — "Are backlog↔artefact relations reciprocal (an item shows
  inbound references from slices/specs/drift), or outbound-only in the durable
  item with reverse lookups deferred to a registry surface?"
- **PRD-010 OQ-003** — the identical question for record↔artefact relations,
  noting it "Mirrors PRD-009 OQ-004."

Both say the answer "blocks the inspect view's completeness claims." This ADR
settles the storage direction once, for the whole entity corpus, so each spec
inherits it rather than re-deciding.

`spec-driver` settled the same question in its **ADR-002** (do not store backlinks
in frontmatter): forward references are the single source of truth, backlinks are
derived data computed at registry sync. Doctrine reaches the same conclusion, but
the argument is **stronger here** because doctrine's design supplies reinforcements
`spec-driver`'s flat frontmatter could not:

- **The storage rule makes a stored backlink a tier violation, not just a smell.**
  Backlinks are derived data. Doctrine separates authored (committed TOML+MD) from
  derived (regenerable, gitignored). Authoring a backlink into a committed TOML
  puts derived data in the authored tier — a direct breach of the storage rule
  ("never queried/derived data in prose," generalised: derived data does not live
  in the authored tier).
- **The derived reverse surface already exists.** `src/registry.rs` scans the
  entity trees into canonical-string id sets + an edge list for `spec validate`. A
  reverse index is an inversion of that edge list — an extension of an existing
  scan, not new machinery (`mem.system.spec.composition-seam` §4).
- **Edges already carry edge-local data.** A spec's `members.toml` row holds a
  `label` (`FR-001` / `NF-001`) that is **mobile** — the same requirement membered
  by two specs carries a different label in each. The label lives on the edge, so
  membership *cannot* be re-expressed as a backlink on the requirement without
  losing per-edge data or duplicating it (`mem.system.spec.composition-seam` §2).
  The requirement entity stores no inbound spec references today; this ADR keeps
  it that way.
- **The materialiser is non-transactional.** `spec req add` is already a two-tree
  write that can orphan on partial failure. A maintained backlink makes it a
  three-tree write and multiplies the partial-write and merge-conflict surface
  (`mem.system.spec.composition-seam` §2).

Nothing stores backlinks today, so this is a forward commitment, not a migration.
The one apparent exception — ADR supersession writing `superseded_by` onto the
predecessor — is reconciled below, not contradicted.

## Decision

**Relations are stored outbound-only on the durable entity; reciprocity is
realised by derivation, not by storing the reverse edge.**

### 1. One canonical authoring side per relation type

Each relation type has exactly one entity that authors the edge:

- spec → requirement membership is authored in the spec's `members.toml`.
- tech spec → spec interaction is authored in the source spec's
  `interactions.toml`.
- knowledge record → artefact (slice / spec / risk / ADR / requirement / drift) is
  authored **on the record** (PRD-010's responsibilities already say "Expose
  *outbound* relations from a record").
- backlog item → artefact is authored on the backlog item.

The target side authors nothing for that relation. Letting both ends author a
forward edge for one relationship recreates the dual-source-of-truth problem this
ADR forbids.

### 2. Reciprocity is derived, not stored

A relation is reciprocal to a *reader* — both directions are navigable — but the
reverse direction is **computed** by the registry scan, never authored on the
target. "Are relations reciprocal?" therefore answers *yes* (navigable) and *no*
(not stored on both ends) at once; the two are not in tension.

### 3. Inbound completeness belongs to the registry-backed surface

The sync-free one-way reader (`spec show` — pure stdout reassembly from an
entity's own files, "cannot go stale," `mem.system.spec.composition-seam` §5) shows
**outbound** relations only. Inbound references are a property of the
**registry-backed surface** (`inspect` / a future survey), which computes them from
the scan. A completeness claim about inbound references is the registry surface's
to make, never the plain reader's.

### 4. Reverse lookup is correct-but-uncached until the feature DAG lands

The registry recomputes the edge list per invocation; the persistent cache is
deferred to the feature DAG (`mem.system.spec.composition-seam` §4). Reverse lookup
is therefore correct and always fresh, merely uncached — acceptable at the current
corpus size. "Uncached" is not "unresolved": the contract is the derived reverse
view, independent of whether its cache is warm (cf. `spec-driver` ADR-009, which
holds the API contract — not the loading strategy — is what must be stable).

### 5. Carve-out: store a reverse edge only when the target is mutated anyway

A reverse edge MAY be co-written on the target entity **only when that entity is
already being mutated for an independent lifecycle reason**. The canonical case is
supersession: when ADR-B supersedes ADR-A, the supersede verb flips ADR-A to a
terminal `superseded` status — ADR-A's file is rewritten regardless. Co-writing
`superseded_by = [B]` there adds **zero** marginal coupling and lets a reader of
the dead record find its successor without a scan. Doctrine's own ADR-002 TOML
already encodes exactly this (`superseded_by` "set on the old ADR by F1; reverse
link is derived").

This is a bounded exception, not a loophole. It is permitted only when **all** hold:

- the relation marks a lifecycle transition on the target (supersession, not mere
  reference);
- the target's file is being rewritten anyway for that transition;
- the reverse edge is the *only* honest place a reader of the terminal target can
  learn its continuation.

Pure navigational backlinks (which specs member this requirement, which records
shape this slice) satisfy none of these — the target is not being mutated — so they
stay derived.

## Consequences

### Positive

- Settles PRD-009 OQ-004 and PRD-010 OQ-003 together with one rule the rest of the
  corpus inherits, instead of per-spec re-litigation.
- Keeps authored TOML free of derived data, honouring the storage rule's tiering.
- Keeps `spec req add` and peers from growing a third write target; preserves the
  one-way reader's "cannot go stale" property.
- Gives the registry scan a single, well-defined extra job (edge-list inversion)
  rather than scattering reverse state across entity files.
- Reconciles the existing supersession behaviour as a principled carve-out rather
  than an inconsistency.

### Negative

- Inbound references require the registry surface to be invoked; the plain reader
  cannot answer "what points at me." This is by design but is a capability the
  reader visibly lacks.
- Reverse lookup pays a scan per call until the feature-DAG cache lands.
- The carve-out is a judgement call at each new relation type: authors must check
  the three conditions rather than apply a flat "never store reverse" rule.

### Neutral

- No migration: nothing stores backlinks today.
- This ADR governs storage direction, not the registry's internal caching strategy
  or the exact `inspect` surface, which remain implementation/spec concerns.
- It does not settle PRD-010 OQ-005 (scope as a field vs a relation), though it
  leans that question toward relation-not-field: if scope is a typed relation, the
  artefact's "what is scoped to me" is derived like any other inbound reference and
  a free-text `scope` field is redundant.

## Verification

- Entity TOML schemas carry outbound edges only; no `*_by` / `membered_by` /
  `shaped_by` inbound field is added to a target schema (the supersession carve-out
  in §5 is the sole sanctioned reverse field, and only on the lifecycle-mutated
  target).
- `spec show` and any sync-free reader render outbound relations only; inbound
  references appear solely through the registry-backed `inspect`/survey surface.
- `spec validate` / the registry scan is the single producer of reverse lookups;
  a new relation type adds its edge to the scan, not a backlink to the target.
- A reviewer rejects any new authored field that stores the reverse of an
  already-authored forward edge unless it meets all three §5 conditions.
