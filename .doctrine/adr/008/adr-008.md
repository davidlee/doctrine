# ADR-008: Project-local jail build isolation and worker confinement for parallel dispatch

## Context

This ADR is **project-local to doctrine-the-repo** — how doctrine dogfoods the
parallel worktree dispatch defined by ADR-006 inside its own bubblewrap jail. It
is downstream of ADR-006 and prescribes nothing for consuming projects.

It also **discharges ADR-006 D2b** (raw-tree confinement), which D2b explicitly
deferred to "sandbox/harness work (ADR-008)". A generic harness cannot confine a
worker to its worktree (observed: Claude Code creates the worktree but lets the
agent write to main). Doctrine's jail *is* a sandbox, so doctrine can solve here
what the framework layer cannot.

Project-local facts that force the decisions:

- `CARGO_TARGET_DIR=/home/david/.cargo/doctrine-target-jail` is a **single shared
  absolute path** (flake.nix). Two worktrees on different branches sharing it
  cargo-lock-serialize *and* thrash — each branch's source invalidates the
  other's fingerprints.
- `~/.cargo/bin/doctrine` is **read-only** in the jail, so `cargo install` is
  impossible in-jail regardless.
- `CARGO_BIN_EXE` is baked at compile time; e2e tests spawn
  `$CARGO_TARGET_DIR/debug/doctrine` (a shared target gives a stale/foreign
  binary — the known stale-`CARGO_BIN_EXE` spawn-fail).
- justfile recipes are env-driven (`cargo build/test/install`), no hardcoded
  paths.

**Empirical evidence (research §5.1, hardened by SL-050/051 dispatch experience).**
The shared-target hazard is not theoretical — the dispatch work surfaced a
**false-green / false-red cluster** that the per-worktree target (D-B1) is the
clean fix for:

- **Shared-target false-green** — two branches sharing one `CARGO_TARGET_DIR`
  cargo-lock-serialize *and* cross-invalidate fingerprints, so a verify can pass
  against another branch's stale artefacts; the mitigation today is a touch +
  re-run ritual ([[mem.pattern.dispatch.shared-target-false-green-touch-rerun]]).
- **Worktree-removal false-red** — `env!(CARGO_MANIFEST_DIR)`-baked test binaries
  point at a removed worktree's path and spawn-fail until recompiled, a false RED
  ([[mem.pattern.dispatch.worktree-removal-stale-manifest-dir-false-red]]); the
  general stale-`CARGO_BIN_EXE` spawn-fail is the same class
  ([[mem.pattern.testing.stale-cargo-bin-exe]]).
- **Debug-vs-release scale timing** — cliff-test bounds must budget ~10× the
  release probe numbers under a debug build
  ([[mem.pattern.testing.debug-vs-release-scale-timing]]).

**Isolation is a perf / false-green concern, not a trust signal** (SL-056 §3): the
agnostic worker-sole-writer guarantee rides the **disk marker + the `import`
`.doctrine/`-rejection belt**, not the build sandbox. So D-B1 is justified *on its
own merits* (parallelism + correct e2e binaries), and where a per-worktree target
is unavailable — claude shares the jail-wide target (SL-056 §8) — the §5.1
mitigation rituals **stand**; nothing about trust regresses.

## Decision

- **D-B1 — Per-worktree `CARGO_TARGET_DIR`.** Each worktree gets its own target
  dir, nested under the jail-redirect root and keyed by worktree/branch
  (`…/doctrine-target-jail/wt/<branch>`), **set at worker spawn** rather than
  baked in the flake. This restores real build parallelism, eliminates
  cross-branch fingerprint thrash, and gives each worktree a correct
  `CARGO_BIN_EXE`. Cost: one full target dir per concurrent worktree (disk) plus
  cold builds.

- **D-B2 — No `cargo install` mid-dispatch (standing structural fact).** The flake
  **ro-binds `~/.cargo/bin/doctrine`**, so an in-jail `cargo install` is not a
  policy we choose to avoid — it is **structurally impossible**, full stop.
  Therefore **there is no install race because there is no install**: this is a
  standing property of the jail, not a mitigation that could regress. Workers and
  their tests run the per-worktree **build-target** binary
  ([[mem.pattern.build.jail-target-redirect]]); the ro `~/.cargo/bin/doctrine`
  remains the stable entrypoint.

- **D-B3 — Per-worker bwrap confinement (spike-first; codex/pi-only).** The OS-floor
  hardening that discharges ADR-006 D2b *where a subprocess exists to wrap*: run each
  worker in a **nested bwrap** that **rw-mounts only its worktree and its target dir,
  ro everything else (including main)**, so a worker *physically cannot* write to
  main — OS-enforced, not prompt-enforced. It **ro-binds the marker only — never
  `.claude/settings.local.json`** (which must stay writable for the harness's
  permission-grant persistence, SL-056 §9). This is **codex/pi-only**: claude's
  `Agent` tool is not a subprocess and cannot be wrapped (SL-056 §4c), so its
  worker-sole-writer stays **accident-fenced + prompt-enforced** until a free
  channel or OS confinement lands.
  **The spike's gating question is feasibility, not cost (OQ-2):** can an
  **unprivileged user namespace be created inside the jail** at all? The outer bwrap
  may seccomp-block `clone(CLONE_NEWUSER)`, in which case no amount of budget buys a
  nested bwrap. The one-liner empirical probe is `bwrap --unshare-user --ro-bind /
  / true` run **in-jail** (SL-056 PHASE-12). If userns creation works → land the
  `dispatch-worker` profile. If it is blocked or too costly → **back out to D-B1 +
  the ADR-006 D2a CLI guard** (the marker-primary guard, now the *agnostic* trust
  floor — SL-056 §3/§5) and leave D2b formally deferred, recorded symmetrically with
  the claude SubagentStart fail-open back-out. D-B3 depends on D-B1 (the confined
  worker needs its target dir rw).

- **D-B4 — `sccache` deferred (noted).** A shared compilation cache would offset
  D-B1's per-worktree cold-build disk/time cost. Deferred until cold builds
  actually cause pain; recorded here so the option is not rediscovered.

- **D-B5 — Keep the flake minimal.** Per-worktree env is set at worker spawn, not
  baked; the justfile is unchanged (already env-driven). The only added surface is
  a `dispatch-worker` bwrap profile (D-B3).

## Consequences

### Positive

- doctrine-the-repo becomes a **reference implementation** of safe parallel
  worktree dispatch, and (if D-B3 lands) of *true* worker confinement — proving
  ADR-006 D2b is solvable wherever a sandbox exists.
- Per-worktree targets (D-B1) make parallel build/test genuinely concurrent
  instead of cargo-lock-serialized, with correct e2e binaries.

### Negative

- Disk grows with concurrent worktrees (N target dirs); a worktree cap or D-B4
  may be needed under disk pressure.
- D-B3's nested bwrap adds plumbing; the spike may fail and force the weaker
  fallback (D-B1 + D2a), leaving D2b deferred.

### Neutral

- Nothing here is exercised until `/dispatch` is implemented and run on doctrine
  itself; this ADR is governance-ahead-of-need so the eventual implementation
  slice inherits the decisions.
- Host-vs-jail mount-path separation is preserved: per-worktree dirs nest under
  the existing jail-redirect root.

## Verification

- D-B1: two parallel worktree builds run without cargo-lock contention, each
  producing a binary its own e2e tests spawn successfully (no stale-`CARGO_BIN_EXE`).
- D-B3 feasibility (OQ-2): the in-jail userns probe `bwrap --unshare-user --ro-bind
  / / true` records a clear land/back-out decision (SL-056 PHASE-12). If landed: a
  worker process, given a write to a path outside its worktree (e.g. main's
  `.doctrine`), is OS-denied, and the confined worker cannot `rm` its ro-bound
  marker. If backed out: D2b's deferral is recorded, never silently dropped.
- D-B2: dispatch performs no `cargo install` (structurally — ro `~/.cargo/bin`);
  the build-target binary is the one under test.
