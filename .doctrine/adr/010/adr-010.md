# ADR-010: Relation modelling: unify the contract and write seam, keep storage bespoke

## Context

Doctrine's entities reference each other; the *storage* of those relations is
bespoke per kind, surveyed during SL-046 design and code-verified for IMP-034:

- **slice** (`src/slice.rs:1077`) — `Relationships{specs, requirements, supersedes}`,
  all `Vec<String>`, parsed-inert.
- **governance** (`src/governance.rs:156`) — `Relationships{supersedes,
  superseded_by, related, tags}`, `Vec<String>`, parsed-inert. The stored
  `superseded_by` is the **ADR-004 §5 sanctioned reverse carve-out** (verb-written
  on the lifecycle-mutated predecessor), **not** a violation — see Decision 4.
- **backlog** (`src/backlog.rs:404`) — `Relationships{slices, specs, drift, needs}`
  plus `after: Vec<AfterEdge{to, rank}>` and `triggers: Vec<Trigger{globs, note}>`;
  live.
- **spec** (`src/spec.rs:180`) — lineage `descends_from`/`parent` as `Option<String>`
  (single-valued outbound, **arity ≤1**) on the `Spec` struct; `members{requirement,
  label, order}` in `members.toml`; `interactions{target, type, notes}` —
  `type` **free-text** (`src/spec.rs:230`) — in `interactions.toml`.
- **review (RV)** (`src/review.rs`) — outbound `[target].ref`, a single subject
  edge (**arity 1**, the entity the review targets) in a bespoke `[target]` table,
  *not* a `[relationships]` block; the sibling `facet` is classification, not a
  relation.
- **rec (REC)** (`src/rec.rs:101`) — `owning_slice` as `Option<String>`
  (single-valued outbound, **arity ≤1** → SL); `decision_ref` is a **free-text**
  DEC ref (no `DEC` kind in `KINDS`) — target-unvalidated, like backlog `drift`.

These last two — surfaced by SL-046's C1 blocker, which made the "all-entity"
reader honest — bring the edge-authoring kinds to **six** of the eleven `KINDS`;
this ADR's vocabulary (Decision 2) must cover all six, not the four
`[relationships]`-block kinds alone.

There is **zero** shared relation type; cardinality is mixed (`Vec` vs `Option`),
structure is mixed (plain ref vs ranked vs labelled vs glob-payload vs
free-text-typed), aliveness is mixed. IMP-034 asks whether **all** of this should
refactor to one uniform generic schema (`[[relation]] kind="…" target="…"` +
optional attributes).

Two adjacent decisions constrain the answer:

- **ADR-004** — relations stored outbound-only; reciprocity derived, never authored
  on the target. Its **§5 carve-out**: a reverse edge MAY be *verb*-co-written when
  the target is already being rewritten for a lifecycle transition (supersession is
  the canonical, sole-sanctioned case). Any schema must compose with both.
- **SL-046** (the relation *reader*, design locked) unifies the *read* of the
  **reference/lineage subset** behind a per-kind `relation_edges` accessor emitting
  `RelationEdge{label, target}`, dispatched by `outbound_for(kind, id)` over
  `integrity::KINDS`. Read-only; excludes `needs`/`after`/`triggers`/`tags`/
  `interactions`.

The forcing function is **SL-048**: blocked on a project-global relation-governance
decision, it wants a cross-kind `link`/`relate` *write* verb. SL-046's read seam
does not supply a writer — **but a writer does not require unified storage either.**
The symmetric construction (per-kind *write* accessors + the same `outbound_for`
dispatch, gated by a code-authoritative vocabulary) delivers one `link` verb over
bespoke storage. What a uniform on-disk block would add is **append-only writes** (a
new table at EOF) instead of edit-preserving array-splice across the bespoke
per-kind schemas — a writer-*safety* gain, not writer *feasibility*.

## Decision

**Unify the relation *contract* and the cross-kind *write seam*; keep storage
bespoke, migrating the on-disk shape only for the clean pure-ref subset where it is
a pure win.** Specifically:

1. **Partition by the guarantee the typed shape carries, not "pure-ref vs payload".**
   Three tiers:
   - **Tier 1 — clean multi-ref** (`Vec<String>`, no arity constraint, no edge
     payload, no free-text type): slice `{specs, requirements, supersedes}`; backlog
     `{slices, specs, drift}`; governance `{supersedes, related}`. The generic block
     is a clean win here.
   - **Tier 2 — constrained / payload edges, stay typed:** spec lineage
     `{descends_from, parent}` (**arity ≤1 outbound**; a `Vec`-shaped generic block
     drops the single-valued guarantee into a runtime validator); spec
     `interactions{target, type, notes}` (`type` is **free-text**, not a validated
     label — incompatible with the vocabulary the block enforces — and `notes` is
     edge payload); backlog `after{to, rank}`, `triggers{globs, note}`; spec
     `members{requirement, label, order}`; review `[target].ref` (**arity 1**
     subject edge in a bespoke `[target]` table); rec `owning_slice` (**arity ≤1**
     → SL). A `Vec`-shaped generic block would drop the arity guarantee on the last
     two into a runtime validator, as it does for lineage.
   - **Out of scope of this ADR** (named to prevent mis-folding): backlog `needs`
     (payload-free by shape, but the dep/sequence axis **SL-047** owns); governance
     `tags` (classification, **not a relation**).

2. **Unify the contract and the write seam — the deliverable SL-048 needs.**
   - A **code-authoritative relation-label vocabulary** (a `RelationLabel` enum + a
     per-source-kind legal-set table declaring which labels are legal and which
     target-kinds they admit) is the single source of truth driving parser,
     validator, writer, *and* the SL-046 reader in lockstep. The table is built **on
     `integrity::KINDS`** (the one corpus-wide id table), not parallel to it; labels
     whose target is free-text (backlog `drift` — no `DRIFT` kind; rec `decision_ref`
     — no `DEC` kind) are marked *target-unvalidated*, not forced into `KINDS`. The
     legal-set table covers **all six** edge-authoring kinds (incl. review, rec), not
     the four `[relationships]`-block kinds alone — under-covering would silently
     narrow the contract below the SL-046 reader.
   - The cross-kind `link`/`relate` writer dispatches over **per-kind write
     accessors** — the mirror of SL-046's `relation_edges` read accessors — gated by
     that vocabulary. This yields one writer **without** a storage-unification
     mandate.

3. **Migrate the on-disk shape only for tier 1, opportunistically.** The generic
   `[[relation]] kind="…" target="…"` block is the storage shape for the clean
   multi-ref subset; it is append-only-writer-friendly and yields one parser there.
   Migrate a kind's tier-1 block **when that kind's *storage* is already being
   rewritten for an independent reason** (e.g. governance when SL-048 activates its
   links), verified by round-trip `doctrine <kind> show`. **A read-only touch does
   not trigger it** — SL-046 adds `relation_edges` read accessors to every kind's
   module *without* migrating any storage, and that is not a migration trigger. **No
   corpus-wide migration is mandated;** tier 2/3 storage is untouched.

4. **`superseded_by` is the ADR-004 §5 sanctioned reverse carve-out — kept, not
   removed.** Supersession flips the predecessor to terminal `superseded` status (its
   file is rewritten regardless), so the verb co-writing `superseded_by` adds **zero**
   marginal coupling and is the only honest place a reader of the dead record finds
   its successor (ADR-004 §5; Verification "the sole sanctioned reverse field";
   ADR-002's TOML is the precedent). It is **verb-written, never hand-authored**: the
   generic block admits outbound labels only, so `superseded_by` is not *authorable*
   there — but the typed carve-out field **stays**. A migrated governance kind
   therefore carries the `[[relation]]` block for `{supersedes, related}` **plus** the
   typed verb-written `superseded_by` beside it. *(This corrects the earlier draft of
   this ADR, SL-046 D4, and IMP-032, which all misread the carve-out as a violation;
   the honest follow-up ceiling is a `validate` cross-check that the stored value
   agrees with the derived `in_edges` reciprocal —
   `mem.pattern.review.superseded-by-is-adr004-carveout`.)*

5. **Outbound-only is enforced in code; reciprocity stays derived.** The vocabulary
   admits outbound labels only; no inverse/derived label is authorable in the generic
   block, so it cannot regress ADR-004. The reader projects outbound edges and derives
   "superseded by" / inbound from `in_edges` (ADR-004 §3) — it must **not** project
   the stored carve-out field.

6. **Compose with SL-048; do not silently expand it.** SL-048's non-goal *"no
   re-modelling existing relation fields — additive"* holds for tier 2/3 and for any
   un-touched tier-1 kind. The contract + write seam are **additive** (SL-048 builds
   them). Any tier-1 storage migration is opportunistic, owned by the slice already
   touching that kind — not a hidden mandate dumped on SL-048.

7. **State the contract in principle; do not bind it to unshipped API names.** SL-046
   is `design`, not shipped. This ADR fixes the tiers, the code-authoritative
   vocabulary, the write-accessor seam, and outbound-only — not the concrete
   `relation_edges`/`RelationEdge` identifiers, which lock once that seam survives
   contact with code.

Rejected alternatives:

- **(A) Full uniform storage** — genericises tier 2/3, losing arity (lineage),
  required-field (`members.{label, order}`), and validated-vs-free-text-type
  (`interactions`) guarantees. Confirmed net loss.
- **(E) Corpus-wide pure-ref migration mandate** (the prior draft of this ADR) —
  rests on the refuted premise that *only* unified storage yields the writer
  (Decision 2 supplies it over bespoke storage). Its real margin is
  append-only-writer safety + one parser, bought at corpus-wide authored-TOML churn
  and intra-entity idiom fragmentation (backlog/spec would split across
  `[[relation]]` + typed tables). Not worth a mandate; folded into Decision 3 as
  opportunistic tier-1 migration.
- **(B-bare) No contract, no write seam** — leaves SL-048 with a bespoke write path
  per source kind and no shared `link` verb.

## Consequences

### Positive

- SL-048 gets its one cross-kind `link`/`relate` writer — over the write-accessor
  seam, at near-zero migration cost.
- The relation contract (outbound-only + a code-authoritative vocabulary) is unified
  and enforced in code, not convention, and drives reader/parser/validator/writer in
  lockstep.
- ADR-004 is reaffirmed: outbound-only at the type level; the §5 carve-out preserved
  and **correctly classified**.
- Tier-2/3 typed guarantees (arity, required fields, validated-vs-free-text type) are
  kept.
- Storage churn is opportunistic and bounded to the clean tier-1 subset, not a
  corpus-wide migration.

### Negative

- The vocabulary table is new surface that must stay in lockstep with the reader's
  overlay labels and the validator — a maintenance coupling, mitigated by sourcing it
  once in code over `integrity::KINDS`.
- Two on-disk idioms coexist (generic tier-1 block + typed tier-2/3 tables); a
  migrated kind (governance) carries both — internal to a kind, accepted as the cost
  of keeping typed guarantees.
- The write-accessor seam is per-kind code (≈ the mirror of SL-046's read accessors)
  — more code than a single generic-array writer, the price of not mandating storage
  migration.

### Neutral

- Aliveness (inert vs live) is unchanged here — a reader/verb property settled by
  SL-046/SL-048, orthogonal to storage shape.
- `interactions` stays in its own typed file; its target-kind validation is
  unchanged. It is **not** folded into the generic block — its free-text `type` and
  `notes` payload disqualify it.
- Whether any given tier-1 kind ever migrates its storage shape is left to the slice
  that next touches it; this ADR permits but does not schedule it.

## Verification

- The shared vocabulary rejects any label outside the per-source-kind legal set and
  rejects every inverse/derived label (no `superseded_by` authorable in the generic
  block) — unit-tested at the vocabulary boundary.
- The `link`/`relate` write seam round-trips through `doctrine <kind> show` for every
  source kind, and refuses an illegal `(source, label, target-kind)` triple.
- The SL-046 reader's overlay labels and this vocabulary derive from the one code
  source; a test asserts they cannot diverge.
- The verb-written `superseded_by` survives a supersession and agrees with the derived
  `in_edges` reciprocal (the `validate` cross-check); the reader does not project it.
- Behaviour-preservation gate: `backlog order` stays byte-identical (`after`/`needs`
  untouched); existing per-kind suites stay green.
- Any tier-1 storage migration is verified by round-trip `show` of every affected
  entity.
