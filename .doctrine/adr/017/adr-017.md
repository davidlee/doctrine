# ADR-017: Actionability gating via inbound needs on unsettled records

## Context

Knowledge / governance records (PRD-010 / SPEC-019) must be able to **gate**
downstream work while unsettled — an `open` question blocks the slice it bears on,
a `held` assumption blocks an idea — without ever being work themselves (*truth is
not work*). SPEC-019 D7 parks every record state as `Terminal`, so today a record
is graph-present but can **never block**; IMP-047 ("trinary actionability") exists
to lift that limit. RFC-008 carried the deliberation.

The collision RFC-008 reconciled:

- **IMP-047 / RFC-007** wording: a record authors an outbound **gating edge** — new
  gating `RelationLabel`(s) + `RELATION_RULES` rows. Gating as *vocabulary*.
- **RFC-003 Layer 1** (the decisive reframing): whether an edge gates work is a
  **consumer decision, not a property of the relation**. Graph-effect is a
  projection; actionability lives in the **dep/seq layer, not the semantic relation
  model**. Design law: *derivable, not relational*.

The requirement to protect (CON-001): **association ≠ gating, and association must
not be hostage to gating** — one must be able to associate a record wherever it is
semantically sensible (`shapes`) without forcing an insensible actionability effect.

RFC-008's load-bearing finding: **blocking requires no new relation vocabulary.**
The engine already does the work. `A.needs = [B]` emits the dep-overlay edge B→A
(`src/priority/graph.rs`); `blocked_by(A)` keeps predecessors whose
`StatusClass != Terminal` (`src/priority/channels.rs`); reciprocity is *derived*,
never stored (ADR-004). So an unsettled record on the dep overlay as a predecessor
of work blocks that work, and settling it (→ `Terminal`) unblocks — for free. The
only thing standing in the way is that all record statuses are currently `Terminal`
(`src/priority/partition.rs`).

New relation vocabulary is required **only** to author gates *outbound from the
record*. That is an ergonomic/product choice, **not** a mechanical necessity — and
authoring outbound forces records into the dep/seq-**author** role (widening
`is_work_like`, giving records a dep table), a larger surface than authoring them
as pure targets. IMP-033 already made `needs`/`after` cross-kind *in the graph*,
but the authoring gate in `resolve_dep_seq_src` (`src/commands/dep_seq.rs`) also
required the **target** to be work-like — so `doctrine needs SL-x QUE-1` was
refused today. SL-158 D2 widened the target gate (`is_admissible_dep_target` =
work-like ∪ records) while keeping the source gate unchanged (records still
cannot author dep/seq). No new relation vocabulary or dep/seq axis is needed.

## Decision

We will model actionability gating as an **inbound `needs` dependency edge on an
unsettled record** — not a new relation, role, or dep/seq axis. The dependent work
declares `needs → <record>` exactly as it declares any other hard prerequisite; an
unsettled record is simply a non-`Terminal` dep predecessor.

Concretely:

1. **Trinary partition** (`src/priority/partition.rs`) is the sole engine delta.
   The record status vocabulary splits into a third class — `Gating` (unsettled,
   non-`Terminal`, non-`Workable`) — alongside `Workable` and `Terminal`. Per-kind
   settle boundary: ASM `held`/`testing`, DEC `proposed`, QUE `open`, CON `active`
   are `Gating`; their settled states are `Terminal`. The channel layer is unchanged
   — `eligible == Workable` keeps records off the worklist; `blocked_by` keeps the
   non-`Terminal` predecessor, so a `Gating` record blocks and a settled one clears.

2. **`shapes` stays purely semantic** — the record's outbound "what I affect"
   surface (honouring PRD-010 §3), graph-inert, optionally roled for its own
   semantic disambiguation (epistemic record→record vs affects record→work). Gating
   never rides `shapes`. Association (`shapes`) and gating (`needs`) are different
   edges on different layers, which makes *association-without-gating* structural
   (CON-001 satisfied without a per-edge gate/inform role).

3. **Outbound gating is a view, not a stored edge.** "What does this record gate?"
   is a **derived reverse-query** over `needs` (computed each pass, as
   `blocked_by`/`blocking` already are), not a stored axis. If batch authoring
   friction proves real, a thin **sugar** affordance may write N inbound `needs`
   edges from a record's perspective — without a stored outbound axis and without
   making records dep/seq authors. Deferred follow-up.

4. **Transitivity is direct-target only.** A gate is a direct `needs → record`
   edge. Propagating a gate along reference/implements edges (e.g. an `active`
   constraint gating the work that depends on a spec it shapes) is a larger,
   separate design with its own blast radius — **deferred**.

5. **Record→record gating is excluded initially.** Records are never `Workable`, so
   record→record `needs` is gating-inert; the existing dep overlay `CyclePolicy::
   Reject` guards the intra-family cycle hazard. No special exclusion is added.

This supersedes IMP-047's "new gating `RelationLabel`" wording: gating is the
existing dep/seq layer applied to a newly non-`Terminal` record class.

## Consequences

### Positive

- **Minimal surface.** Zero new relation vocabulary; the entire mechanism is the
  trinary partition. The M-Pr/M-E/M-L option space dissolves.
- **RFC-003-clean.** Actionability lives in the dep/seq layer; the semantic relation
  model (`shapes`) is untouched; graph-effect stays a consumer concern.
- **Settle→unblock is free** and already proven — `channels.rs` keeps the
  non-`Terminal` predecessor; the existing closed-prerequisite test is the gating
  mechanism in miniature.
- **Records stay pure targets** — no record-as-dep/seq-author surface, no widening
  of `is_work_like`, no stored duplicate axis to reconcile.

### Negative

- **Inbound authorship ergonomics.** A record cannot unilaterally broadcast "these
  are unsafe until I settle"; each dependent opts into the gate by authoring
  `needs`. Mitigated by the derived hub-view and the deferred batch sugar.
- **No transitive canon gating yet** (deferred) — gating an unsettled constraint
  through the spec it shapes onto dependent work requires the propagation follow-up.

### Neutral

- The third status class needs a name (`Gating` adopted; cosmetic).
- `shapes` roles remain a separate, smaller piece (semantic disambiguation),
  decoupled from gating and ridable on ADR-016.

## Verification

- **Partition canary** (`partition.rs`): per-kind `workable ∪ gating ∪ terminal ==
  <KIND>_STATUSES` extends the existing drift canary to the trinary split. The
  `every_knowledge_status_classifies_terminal_never_workable` expectation flips for
  the unsettled states (expected — consumer revision, not a regression).
- **Channel behaviour** (`channels.rs`): the existing
  `blocked_by` tests (a non-`Terminal` predecessor blocks; a `Terminal` one does
  not) cover gating unchanged — a `Gating`-class record predecessor blocks; settling
  it to `Terminal` clears.
- **Admissibility**: a work item authoring `needs → <record>` resolves and emits the
  B→A edge (records are scanned graph nodes; the `needs` work-like gate is
  source-only).
