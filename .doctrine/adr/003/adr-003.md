# ADR-003: Canonical change loop: slice-first, observe, reconcile, close

## Context

Doctrine's change model is a re-implementation of `spec-driver`'s, with pared-back
vocabulary: a **slice** is what `spec-driver` calls a *delta* (`doc/slices-spec.md`
§ Overview states this directly). The intent is the same model, not a different
one — the gaps below are doctrine's **age**, not divergent design.

`spec-driver` settled the workflow doctrine across two accepted ADRs:

- **ADR-004** (canonical workflow loop): delta-first implementation, *observation
  precedes reconciliation*, specs stay authoritative by **explicit reconciliation
  after the fact** rather than by premature assertion of future truth; closure
  requires owning-record reconciliation ("implemented now, fix specs later" is
  drift, not closure).
- **ADR-008** (normative lifecycle truth): specs own normative truth; audits and
  contracts are **observed-evidence overlays**; drift between them is reconciled
  **explicitly**, never silently by timestamp or artefact-kind precedence.

Doctrine inherits the intent but not yet the machinery. Product spec entities
exist (SL-015 composition seam; PRD-001..010, all `draft`), but there are **no
tech specs, no revisions, no reconciliation verb, no contracts**, no per-phase
review step, and no parallel phase dispatch. The `/audit` and `/close` skills are
tuned to the *slice* lifecycle, not the *spec* lifecycle. `doc/slices-spec.md`
already names the missing piece — "**audit / patch** (post-implementation
reconciliation of specs against what shipped) attaches as a later lifecycle stage"
(§ Forward compatibility) — and confirms nothing is reconciled today: `/close`
reconciles only the slice *status* against the phase rollup, never the specs.

Two forces shape the decision:

1. **The apparent contradiction.** Doctrine forbids code without an approved
   plan and authors `design.md` up front, which reads as the spec-first
   aspirational sequencing ADR-004 rejects. It is not — the two operate at
   different layers, and this ADR must say so explicitly or the contradiction
   festers.
2. **Contracts are deferred, honestly.** `spec-driver`'s contracts corpus
   (deterministic observed-truth derived from code; ADR-007) is an **efficiency
   mechanism, not a conceptual requirement** — and even there it is only
   supported for some languages. Doctrine defers it; its observed-truth substrate
   today is close reading of the implementation surface at audit plus memory
   git-anchors. The model must not depend on contracts existing.

## Decision

### 1. The canonical loop

For code-changing work the canonical doctrine loop is:

```text
slice -> design -> plan -> phases
  -> per phase: phase-plan -> implement (TDD red/green/refactor) -> complete -> [review]
  -> audit
  -> reconcile
  -> close
```

A slice is `spec-driver`'s delta; `design.md` is the in-bundle design record (its
*DR*). This is the canon for code-changing work; lower-ceremony variants (§ 10)
are concessions, not co-equal truths.

### 2. Design-before-code is not spec-first-aspirational editing

Doctrine has **two distinct layers**, and "no code without an approved plan"
governs only the first:

- **Per-slice `design.md`** — the HOW of *this* change, authored up front. This is
  legitimate and equals `spec-driver`'s DR. Authoring it before code is correct.
- **Evergreen specs** (PRD / tech) — the normative record of accepted product and
  architecture truth.

The plan gate does **not** license editing evergreen specs to describe future
reality as if it already shipped. Up-front design is not aspirational spec
editing. This resolves the apparent contradiction with ADR-004 §2/§8.

### 3. Observation precedes reconciliation

For code-changing work, evergreen specs are **not** edited aspirationally ahead of
implementation to assert future technical reality. Instead: the slice scopes the
change; design and the plan shape execution; implementation changes the system;
audit establishes observed truth; **reconcile** updates the specs to the accepted
reconciled outcome; close confirms the owning records are coherent.

### 4. Specs own normative truth; audit findings are observed evidence

Specs are the authoritative record of normative truth — what is accepted,
intended, planned, active, or retired. Audit findings (and, when built, contracts)
record **observed** truth — what the implementation actually does. When the two
disagree, that is **drift**, not an automatic replacement of spec truth.

### 5. Drift is reconciled explicitly, never by precedence

New evidence is a **prompt to reconcile**, not a mechanism that rewrites authority
by implication. The system must never decide that newer evidence "wins" by
timestamp, artefact kind, or overlay priority. (Doctrine's memory staleness, which
flags decay from git-anchors, is a *surfacing* signal under this rule — it informs
reconciliation, it does not override spec authority.)

### 6. Per-phase review lightens the whole-slice audit

Review need not wait for ostensible completion. `/review` is an **optional,
partial, per-phase** check run as each phase completes: confirm the phase's gates
(`EX-` / `VT-`) are *truly* met, that quality is sufficient, and that the backlog
is current. It may run **sequentially** (blocking the next phase) or
**non-blocking** (proceeding while the next phase begins).

`/review` is incremental confidence and hygiene; `/audit` (§ 7) is the whole-slice
critical reading and reconciliation-prep. Per-phase review makes the final audit
more **tractable** (less to read at once), enables **mid-course correction** (catch
a failed gate before it compounds), and keeps **review context focused** (one
phase, not the whole slice). It does **not** reconcile specs and does **not**
replace audit — audit still owns the holistic implementation-surface read and the
identification of spec changes (§ 7).

### 7. Audit prepares; reconcile writes — the seam

`/audit` and `/reconcile` are **distinct steps** with a hard edge:

- **`/audit` owns**: close critical reading of the implementation surface;
  correctness and quality review; drift identification and remediation; backlog
  propagation; triage of gaps. **Plus**: assembling the reconciliation context so
  that reconcile needs no further research, and **identifying** the spec changes
  to resolve. Audit does **not** write spec changes.
- **`/reconcile` owns**: **writing** the identified spec changes against observed
  truth. This is the point at which the specs regain authority.

Audit is already heavily loaded; folding spec writing into it would overload it
and blur the seam. Identification (audit) and authoring (reconcile) are separated
deliberately.

### 8. Reconcile precedes close; closure is a gate

The loop order is `audit -> reconcile -> close`. A slice does **not** reach a
terminal status while its owning specs remain drifted. "Implemented now, fix the
specs later" is drift, not closure (ADR-004 §10). Closure means the owning records
— slice status *and* the specs the slice touched — are coherent.

(Enforcement is deferred — see § 11 — so this is doctrine maintained by skill
discipline today, not a command gate. The doctrine is set now so the tuning has a
target.)

### 9. PROD and TECH specs differ in tolerance for forward intent

The loop applies to both kinds, but not with equal strictness (ADR-004 §4–5):

- **PROD specs** may reasonably carry planned or newly-accepted intent ahead of
  full implementation, **provided** lifecycle/coverage state distinguishes
  `planned` from `verified` reality.
- **TECH specs** should be reconciled from observed implementation after code
  changes; technical coherence is a property of the whole design rather than a
  checklist of isolated requirements, so aspirational TECH editing is more
  dangerous and generally not advised.

Doctrine has no tech specs yet; this posture is set now so it is latent rather than
retrofitted.

### 10. Ceremony shapes strictness, not truth

Skills and configuration govern how much scaffolding is expected and how hard
agents steer back to canon. They do **not** create multiple workflow truths. The
design-doc-less trivial slice (`doc/slices-spec.md`) and other low-ceremony entries
are concessions for proportionate work, not redefinitions of the golden path.
Likewise, **execution orchestration** (serial `/execute` vs parallel `/dispatch`,
§ 11) shapes *throughput*, not truth: how phases are run does not change what the
loop means or what closure requires.

### 11. Deferred machinery (the age, not the intent)

The following are explicitly **not yet built**. The model is held stable so they
attach without restructuring (`doc/slices-spec.md` § Forward compatibility):

- **Contracts** — the deterministic observed-truth corpus. Efficiency, not
  concept; language-limited even in `spec-driver` (ADR-007 is its template).
  Until then, observed truth = audit close-reading + memory anchors.
- **Tech specs** — the spec entity supports the kind (the composition seam makes
  `interactions` tech-only), but none are authored; § 9's TECH posture is latent.
- **The reconcile artefact (≈ `spec-driver` *revision*)** — the durable record of
  what reconcile changed and why. Name provisional. The `/reconcile` skill and any
  `slice reconcile` / spec-patch CLI are unbuilt; reconcile is a manual skill step
  for now.
- **The `/review` skill** — the per-phase incremental review (§ 6) is unbuilt;
  today it is manual discipline via `/code-review`. Non-blocking review (running
  while the next phase proceeds) needs harness support to schedule and track.
- **`/dispatch`** — parallel / worktree / batched sub-agent execution of a slice's
  phases. `spec-driver` has it; doctrine ships a reserved **placeholder** skill
  that routes to serial `/execute`. Deferred, and its implementation must contend
  with harness- and vendor-specific orchestration (worktree isolation, sub-agent
  spawning, batching). Under dispatch, per-phase `/review` (§ 6) becomes the
  natural per-worktree gate before integration.
- **Lifecycle / coverage states** — per-requirement `planned`/`in-progress`/
  `verified` plus baseline states (`spec-driver`'s `asserted` / `legacy_verified`)
  that § 9 depends on. Not modelled.
- **Closure-gate enforcement** — § 8 is doctrine-by-discipline; no command gates a
  slice `done` on spec reconciliation (`doc/slices-spec.md`: "no closure gate in
  v1").
- **Audit / close skill tuning** — the current skills serve the slice lifecycle;
  tuning them to the full review→audit→reconcile lifecycle, and authoring
  `/review` and `/reconcile`, is follow-on work.

## Consequences

### Positive

- One authoritative answer to "what is doctrine's golden path?", in doctrine's own
  vocabulary — and it resolves the slice-design-first vs delta-first tension by
  **layering** it (§ 2) rather than choosing a side.
- Closure means something: a closed slice's owning specs are coherent (gated when
  built; by discipline now).
- A clean audit/reconcile seam (§ 7) keeps audit focused on review/drift/triage/
  context-prep and gives spec writing its own home; neither step bloats.
- Per-phase review (§ 6) makes the final audit tractable, enables mid-course
  correction, and keeps each review's context focused — and composes with
  `/dispatch` as the per-worktree integration gate.
- Sets the stable target the deferred machinery (§ 11) attaches to — no
  restructuring required when contracts, tech specs, the reconcile artefact, the
  `/review` and `/reconcile` verbs, dispatch, and the closure gate land.
- Defers contracts **honestly**: named as efficiency, not concept, so their
  absence is not a hole in the model.

### Negative

- Doctrine runs ahead of enforcement: § 7's seam and § 8's gate are
  discipline-only until skills and CLI catch up, so drift can occur unflagged.
- A distinct `/reconcile` step adds a stage to every non-trivial slice, and
  reconcile-before-close lengthens the path to `done`.
- Non-blocking per-phase review (§ 6) and parallel dispatch (§ 11) both lean on
  harness/vendor capabilities doctrine does not yet abstract — a portability and
  coordination cost when they are built.
- The TECH-spec posture (§ 9) is set with zero tech specs in existence — it may
  need revision once they are real.
- Without contracts, evergreen-spec reconciliation leans entirely on the quality
  of audit close-reading — the weakest substrate until contracts land.

### Neutral

- No runtime enforcement is required today; this ADR sets canon, and gates remain a
  separate follow-on concern (mirroring ADR-004's stance).
- It does not forbid low-ceremony or trivial slices; they remain concessions.
- It does not mandate one reconcile-artefact timing or shape (cf. ADR-004 §6) —
  reconcile is a *step*; the durable artefact's schema is deferred.
- It does not mandate parallel execution; serial `/execute` remains valid, and
  `/dispatch` is an optional throughput concession (§ 10).
- Authored as a single ADR; it may later split into a loop ADR and a
  lifecycle-truth ADR if the normative/observed doctrine grows its own machinery.

## Verification

- `.doctrine/state/boot.md` "Core process" and the routing table name the loop as
  `… per phase [review] … audit → reconcile → close`, with review distinct from
  audit, and reconcile distinct from both neighbours.
- The `/review` skill (when authored) runs per-phase gate/quality/backlog checks,
  optionally non-blocking, and is distinct from the whole-slice `/audit`.
- The `/audit` skill identifies the spec changes and assembles the reconciliation
  context; it does **not** write spec edits.
- The `/reconcile` skill (when authored) writes spec changes against observed
  truth and is the **sole** spec-reconciliation writer in the loop.
- The `/close` skill refuses a terminal status while owning specs are drifted
  (discipline now; gated when the closure gate is built).
- `/dispatch` (when authored) orchestrates § 1's per-phase execution in parallel
  without changing the loop's meaning or closure requirements (§ 10).
- Product/tech spec edits made during a slice are reconciliation outputs
  (post-audit), not aspirational pre-implementation assertions — except for the
  PROD planned-intent that § 9 permits.
