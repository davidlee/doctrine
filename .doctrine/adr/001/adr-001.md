# ADR-001: Module layering: leaf ← engine ← command, no cycles

## Context

`src/` is a single crate, ~16 modules, flat tree (~7.2k prod LoC). A
2026-06-05 coupling assessment found the dependency graph structurally honest
but **ungoverned in one axis**.

The codebase already has a *vertical* architecture rule — the pure/imperative
split (`doc/slices-spec.md` §Architecture): the CLI shell is thin, decisions
live in pure functions, impurity sits behind a seam. That rule is well-honored.

It has **no *horizontal* rule** — nothing states which module may depend on
which. The implicit tiering (utilities ← core engine ← command shells) holds in
practice, but because it is unwritten, drift is invisible to review. The
`slice ↔ state` import cycle (SL-016) fell through exactly this gap: no rule
forbade it; only an ad-hoc grep caught it. As the module count grows, "caught by
someone happening to grep" does not scale.

`Cargo.toml` already carries an empty `[workspace]` stanza — multi-crate was
anticipated but not taken.

## Decision

We adopt an explicit **horizontal layering rule** for `src/`. Every module
belongs to one of three tiers; dependencies point in one direction only, and the
module graph is **acyclic**.

| Tier | What it is | May depend on |
|---|---|---|
| **leaf / seam** | primitives & IO seams; no domain knowledge | nothing in-crate (or other leaves) |
| **engine** | core domain model & state machinery, pure-first | leaf tier |
| **command** | CLI shells — one per verb, thin | engine + leaf |

The authoritative per-module tier assignment lives at
`.doctrine/adr/001/layering.toml` — a companion TOML surface under this ADR's
directory. The map classifies 67 units (23 leaf / 18 engine / 26 command) with
per-unit rationale, mixed-umbrella sub-classifications (`catalog::scan`,
`catalog::hydrate`, `priority::graph`, `status::run_status`), frozen
`[[accepted_violation]]` entries (10 upward edges baselined with follow-ups), and
`[tangle_baseline]` per-tier cyclic-edge counts (leaf=0, engine=0, command=120).

Rules:

1. **Dependencies point downward only**: command → engine → leaf. Never upward.
   Hard-gated over literal `crate::` path edges by the `syn`-based fitness
   gate; a small *enumerated, frozen* `ACCEPTED_VIOLATIONS` baseline (10 edges)
   is recorded in `layering.toml` and may shrink, never grow.
2. **Cycles are ratcheted, not globally forbidden.** The engine tier is a
   clean DAG (0 cyclic edges; CHR-015 done). The command tier has a large
   intra-tier SCC (~32 modules, 120 cyclic edges) — normal for a CLI verb
   layer, openly recorded as unmet-and-tracked. The per-tier cyclic-edge count
   (same-tier edges inside non-trivial SCCs) is frozen in `[tangle_baseline]`
   and may not grow — any increase fails `just gate`.
3. **Engine purity is deferred** — the impure-leaf refinement (tag
   `git`/`clock` impure, forbid engine→impure-leaf) is follow-up work. The
   existing pure/imperative split remains the governing guidance.

The tier of a module is a property of *what it knows*, not where its file sits.
Folder structure under `src/` is **not** mandated by this ADR (folders organize,
they do not enforce) — see Consequences.

## Consequences

### Positive

- The dependency direction is now a reviewable rule. A cycle or upward edge is a
  named defect a reviewer (human or agent) can cite, not a matter of taste.
- Gives the eventual workspace split a blueprint: the three tiers are candidate
  crate boundaries.
- Restates the pure/imperative rule in dependency terms, unifying the vertical
  and horizontal governance into one mental model.

### Negative

- Enforcement is **mostly automated** — the `syn`-based fitness gate
  (`tests/architecture_layering.rs`, under `just gate`) hard-gates cross-tier
  direction and ratchets intra-tier cycles. Review covers what the literal
  `crate::` path scope cannot: macro/re-export laundering, and whether a new
  `[[accepted_violation]]` entry is justified.
- Some current modules are imperfect fits and must be watched, not pretended
  away: `install` is depended on by the engine (`state → install`) because it
  exposes shared scaffolding helpers — a command module doubling as a utility
  provider. Its helper surface arguably belongs in a lower tier; flagged as a
  follow-up, not resolved here.

### Neutral

- Single crate remains the chosen shape. The `[workspace]` stanza stays inert
  until a split is warranted.

## Verification

- **Now — the `syn`-based fitness gate.** SL-112 shipped
  `tests/architecture_layering.rs` — a `cargo test` under `just gate` that
  extracts the production `crate::` dependency graph via `syn` (canonical AST;
  not brittle regex) and enforces: (1) cross-tier direction hard-gated over
  literal `crate::` path edges, modulo a frozen `ACCEPTED_VIOLATIONS` baseline;
  (2) intra-tier cycles ratcheted by a per-tier cyclic-edge count that may not
  grow; (3) mixed umbrellas forced to sub-classify. The authoritative tier
  assignment is `.doctrine/adr/001/layering.toml` — a companion TOML surface
  under this ADR's directory. A new upward edge fails `just gate` naming the
  offending edge.
- **Later — the crate graph (free, compiler-enforced).** Rust's real fitness
  function for layering is the crate boundary: Cargo forbids circular crate
  dependencies *at compile time*. When the engine stabilizes, promote the engine
  tier to its own workspace crate; the compiler then makes upward/cyclic edges a
  build error. Trigger: engine churn settles, or a second cycle appears.
