# ADR-002: Global orientation memory class: repo-empty, unanchored, evergreen

## Context

Doctrine's memory store (SL-005/007/008) serves **project-local capture**:
git-anchored TOML+MD entities recording what an agent learned in *this* client
repo. It does not serve **framework orientation** — durable, repo-agnostic
knowledge about how to drive doctrine itself, the kind that should travel with
the binary to every client and surface through `doctrine memory retrieve
--path-scope`. SL-018 adds that corpus. Shipping it as native memory entities
(one format, one retrieval path, scoped reach the boot snapshot can't give)
collides with three load-bearing rules of the existing store:

1. **Scoped ⇒ anchored.** `doc/memory-spec.md` § Scope & anchoring states an
   unanchored memory is "permitted only for unscoped memory." An orientation
   memory is path/command-scoped (that scope is the whole point) yet has no
   honest git anchor — it is minted upstream, in no client's repo. A real anchor
   would assert something false about the client's git.

2. **`repo` is the cross-repo filter.** The retrieval partition drops a memory
   whose `repo` is non-empty and ≠ the querying repo. A framework memory carrying
   any real repo-id would self-exclude from every client — dead on arrival. The
   honest encoding is `repo=""`, which `base_filter` admits in every partition.
   That admission path is **dormant today**: `record` always derives a non-empty
   `repo` from cwd git, so zero `repo=""` memories exist in production. SL-018 is
   the first to light it — there is no lived baseline to inherit.

3. **Staleness decay.** The current staleness model gives a scoped-but-unattested
   memory (no `verified_sha`) the *days-since-`reviewed`* metric. An evergreen
   orientation corpus would be branded progressively `stale` by simple elapsed
   time, and the only cure — bumping `reviewed` on every sync — would break the
   idempotent diff that makes the per-session refresh (SL-018 M1) cheap.

These are not implementation accidents; they are correct rules for the capture
store. The orientation corpus is a genuinely **different class** of memory, and
the spec is canon — it must be amended to sanction the class before the corpus is
authored or its admission path is relied upon, not bent silently around it.

## Decision

We recognise a distinct **global / unanchored / path-scoped / derived** memory
class and bless it in `doc/memory-spec.md`.

A member of the class is defined by its signature, not a new `memory_type`:

- **`repo = ""`** — *global*. Admitted in every retrieval partition (a non-empty
  `repo` still hard-filters cross-repo, unchanged). The empty repo-id is the
  honest encoding of "repo-agnostic," not a hack around the filter.
- **`anchor_kind = "none"`** — *unanchored*. It asserts nothing about any
  client's git. We sanction **scoped + unanchored** for this class specifically,
  the documented exception to the otherwise-correct "unanchored permitted only
  for unscoped" rule.
- **path-scoped** — it still carries ≥1 `scope.paths`/`globs`/`commands` (the
  retrieval-key floor holds; a tag-only global memory is not permitted), so it
  surfaces only on a scope match and does not dilute focused queries.
- **derived** — materialised from the binary into a gitignored tree, never
  committed to the capture store; surfaced through a **second indexer scan root**
  unioned with `items/`. The capture tree and its scoped⇒anchored invariant are
  untouched.

We further define a **fourth staleness disposition** for this class. A memory
with `repo=""` + `anchor_kind=none` (+ no `verified_sha`) is **evergreen /
reference-grade**: exempt from the days-since-`reviewed` decay, it renders an
explicit, non-decaying **`reference`** staleness state — never a decaying
`stale`. Evergreen-ness is *derived* from the class signature, not stored and not
re-stamped, so it costs nothing and cannot break the idempotent sync.

## Consequences

### Positive

- Orientation knowledge gains **per-scope retrieval** — "I'm about to touch
  `install/manifest.toml`, what should I know" — which the static boot snapshot,
  skills, and `doc/*` cannot offer. One memory format, one retrieval path.
- **Honest tiering.** Shipped copies are derived (regenerable from the binary),
  so they are gitignored and live outside the committed capture tree; provenance
  is honest (`repo=""`, `anchor_kind=none` assert nothing false).
- The capture store's invariants stay **pristine** — the class is additive
  (a second scan root, a derived signature), not a mutation of `items/`.

### Negative

- It **lights the dormant `base_filter` admission path** for the first time. With
  no lived baseline, that path is pinned by a **required golden test** (SL-018
  PHASE-02), not assumed equivalent to today's behaviour.
- The class is a **sanctioned exception** to the spec's scoped⇒anchored rule.
  Reconciled by the accompanying memory-spec amendment (this is not an
  improvisation), but it does widen the legal shape-space — a hand-authored
  `repo=""`/anchored or `repo=""`/scopeless memory is now a lint target
  (SL-018 master-lint enforces the full signature).

### Neutral

- No `memory_type` enum change. The class is a *signature*, orthogonal to type;
  reference-grade content is authored as `signpost` (the enum has no `reference`
  member, and `MemoryType::parse` rejects the literal).
- `reviewed` is still seeded at authoring time for the recency tiebreak (sort key
  #8), though it no longer governs staleness for the class — stable across syncs.

## Verification

- The memory-spec amendment (this ADR's companion edit) defines, consistently
  across § Scope & anchoring and § Retrieval & ranking: the scoped+unanchored
  carve-out, the `repo=""` partition admission, and the evergreen `reference`
  staleness row + enum member.
- SL-018 PHASE-02 implements the evergreen staleness disposition in code and pins
  the `repo=""`/`anchor_kind=none` admission with a required golden test.
- SL-018 PHASE-04 master-lint enforces the full class signature on every shipped
  master (`repo=""`, `anchor_kind=none`, ≥1 path/glob/command, valid
  `memory_type` ≠ `reference`).
- The behaviour-preservation gate (SL-005/007/008 suites green unchanged) proves
  the capture store's existing semantics are untouched.
