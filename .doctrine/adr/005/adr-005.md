# ADR-005: Shipped knowledge is tiered by access pattern; skills route, reference docs explain

<!-- Reference forms: doc/glossary.md § reference forms. -->

## Context

Doctrine's value is correct *agent* behaviour in **client** repos. The knowledge
that produces it — how to use doctrine, what the conventions are, which CLI verb
does what — is currently mis-homed and duplicated:

- **Load-bearing usage rules lived where clients never see them.** The storage
  rule, reference forms, and "use the CLI" sat in this repo's `CLAUDE.md`, which is
  *build-doctrine* guidance — never shipped into a client install. This produced a
  concrete failure: an agent judged a requirement "hollow" by reading its empty
  `.md` prose tier, missing the substance in the sibling `.toml` (the SL-020
  inquisition C4 false charge; partially fixed in commit 8206b67). Recorded as
  `mem.system.governance.ship-usage-guidance-to-clients`.
- **Skills restate implementation detail.** Each skill re-bakes CLI command shapes,
  storage-tier mechanics, and hand-editing rules. This duplicates (against the repo's
  no-parallel-implementation rule) and drifts (a CLI change strands N skills).
- **Two access patterns already govern the surface, implicitly.** The boot snapshot
  (`@.doctrine/state/boot.md`) is **push** — inlined into every session's cached
  prefix, resident without a lookup. Skills are **pull** — read only when routed to.
  A doc that a skill merely *points at* is **pull-squared**: invoke the skill, then
  go read the doc. Demoting a load-bearing rule into a pulled pointer is how the C4
  class of error recurs.
- **The glossary is authoritative but unshipped.** `doc/glossary.md` (vocabulary,
  ids, reference forms, verification taxonomy) is not in the embed set (`install/`,
  `plugins/`, `memory/`), so client installs lack the foundational conventions.

This is a corpus-wide structural question — where each kind of usage knowledge
lives, by how it is accessed — not a one-off edit. Hence an ADR.

## Decision

**We will tier shipped knowledge by access pattern, and make skills route rather
than restate.** Three tiers:

1. **PUSH — the boot snapshot (resident, compact).** Carries the workflow digest
   (`install/routing-process.md`, which already rides boot) **and** the load-bearing
   usage rules an agent must never get wrong without a lookup: reference forms,
   storage tiers (structured→TOML / prose→MD, read via `show`), and "use the CLI,
   don't guess." These fire without an invoke. The snapshot stays compact —
   load-bearing only, never reference detail.

2. **PULL-reference — shipped `install/*.md` docs** that skills and boot point to.
   A pull-reference is only *visible* to a client if a skill or the boot digest
   **points at it** (install copies `install/*` → `.doctrine/*`, but an unreferenced
   `.doctrine/*.md` is read by no one — the `AGENTS.md` lesson). The set:
   - `glossary.md` — **shipped**, by **relocation under `install/`** (R-OQ-1), not by
     embedding `doc/` wholesale (which would ship ~9 unwanted build-specs). Its
     `entity-model.md` cross-link is resolved at ship (inline or drop — no dangling
     link into an unshipped doc).
   - one CLI / hand-editing reference — a **new shipped doc under `install/`**
     (R-C5), *not* `install/rules/AGENTS.md` (that lands at `.doctrine/rules/` where
     nothing reads it). It carries hand-editing mechanics, storage-tier read/write,
     edit-preserving rules, and *which verb for what*; **points to `doctrine --help`**
     for exact command shapes and **never duplicates** it; and is **pointed-at** by
     the skills that need it + the boot digest so it is reachable.

3. **Skills — thin task routers.** Each states *when* and *what to do for this
   stage*, and references tiers 1–2 for the shared *how*.

**Invariants:**
- A rule an agent needs *before* it would invoke any skill belongs in the PUSH tier,
  never only in a pulled doc.
- There is exactly **one** workflow doc (`routing-process.md`) — no parallel summary.
- No shipped doc duplicates `doctrine --help`; it points there for shapes.
- **The restate line (R-OQ-4, ratified canon):** a skill **MAY** name a verb and
  cite a rule by its name (a pointer); a skill **MUST NOT** reproduce flag syntax,
  option/enum tables, or storage-tier mechanics as prose.
- A pull-reference doc must be **pointed-at** (by a skill or boot) — shipping ≠
  reachable.

## Consequences

### Positive

- Single source per fact (DRY); skills shrink to routing.
- Client installs receive the usage knowledge that produces correct behaviour.
- The C4 class of error (wrong-tier reading, invented conventions) is prevented at
  the resident tier, not left to a pull-squared pointer.
- A CLI change updates one reference doc, not N skills.

### Negative

- Migration cost: audit every skill, author two reference docs, ship the glossary,
  wire the boot/embed set, de-duplicate.
- The boot snapshot is cache-precious; adding the load-bearing rules risks bloat and
  must be policed for compactness.
- Two shipped reference files to maintain; the rust-embed re-embed footgun
  (`mem.pattern.embed.rustembed-recompile-and-symlinks`) means a doc edit needs a
  crate rebuild to ship.

### Neutral

- `routing-process.md` may be renamed/promoted to a clearer "workflow" identity.
- The glossary's home (relocate under `install/` vs add `doc/` to the embed) is a
  mechanism choice, deferred to the implementing slice.

## Verification

- **VT** — the glossary and the CLI/editing reference are present in a fresh client
  install (embed/ship test).
- **VT** — the reference-forms block is present in a regenerated boot snapshot
  (assertion alongside the existing `boot --check`).
- **VA** — no skill violates the restate line (R-OQ-4): named file:line offenders
  reproducing flag syntax / option tables / storage mechanics prose are removed;
  pointers are permitted. Scope is the **evidence-bound** set (R-C1), not a blanket
  20-skill sweep.
- **VA** — no shipped doc reproduces `doctrine --help` content; each pull-reference
  is pointed-at by a skill or boot (reachability check).

## Open Questions — resolved at review (2026-06-08, inquisition.md)

- **R-OQ-1** — glossary ship: **relocate under `install/`** (curated; no `doc/`
  wholesale embed). Resolve the `entity-model.md` link at ship — no dangling link.
- **R-OQ-2** — **keep compile-time embed** (status quo). The rust-embed footgun is a
  known dev-loop cost, not a shipping defect; runtime-read is out of scope.
- **R-OQ-3** — **leave `routing-process.md` as-is**, gain the reference-forms block.
  A rename would churn every `@`-import + hook reference for cosmetic gain.
- **R-OQ-4** — **ratified** as an Invariant (the restate line, MAY-pointer /
  MUST-NOT-reproduce). The de-dup gate is now testable.
- **R-OQ-5** — **append** the reference-forms block to `routing-process.md` (one
  Static asset; precedent — the read-rule already rides it). Not a new boot section.
