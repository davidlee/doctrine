# RFC-008: Actionability gating: association vs graph-effect

## Context

<!-- The forces at play: the problem, the question, and why it merits structured
     discussion. An RFC asserts NO canon ‚Äî it is governance-neutral deliberation. -->

IMP-047 / RFC-007 workstream 1 ("trinary actionability") proposes a third priority
status-class so that unsettled epistemic/governance records (PRD-010 / SPEC-019) can
**gate** downstream work without ever appearing in `next` as work. The partition half
is mechanical and settled (see SL-158 notes). The **gating-edge** half is not: it is an
unsettled graph-model decision that RFC-003's holistic relation-model review reopens,
with corpus-wide blast radius. SL-158 was opened to build it, hit this question in
`/design`, and is **parked** (`proposed`) pending this RFC. This RFC carries the context
across the several deliberation passes the decision needs; SL-158 resumes once it lands.

It may **emit an ADR** if a mechanism is chosen.

### The collision

- **IMP-047 / RFC-007** wording: "a record authors an outbound **gating edge**‚Ä¶ new
  gating `RelationLabel`(s) + `RELATION_RULES` rows." I.e. gating as *vocabulary*.
- **RFC-003 Layer 1** (newer review): "whether an edge *gates work* is a **consumer
  decision, not a property of the relation**." Graph-effect is a *projection* over
  selected edges, declared in the consumer; actionability (`needs`/`after`/"IMP-047
  `gates`") lives in the **dep/seq layer, not the semantic relation model.**
- **RFC-003 design law**: *derivable, not relational* ‚Äî don't encode in the relation
  what the consumer can project.

RFC-007 looks to have inherited IMP-047's older label-wording without reconciling it
against RFC-003. This RFC is that reconciliation.

### The requirement (the thing to protect)

Records already author `Shapes` (record ‚Üí wide artefact set) as their durable semantic
"what this affects" relation ‚Äî today **graph-inert**. The temptation is to make every
`Shapes` edge gate. The protect-this:

> **Association ‚âÝ gating, and association must not be hostage to gating.** One must be
> able to associate an epistemic record wherever it is *semantically* sensible without
> that association forcing an *insensible actionability* effect.

A countervailing positive (so we don't over-correct): **gating canon is acceptable.**
Governance is not itself actionable, but an *unsettled* governance record transitively
gating the work that depends on it can be exactly right. So the goal is not "never gate
canon" ‚Äî it is "let gating select *which* associations bite, by intent, not by accident."

## Discussion

<!-- The position(s) explored, pros/cons, alternatives considered. -->

### Shorthand (for the scenarios)

```
node state:  QUE-001(open)       record, UNSETTLED  ‚Üí would-gate class
             QUE-001(answered)   record, SETTLED    ‚Üí terminal, inert
             SL-020[design]      artefact, workable
             SL-020[BLOCKED]     artefact, blocked by a live gate

edges:       A ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ B     semantic association ‚Äî relation model, NO graph-effect
             A ‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂ B     actionability gate   ‚Äî dep overlay; B BLOCKED while A unsettled,
                                                        auto-clears when A settles (Terminal)
```

The mechanical partition (unsettled record ‚Üí would-gate class; settled ‚Üí terminal) and
the channel layer (`eligible == Workable` excludes records from the worklist;
`blocked_by` keeps non-terminal dep-predecessors so a would-gate node blocks) are
**common to every option below** and are not in dispute. The dispute is **which `shapes`
associations become `‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂` edges, and how that is decided.**

### Mechanism options

| id | mechanism | who decides a gate | layer (RFC-003) | assoc‚âÝgating? |
|---|---|---|---|---|
| **M0** | status quo ‚Äî records all-terminal, gate nothing | ‚Äî | ‚Äî | n/a (no gating) |
| **M-P** | blanket projection: every `shapes` gates while unsettled | nobody (automatic) | consumer | **NO ‚Äî coupled** |
| **M-Pk** | projection filtered by **target kind** (only work kinds gate) | consumer heuristic | consumer | partial (by kind, not intent) |
| **M-Pr** | projection refined by a **role on `shapes`** (`gates` vs `informs`) | author, per edge | consumer + 1 role | **yes** |
| **M-E** | distinct **`gates` dep/seq axis** (orthogonal to `shapes`) | author, explicit edge | dep/seq | **yes** |
| **M-L** | distinct **`Gates` RelationLabel** + `RELATION_RULES` rows | author, explicit edge | vocabulary | yes ‚Äî but RFC-003 says **wrong layer** |

### Scenarios ‚Äî the discriminators

**S1 ‚Äî the canonical gate (works everywhere).** An open question must block a slice's
design; answering it unblocks.

```
QUE-001(open) ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ SL-020        ‚ü∂ project ‚ü∂   QUE-001(open) ‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂ SL-020[BLOCKED]
                                                       answer QUE-001 ‚Üí (answered) ‚Üí SL-020[design]
```
Every option M-P..M-L satisfies S1. M-P/M-Pk for free (shapes alone gates); M-Pr needs
`role=gates`; M-E/M-L need an explicit gate edge. **Not a discriminator** ‚Äî it only
shows the partition+channel mechanics, which all share.

**S2 ‚Äî shape WITHOUT gate (the killer).** A proposed decision *bears on* a risk
semantically, but must not block it.

```
DEC-005(proposed) ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ RSK-004[open]

  M-P  : RSK-004[BLOCKED]   ‚úó couples association to gating
  M-Pk : RSK-004[BLOCKED]   ‚úó RSK is a "work kind" ‚Üí kind-filter too coarse, still gates
  M-Pr : RSK-004[open]      ‚úì author wrote shapes(informs), no gate
  M-E  : RSK-004[open]      ‚úì author added no gates edge
```
S2 **kills M-P and M-Pk** against the requirement: only *per-edge author intent*
(M-Pr / M-E) lets you associate without gating. Target-kind is the wrong axis ‚Äî gate-ness
is an intent, not a property of the target's kind.

**S3 ‚Äî gating canon, transitively (the positive ‚Äî and a hidden cost).** An active
constraint shapes a spec; a slice implements the spec. User's intuition: the constraint
should gate the *dependent work*, not (uselessly) the canon node.

```
CON-003(active) ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ SPEC-019      SL-040[design] ‚îÄ‚îÄreferences(implements)‚îÄ‚îÄ‚ñ∑ SPEC-019

  direct-gate-the-canon-node:  CON-003 ‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂ SPEC-019[BLOCKED]
     but SPEC-019 is canon ‚Äî never workable anyway ‚Üí blocking it changes NOTHING in `next`.
     SL-040 is NOT blocked: `references` is a CONSEQUENCE edge, not a dep edge ‚Äî gating
     does NOT flow along it today.
```
S3 surfaces that **transitive gating of canon is not free.** Gating lives on the dep
overlay; `references`/`implements` are consequence overlays. To make "unsettled CON gates
the work that depends on SPEC-019," gating must either (a) target the dependent work
**directly** (`CON-003 ‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂ SL-040`), or (b) **propagate** along reference/implements
edges ‚Äî a new propagation rule with its own blast radius. **Open design axis, independent
of M-P..M-L.**

**S4 ‚Äî intra-family (cycle hazard).** A held assumption shapes an open question.

```
ASM-002(held) ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ QUE-001(open)   [and possibly QUE-001 ‚îÄ‚îÄshapes‚îÄ‚îÄ‚ñ∑ ASM-002]

  M-P  : ASM ‚ïê‚ïêgate‚ïê‚ïê‚ïê‚ïê‚ñ∂ QUE ‚Äî but QUE is a record, never workable ‚Üí worklist effect NIL;
         and if the reverse shape exists, a dep-overlay CYCLE (Reject policy) forms.
  M-Pk : record-target excluded ‚Üí association only, no gate, no cycle.   ‚úì (this is where kind-filter helps)
  M-Pr/M-E : author simply doesn't mark/author a gate ‚Üí no cycle.        ‚úì
```
S4 shows record‚Üírecord gating is mostly inert (records are never workable) but carries a
**dep-cycle hazard** under blanket projection. Per-edge intent or a record-target
exclusion both avoid it.

### Reading of the scenarios

- The **requirement (S2)** eliminates **M-P** and **M-Pk**: gate-ness must be author
  intent, per edge ‚Äî not automatic, not target-kind-derived.
- That leaves **M-Pr** (role on `shapes`) vs **M-E** (distinct `gates` axis) vs **M-L**
  (label). **M-L** is disfavoured by RFC-003 (wrong layer ‚Äî vocabulary). So the live
  fork is **M-Pr vs M-E.**
  - **M-Pr** keeps one relation (`shapes`) with a closed role `{gates, informs}`; gating
    is a consumer projection over the `gates`-role subset. Minimal vocabulary; rides
    SL-149's role machinery (ADR-016); RFC-003-clean (graph-effect stays in the consumer,
    intent rides a role). Risk: roles currently refine `references`, not `shapes` ‚Äî extends
    the role grammar to a second label.
  - **M-E** makes `shapes` purely semantic and `gates` a separate dep/seq axis alongside
    `needs`/`after`. Fully orthogonal; "the dep/seq layer" taken literally. Risk: a second
    near-duplicate axis (`gates` vs `needs`) ‚Äî a record's `gates` is structurally an
    inverted `needs`; coordinate with IMP-033 rather than fork.
- **S3 (transitive canon gating)** is an **orthogonal** axis to the M-fork: whichever of
  M-Pr/M-E wins, decide separately whether a gate (i) targets dependent work directly,
  (ii) targets the canon node and propagates along reference edges, or (iii) both. Direct
  targeting is simplest and avoids a propagation rule; propagation is more expressive but
  larger.
- **S4** argues for either a record-target exclusion or relying on per-edge intent to
  avoid record‚Üîrecord gate cycles, regardless of the M-fork.

## Outcome

<!-- If resolved: what was concluded. If open: what remains unanswered. -->

**Resolved (2026-06-26) ‚Üí [ADR-017](../../adr/017/adr-017.md).** Gating is an **inbound
`needs` dependency edge on an unsettled record** ‚Äî no new relation, role, or dep/seq
axis. The dependent work declares `needs ‚Üí <record>` like any other prerequisite; the
**trinary partition** (unsettled record ‚Üí non-`Terminal` `Gating` class) is the sole
engine delta, and the existing channel layer blocks/unblocks for free.

**The load-bearing reframing (the decision turns on this).** Blocking requires *no new
vocabulary*: `A.needs=[B]` already emits the dep edge B‚ÜíA, and `blocked_by` keeps the
non-`Terminal` predecessor (channels.rs). **New vocabulary is required only to author
gates *outbound* from records** ‚Äî and outbound is an ergonomic/product choice, not a
mechanical necessity. So **direction is the load-bearing decision; the M-Pr/M-E
"mechanism" fork is a *consequence* of choosing outbound**, not the primary axis. The
decisions reorder accordingly:

- **D-c ‚Äî authorship direction (the real fork). ‚Üí dependent-authored inbound `needs`.**
  *Dependent-authored*: work declares `needs ‚Üí record`; no new vocabulary; trinary
  partition + existing dep overlay supply blocking. *Record-authored (outbound)*:
  preserves "a record shows what it affects" and ADR-004 outbound authorship, but
  forces records into the dep/seq-**author** role (widen `is_work_like`, give records a
  dep table) ‚Äî a larger surface, and a *stored* duplicate of an inverted `needs`.
  **Chosen: dependent-authored.** "A record shows what it affects" is honoured by
  `shapes` (the semantic plane); outbound *gating* is a **derived reverse-query** (hub
  view) plus an optional deferred **batch sugar** that writes inbound edges ‚Äî every
  outbound ergonomic without the stored axis.
- **D-a ‚Äî mechanism (collapsed by D-c). ‚Üí reuse `needs`; no M-Pr/M-E.** Once
  dependent-authored wins, M-P/M-Pk (S2), M-L (RFC-003), M-Pr and M-E all dissolve ‚Äî
  the existing dep axis is the mechanism. (Had outbound won, this would have been the
  M-Pr-vs-M-E fork; M-E favoured, but moot.)
- **D-b ‚Äî transitivity (S3). ‚Üí direct-target only.** A gate is a direct `needs ‚Üí
  record`. Propagation along reference/implements (unsettled CON gating work via the
  spec it shapes) is a larger, separate design ‚Äî **deferred**.
- **D-d ‚Äî record‚Üírecord & cycles (S4). ‚Üí no record‚Üírecord gates initially.** Records
  are never workable (gating-inert); existing `CyclePolicy::Reject` guards the hazard.
  No special exclusion added.
- **D-f ‚Äî intra-record `shapes` semantics. ‚Üí dissolved for gating.** Gating never rides
  `shapes`, so the two-planes concern no longer entangles the gating model. `shapes`
  stays purely semantic; whether it wants a role to disambiguate record‚Üírecord
  (epistemic) from record‚Üíwork (affects) is a *separate, smaller* relation-model
  question (ADR-016), decoupled from this ADR.
- **D-e ‚Äî naming. ‚Üí `Gating`** (the third status-class; cosmetic).

**Caveat on "zero vocabulary."** Zero new *relation* vocabulary ‚Äî the delta is the
trinary partition. Inbound `needs ‚Üí record` is already admissible: a work item may
target a record (IMP-033 made `needs`/`after` cross-kind; the work-like gate on `needs`
constrains only the *source*), and records are already scanned graph nodes classified
by `partition.rs` (currently all-`Terminal`, which is exactly why gating cannot yet
bite). Implementation confirms the target-side resolve path for record kinds.

**Sequencing (ws1‚Üîws3).** ws1 (now reduced to the trinary partition) lands first as the
target; a minimal record-authoring path (ws3 ‚Äî the `/knowledge` authoring skill,
[IMP-182](../../backlog/improvement/182-knowledge-authoring-skill-route-agents-to-create-transition-asmdecquecon-records))
follows close behind to validate the gate against a real population. The mechanism is
not blocked on the full skill; the four self-dogfooding seeds suffice to prove the bite.

**Next:** [ADR-017](../../adr/017/adr-017.md) is accepted; **SL-158 unparks** against the
shrunk scope (trinary partition + the separate `shapes`-roles piece).

### Epistemic kind taxonomy

The gating mechanism gates records of specific kinds. A survey of the 262 active
doctrine memories and two concept maps (CM-001, CM-002) identified gaps in the
current four-kind taxonomy (ASM, DEC, QUE, CON): **observation** (settled empirical
truth) and **hypothesis** (testable proposed answer) as candidate kinds, plus an
open question about risk-as-knowledge. These bear on gating design ‚Äî different
kinds have different gating postures, and the `shapes` edge semantics vary by kind.
See **RFC-009** for the full analysis.
