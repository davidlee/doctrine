# RFC-008: Actionability gating: association vs graph-effect

## Context

<!-- The forces at play: the problem, the question, and why it merits structured
     discussion. An RFC asserts NO canon — it is governance-neutral deliberation. -->

IMP-047 / RFC-007 workstream 1 ("trinary actionability") proposes a third priority
status-class so that unsettled epistemic/governance records (PRD-010 / SPEC-019) can
**gate** downstream work without ever appearing in `next` as work. The partition half
is mechanical and settled (see SL-158 notes). The **gating-edge** half is not: it is an
unsettled graph-model decision that RFC-003's holistic relation-model review reopens,
with corpus-wide blast radius. SL-158 was opened to build it, hit this question in
`/design`, and is **parked** (`proposed`) pending this RFC. This RFC carries the context
across the several deliberation passes the decision needs; SL-158 resumes once it lands.

It may **emit an ADR** if a mechanism is chosen.

### The collision

- **IMP-047 / RFC-007** wording: "a record authors an outbound **gating edge**… new
  gating `RelationLabel`(s) + `RELATION_RULES` rows." I.e. gating as *vocabulary*.
- **RFC-003 Layer 1** (newer review): "whether an edge *gates work* is a **consumer
  decision, not a property of the relation**." Graph-effect is a *projection* over
  selected edges, declared in the consumer; actionability (`needs`/`after`/"IMP-047
  `gates`") lives in the **dep/seq layer, not the semantic relation model.**
- **RFC-003 design law**: *derivable, not relational* — don't encode in the relation
  what the consumer can project.

RFC-007 looks to have inherited IMP-047's older label-wording without reconciling it
against RFC-003. This RFC is that reconciliation.

### The requirement (the thing to protect)

Records already author `Shapes` (record → wide artefact set) as their durable semantic
"what this affects" relation — today **graph-inert**. The temptation is to make every
`Shapes` edge gate. The protect-this:

> **Association ≠ gating, and association must not be hostage to gating.** One must be
> able to associate an epistemic record wherever it is *semantically* sensible without
> that association forcing an *insensible actionability* effect.

A countervailing positive (so we don't over-correct): **gating canon is acceptable.**
Governance is not itself actionable, but an *unsettled* governance record transitively
gating the work that depends on it can be exactly right. So the goal is not "never gate
canon" — it is "let gating select *which* associations bite, by intent, not by accident."

## Discussion

<!-- The position(s) explored, pros/cons, alternatives considered. -->

### Shorthand (for the scenarios)

```
node state:  QUE-001(open)       record, UNSETTLED  → would-gate class
             QUE-001(answered)   record, SETTLED    → terminal, inert
             SL-020[design]      artefact, workable
             SL-020[BLOCKED]     artefact, blocked by a live gate

edges:       A ──shapes──▷ B     semantic association — relation model, NO graph-effect
             A ══gate════▶ B     actionability gate   — dep overlay; B BLOCKED while A unsettled,
                                                        auto-clears when A settles (Terminal)
```

The mechanical partition (unsettled record → would-gate class; settled → terminal) and
the channel layer (`eligible == Workable` excludes records from the worklist;
`blocked_by` keeps non-terminal dep-predecessors so a would-gate node blocks) are
**common to every option below** and are not in dispute. The dispute is **which `shapes`
associations become `══gate════▶` edges, and how that is decided.**

### Mechanism options

| id | mechanism | who decides a gate | layer (RFC-003) | assoc≠gating? |
|---|---|---|---|---|
| **M0** | status quo — records all-terminal, gate nothing | — | — | n/a (no gating) |
| **M-P** | blanket projection: every `shapes` gates while unsettled | nobody (automatic) | consumer | **NO — coupled** |
| **M-Pk** | projection filtered by **target kind** (only work kinds gate) | consumer heuristic | consumer | partial (by kind, not intent) |
| **M-Pr** | projection refined by a **role on `shapes`** (`gates` vs `informs`) | author, per edge | consumer + 1 role | **yes** |
| **M-E** | distinct **`gates` dep/seq axis** (orthogonal to `shapes`) | author, explicit edge | dep/seq | **yes** |
| **M-L** | distinct **`Gates` RelationLabel** + `RELATION_RULES` rows | author, explicit edge | vocabulary | yes — but RFC-003 says **wrong layer** |

### Scenarios — the discriminators

**S1 — the canonical gate (works everywhere).** An open question must block a slice's
design; answering it unblocks.

```
QUE-001(open) ──shapes──▷ SL-020        ⟶ project ⟶   QUE-001(open) ══gate════▶ SL-020[BLOCKED]
                                                       answer QUE-001 → (answered) → SL-020[design]
```
Every option M-P..M-L satisfies S1. M-P/M-Pk for free (shapes alone gates); M-Pr needs
`role=gates`; M-E/M-L need an explicit gate edge. **Not a discriminator** — it only
shows the partition+channel mechanics, which all share.

**S2 — shape WITHOUT gate (the killer).** A proposed decision *bears on* a risk
semantically, but must not block it.

```
DEC-005(proposed) ──shapes──▷ RSK-004[open]

  M-P  : RSK-004[BLOCKED]   ✗ couples association to gating
  M-Pk : RSK-004[BLOCKED]   ✗ RSK is a "work kind" → kind-filter too coarse, still gates
  M-Pr : RSK-004[open]      ✓ author wrote shapes(informs), no gate
  M-E  : RSK-004[open]      ✓ author added no gates edge
```
S2 **kills M-P and M-Pk** against the requirement: only *per-edge author intent*
(M-Pr / M-E) lets you associate without gating. Target-kind is the wrong axis — gate-ness
is an intent, not a property of the target's kind.

**S3 — gating canon, transitively (the positive — and a hidden cost).** An active
constraint shapes a spec; a slice implements the spec. User's intuition: the constraint
should gate the *dependent work*, not (uselessly) the canon node.

```
CON-003(active) ──shapes──▷ SPEC-019      SL-040[design] ──references(implements)──▷ SPEC-019

  direct-gate-the-canon-node:  CON-003 ══gate════▶ SPEC-019[BLOCKED]
     but SPEC-019 is canon — never workable anyway → blocking it changes NOTHING in `next`.
     SL-040 is NOT blocked: `references` is a CONSEQUENCE edge, not a dep edge — gating
     does NOT flow along it today.
```
S3 surfaces that **transitive gating of canon is not free.** Gating lives on the dep
overlay; `references`/`implements` are consequence overlays. To make "unsettled CON gates
the work that depends on SPEC-019," gating must either (a) target the dependent work
**directly** (`CON-003 ══gate════▶ SL-040`), or (b) **propagate** along reference/implements
edges — a new propagation rule with its own blast radius. **Open design axis, independent
of M-P..M-L.**

**S4 — intra-family (cycle hazard).** A held assumption shapes an open question.

```
ASM-002(held) ──shapes──▷ QUE-001(open)   [and possibly QUE-001 ──shapes──▷ ASM-002]

  M-P  : ASM ══gate════▶ QUE — but QUE is a record, never workable → worklist effect NIL;
         and if the reverse shape exists, a dep-overlay CYCLE (Reject policy) forms.
  M-Pk : record-target excluded → association only, no gate, no cycle.   ✓ (this is where kind-filter helps)
  M-Pr/M-E : author simply doesn't mark/author a gate → no cycle.        ✓
```
S4 shows record→record gating is mostly inert (records are never workable) but carries a
**dep-cycle hazard** under blanket projection. Per-edge intent or a record-target
exclusion both avoid it.

### Reading of the scenarios

- The **requirement (S2)** eliminates **M-P** and **M-Pk**: gate-ness must be author
  intent, per edge — not automatic, not target-kind-derived.
- That leaves **M-Pr** (role on `shapes`) vs **M-E** (distinct `gates` axis) vs **M-L**
  (label). **M-L** is disfavoured by RFC-003 (wrong layer — vocabulary). So the live
  fork is **M-Pr vs M-E.**
  - **M-Pr** keeps one relation (`shapes`) with a closed role `{gates, informs}`; gating
    is a consumer projection over the `gates`-role subset. Minimal vocabulary; rides
    SL-149's role machinery (ADR-016); RFC-003-clean (graph-effect stays in the consumer,
    intent rides a role). Risk: roles currently refine `references`, not `shapes` — extends
    the role grammar to a second label.
  - **M-E** makes `shapes` purely semantic and `gates` a separate dep/seq axis alongside
    `needs`/`after`. Fully orthogonal; "the dep/seq layer" taken literally. Risk: a second
    near-duplicate axis (`gates` vs `needs`) — a record's `gates` is structurally an
    inverted `needs`; coordinate with IMP-033 rather than fork.
- **S3 (transitive canon gating)** is an **orthogonal** axis to the M-fork: whichever of
  M-Pr/M-E wins, decide separately whether a gate (i) targets dependent work directly,
  (ii) targets the canon node and propagates along reference edges, or (iii) both. Direct
  targeting is simplest and avoids a propagation rule; propagation is more expressive but
  larger.
- **S4** argues for either a record-target exclusion or relying on per-edge intent to
  avoid record↔record gate cycles, regardless of the M-fork.

## Outcome

<!-- If resolved: what was concluded. If open: what remains unanswered. -->

**Open — deliberation seed.** Diagnosis and the option space are framed; no mechanism is
chosen. Decisions to converge (in roughly this order):

- **D-a — coupling & locus (the core fork).** M-Pr (role on `shapes`) vs M-E (distinct
  `gates` axis). M-P/M-Pk rejected by S2; M-L disfavoured by RFC-003. *Leaning: resolve
  between M-Pr and M-E; both satisfy the requirement.*
- **D-b — transitivity (S3).** Does a gate target dependent work **directly**, target the
  canon node and **propagate** along reference/implements, or both? (Independent of D-a.)
- **D-c — direction.** Outbound-from-record (ADR-004-clean, "a record shows what it
  affects") vs dependent's `needs → record`. *Leaning outbound-from-record.*
- **D-d — record→record & cycles (S4).** Per-edge intent alone, or an explicit
  record-target exclusion, to keep the dep overlay acyclic.
- **D-e — naming.** The third status-class: `Gating` / `Ambient` / `Pending` (cosmetic;
  last).

**The decision is design-led, not census-led — the population is empty.** As of
2026-06-26 there are **zero knowledge records** and **zero `shapes` edges**: the
ASM/DEC/QUE/CON kinds are recently added (SL-059) and have **no authoring skill**, so
none get created. Unlike RFC-003's P1 (which classified a live 113-edge population),
RFC-008 has nothing to classify — converge on the canonical pairs (QUE→SL, ASM→IDE,
CON→REQ/SL, DEC→ISS) and forward judgement. Re-run a `relation census` of `shapes` once
records exist, to *confirm* (not to drive) the choice.

**The ws1↔ws3 chicken-egg is load-bearing here.** This gating mechanism (ws1) gates on
records that do not yet exist and cannot yet be ergonomically authored (ws3 — no
skill). So RFC-008 must also weigh **sequencing**: does the mechanism land first (giving
record-authoring a target), or does record-authoring (ws3) come first (giving the
mechanism a population to validate against)? They are mutually reinforcing; pick a
deliberate order rather than assuming ws1 leads.

If D-a/D-b/D-c converge, **emit an ADR** (gating layer + edge model) and unpark SL-158
to implement it.
