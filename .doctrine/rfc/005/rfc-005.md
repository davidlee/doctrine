# RFC-005: Dispatch funnel integrity: hazard survey and correctness posture

## Context

The dispatch funnel — spawn workers on isolated worktrees, import their deltas
onto a coordination branch, verify, cut a review bundle, and integrate onto the
trunk — is the highest-throughput change path in Doctrine. It runs under
concurrent multi-agent load on a shared single clone. Correctness hazards in the
funnel are high-leverage: a silent wrong-base worker run, missing ledger commit, or
dicey orchestrator integration merge undermine the contract the funnel offers.

Worker provisioning has, since inception, a top-level fork (SL-056): two arms, one
for Claude Code subagents, and another for every other harness which runs as a
subprocess (equivalent to `claude -p` - now billed at API rates, hence excluded).

The Claude arm has been by far the most painful to work with. This RFC exists
to a) survey the remaining dispatch hazards as a whole, and b) to re-examine the
decisions and evidence which inform the current approach, seeking a simpler
and more robust overall design.

## Current posture (2026-06-27)

This RFC is **working memory** for dispatch-funnel correctness — it tracks what's
live, not the full history. The per-class survey below is kept as a decision
ledger; read it for *why*, not for *what's next*. What's next is here.

### H1 — Base integrity: worker lands on the wrong (or moving) base

> **[SL-152 — RESOLVED, eliminated.]** SL-152 built the `WorktreeCreate`-replace
> creation (phases 01–05 landed, proven live on 2.1.181). Doctrine is now the
> creator — no native creation remains to fall back to, so the race below
> **cannot occur**. The reframe/cost/structural-question subsections are the
> ledger of the decision; their open posture is superseded. Close pending SL-152
> close (only the droppable PHASE-06 plugin packaging remains).

**The hazard.** Under multi-agent git-lock contention, the claude dispatch arm's
`isolation: worktree` silently falls back to the main worktree, where
`baseRef: "head"` then tracks a moving `main`. The worker executes on a wrong,
dirty, moving base instead of the coordination tip B.

**Evidence.** ISS-034 (closed, mitigated) is the canonical incident — 3
consecutive wrong-base spawns on SL-121 PHASE-02 while other agents advanced
`main`. The third attempt showed the worker's working directory **clobbered to
`main` mid-run** (first reads: correct isolated base `bbfc90bb`; moments later:
`63d96f90` on `main`, dirty with unrelated files, `merge-base --is-ancestor
bbfc90bb HEAD` → false).

**Current posture.** SL-123 hardened the post-run `verify-worker` belt:
`not-isolated` / `branch-mismatch` detection + pre-funnel footer gate now turn a
contention-induced fallback-to-main into a **loud, pre-import halt** — fail-closed,
not fail-safe. The isolation race itself is unchanged; throughput under churn is
zero (worker spawns, fails, must be retried or routed to the subprocess arm).

**Reframe (2026-06-25, wtc-probe — H1 is collapsible, not just mitigable).** The
race exists only because the **harness** creates the worker's worktree. An
empirical probe on claude-code **2.1.181** proved a `WorktreeCreate` hook **fully
replaces** native creation for an `Agent isolation:worktree` spawn: a named
`dispatch-worker` landed at a **doctrine-chosen base** (`68250bcd`), overriding
`worktree.baseRef="head"`, in doctrine's own worktree path. When **doctrine is
the creator** there is no native creation to fall back to — the fallback-to-main
that *is* H1 **cannot occur**, and a hook failure aborts the spawn fail-closed
(the only hook event where any non-zero exit blocks). This makes H1 a **fixable
mechanism defect**, not a permanent harness tax to route around. Evidence +
residual: [[mem.pattern.dispatch.worktreecreate-replace-base-control]].

**Cost (probed, not fatal).** The `WorktreeCreate` payload is thin (no
`agent_type`/base/path) and the **matcher does not scope** — the hook fired for a
`general-purpose` subagent too, so it is **repo-global**, intercepting every
`isolation:worktree` spawn (ADR-011 D7 σ blast-radius). Discrimination must be
**out-of-band**: an orchestrator-dropped marker (sole-writer, serial → race-free)
selects `doctrine worktree fork --worker` at base B; absent a marker the hook
passes through (replicate default creation) so benign isolated subagents still
work. This also folds H3 (same fork tooling both arms) and H4 (provision inside
the hook) into one seam.

**Structural question.** Build the `WorktreeCreate`-replace creation now (collapses
H1 + converges H3 + provisions H4), or hold at SL-123's post-run belt? The
wasted-worker-run residual (SL-123 §5.2 / OQ-2) plus the now-proven mechanism tips
toward building. Carried by IMP-072 (re-scoped) → see OQ-4.

> **[SL-152 — answered: built.]** Built, not held. SL-123's post-run belt is
> retained as rarely-fired defence-in-depth (SL-152 non-goal: not removed). The
> wasted-worker-run residual is gone — the hook aborts fail-closed *before* the
> first worker command.

**Sibling memory.** `mem.signpost.doctrine.dispatch-claude-arm-wrong-base`
codifies the base-guard pattern (grep prerequisite seams + assert clean tree +
STOP on mismatch) and the recommendation to prefer the subprocess arm under
active `main` churn.

### H2 — Integration integrity: the trunk integration step has correctness hazards

**The hazard.** `dispatch sync --integrate --trunk refs/heads/main` advances the
trunk ref. Historically it operated against the **live shared checkout** with no
leg-awareness and no pre-gate, so it could move the ref without syncing the
checkout — leaving the index holding the slice's code as **staged
reverse-deletions**; a later unrelated commit rode that stale index and
**silently reverted the integration**. Witnessed on SL-122 close (slice code
briefly vanished from `main`). A second, independent failure mode:

- **Shared-trunk race (mem close-integrate).** The FF-only + expected-tip CAS
  races when another agent commits to `main` between `candidate create` and
  `sync --integrate`. FF refusal is correct but the agent is blocked until trunk
  stabilizes or a superseding candidate is created. The close driver cannot
  self-unblock when the dirt is another agent's uncommitted work. Unchanged;
  workaround is chaining `create→admit→integrate` in one shell to shrink the
  window.

**Verification (pass 1 — `research-ISS-038.md`, HIGH confidence).** A code+history
trace re-baselines the phantom and downgrades it from "open, unmitigated":

- **The SL-122 incident rode a stale binary.** SL-121 PHASE-02 landed the
  integrate rework (commit `426f024e`, 11:30) *before* the incident (16:48); the
  binary that ran predated it (shared `CARGO_TARGET_DIR` footgun, AGENTS.md
  `just rebuild-stale`). The phantom was created by the **old** `replay_ref`
  pure-CAS path (no leg-awareness, no pre-gate) — **not** the current mechanism.
  The proximate cause was as much **verification hygiene** as mechanism defect.
- **The tracked-dirty chain is now closed.** Current integrate
  (`dispatch.rs:1548-1740`) branches on checkout state (`worktree_for_ref`):
  checked-out → `ff_advance_in_worktree` (FF-merge syncs ref+index+tree
  atomically, no phantom; proven `git.rs:3803-3827`); not-checked-out →
  `advance_pure_ref` (pure CAS, nothing to desync). The **M4 dirty pre-gate**
  (`dispatch.rs:1599-1611`, SL-121) bails the whole integrate before any ref
  moves if a checked-out target is **tracked-dirty**.

**Residuals (still open).**

| # | Residual | Locus | Severity |
|---|----------|-------|----------|
| R1 | None-leg `RacedDesync` phantom — a checkout materializes dirty in the CAS→re-probe window; the ref has already advanced, so it warns rather than refuses | `dispatch.rs:1700` | **low** likelihood × **high** impact |
| R2 | `/close` carries the ISS-030 detector but **no recovery procedure** (corpus-wide search: none) | `close/SKILL.md:88-103` | **medium** — detector fires, agent left without remedy |
| R3 | IMP-122 F-1: re-resolve `target_ref` before `reset --hard` (else a concurrent advance is silently clobbered) | `dispatch.rs:~1698` → `resync_worktree_hard` | low × medium |
| R4 | IMP-122 F-2: untracked-collision guard before `reset --hard` (unlike `merge --ff-only`, it overwrites silently) | `git.rs:1274-1277` | low × low |

R1/R3/R4 are all **None-leg resync** hazards. R3/R4 = IMP-122 (hazards *during*
the resync, after the phantom exists) — genuinely separate from R1 (the phantom
itself). All three dissolve if integrate becomes checkout-independent (OQ-5). R2
is a cheap, independent skill fix. The M4 pre-gate is tracked-only
(`tree_clean` uses `--untracked-files=no`); untracked dirt is by-design excluded
and is not a phantom vector on either leg.

**Structural question.** With the tracked chain closed and the residual a
low-likelihood None-leg window, is the checkout-independent rewrite (OQ-5)
warranted for R1's impact + the R3/R4 dissolution — or is proportionate
hardening (close R2; land the IMP-122 guards; add a None-leg pre-CAS mid-window
gate) the right altitude?

**The hazard.** The two dispatch arms (claude `isolation: worktree` vs subprocess

failure modes. Several defects are arm-specific, creating an uneven correctness
surface:

- **Claude-only:** ~~isolation race / wrong base (H1)~~ (SL-152: resolved),

  ~~`WorktreeCreate` hook not yet wired (IMP-072)~~ (SL-152: wired).
- **Subprocess-only:** deterministic fork, no fallback-to-main, but codex/pi
  workers instead of claude.
- **Both arms:** gitignored build artifact provisioning gap (H4).

**ISS-039** (resolved by SL-154) was a concrete arm-specific defect: the claude
arm wrote `boundaries.toml` to the live coordination worktree but never committed

**Structural question.** Should the two arms converge on a single, deterministic
provisioning path (the subprocess model), or should the claude arm's unique
capabilities (in-harness workers) justify maintaining two paths with
arm-specific hardening?

### H4 — Worker provisioning: gitignored build artifacts absent from forks

> **[SL-152 — partially eaten.]** The claude **worker fork** is now provisioned
> inside `create-fork` (same `.worktreeinclude` copier as the subprocess arm), so
> the worker-fork leg below is closed on both arms. The **coordination-tree** gap
> (`web/map/dist/` compile failure) and the **general-provisioning** question
> (IDE-017, Tension 2) are untouched and survive.

**The hazard.** `doctrine worktree fork --worker` and `dispatch setup`
coordination worktree both provision only tracked files + the `.worktreeinclude`
allowlist. Gitignored build artifacts (`web/map/dist/` for RustEmbed,
`web/map/node_modules/` for JS lint) are absent. Consequences:

- **In worker forks:** `map_server` runtime tests fail (empty embed); `just gate`
  fails on `lint-js` (no eslint). Funnel verify inside the worker fork is
  unreliable.
- **In coordination tree:** `cargo build` fails on missing `web/map/dist/`
  (`RustEmbed` compile error), blocking the entire funnel verify step even for
  Rust-only deltas.

**Current posture.** CHR-017 (closed) addressed `node_modules` via the
skip-loud guard pattern (option A). The `web/map/dist/` gap is worked around by
copying the prebuilt artifact from the main worktree into the coordination tree
before verify. IDE-017 (open, idea) captures the latent gap: if
`.worktreeinclude` ever grows to include per-worktree-divergent untracked state,
the current stamp-provision mechanism (sourced from primary worktree) cannot
supply it.

**Structural question.** Is per-dep skip-loud guards sustainable, or do two
distinct gitignored targets (plus the general class) argue for a general
provisioning answer — symlink or copy gitignored deps into forks, as CHR-017
option B/B′ proposed?

### H6 — Silent destructive integration + unenforced checkout discipline

> **NEW (2026-06-27), apex correctness.** The silent-destructive sibling of H2.
> Where H2/ISS-038 silently *reverts code*, H6/ISS-056 silently *deletes the
> authored corpus*. Surfaced live on the SL-164 drive; nothing was permanently
> lost (reflog + `origin/edge` intact) but both branch tips and the working
> trees were corpus-less until hand-repaired.

**The hazard (ISS-056).** A `/dispatch` drive whose coordination branch forks
from a base that **predates the authored `.doctrine` corpus** produces phase
commits carrying a corpus-less tree. Merging/promoting that bundle onto a
corpus-bearing trunk (`edge`) records a **clean deletion of the entire corpus**
(4816 files on SL-164) — **no conflict, no abort, no diagnostic**. The
ISS-036 setup gate did *not* fire: the stale base held the slice's own *plan*
(so the plan-presence assert passed) yet lacked the broader corpus. The
existing FF-only / leg-aware integrate guards did not fire either: by the time
`integrate --trunk main` ran, `main` had already been promoted *back* to the
stale base, so the FF was clean *relative to main's then-current tree* — the
deletion manifested in the **manual `git merge` onto the live `edge` tree** and
the **`git fetch . edge:main` promotion**, neither of which is a doctrine verb.

**The shared root with H2 (the load-bearing finding).** The funnel's safety
model assumes **`main` is never checked out** and **the primary worktree stays
on `edge`** — the same etiquette SL-157 banked on to delete R1's defense. That
invariant is **behavioral and unenforced**, and agents violate it: on the SL-164
drive an agent (deepseek) switched the primary worktree between `main` and
`edge` four times while running raw git. `never` is not `never`. Every H6 guard
must therefore be **mechanism-level — hold regardless of which branch is
checked out in which worktree, and regardless of raw git run on a live tree.**

**Why distinct from the family.**

- **ISS-038 (H2)** — *dirty shared checkout → phantom index → silent revert of
  **code*** (corpus survived). H6 is the opposite half: *stale fork base →
  corpus-less bundle → silent deletion of the **corpus*** (code survived). Same
  outcome shape, opposite halves, different mechanism, different fix.
- **RSK-010 / SL-127** closed the *loud* stale-base variants (ancestor-dominant
  ladder + mid-drive refresh — the *conflict* and *hard-abort* paths). They do
  not detect a base fresh enough to hold the slice's plan yet stale enough to be
  missing the rest of the corpus.
- **ISS-036** hard-aborts setup when the base lacks the slice's *own plan* —
  one notch short of the corpus check H6 needs.
- **SL-157** shipped the integrate strip but left R1 re-openable (above) and
  never touched base-freshness, corpus preservation, or the promotion ritual.

**Candidate fixes (mechanism-level, defense-in-depth).**

| # | Guard | Surface | Catches | Notes |

## Discussion

### The correctness perimeter

The dispatch funnel's correctness contract is: **worker runs on a stable base B,
delta is imported and verified, integration is atomic and lossless.** The four
hazard classes H1–H4 each attack one link in that chain:

```
spawn → [H1: wrong base] → execute → import → verify → [H4: missing artifacts]
→ [H3: arm asymmetry / missing ledger] → review → integrate → [H2: phantom revert]
```

Today (**post-SL-154**), H1 is **resolved** (creation-seam fix — was fail-closed
under SL-123, now eliminated), H2 is **partially mitigated** (SL-121 closed
the tracked-dirty phantom chain; a low-likelihood None-leg residual R1 and the
missing `/close` recovery R2 remain — see H2 verification), H3 is **resolved**
(SL-154 closed the last residual, the ISS-039 ledger commit), and H4 is closed on
the worker-fork leg but still has tactical workarounds and no general answer for
the coord-tree leg. The funnel is **operationally usable** (many slices have
shipped through it) but **not formally robust** — H2 integrate remains the open
structural hazard; correctness depends on environmental conditions
(no concurrent `main` commits during integrate, build artifacts pre-copied, the
right arm chosen for the load, **and a current binary** — the SL-122 phantom was
a stale-binary artifact as much as a mechanism defect).

**Headline re-rank (post-verification).** The survey originally cast H2 as the
sole unmitigated correctness hazard. Verification moves it off the top: the
biting, recurring risk is **H1** — the wrong-base isolation race is fail-closed
but **zero-throughput under `main` churn** (observed failing 3× consecutively),
an active operational tax every busy drive pays. H2's live exposure is now a
narrow None-leg window (R1) plus a cheap skill gap (R2). The largest *structural*
lever remains OQ-5 (checkout-independent integrate dissolves R1/R3/R4 at once),
but as a *present* risk H1 > H2.

> **[SL-152 → SL-154 — re-rank inverted, then confirmed.]** SL-152 eliminated
> H1, reversing the "H1 > H2" ranking above: **H2 is the top open structural
> hazard**, and **OQ-5** (checkout-independent integrate) is the live lever.
> SL-154 closed H3 without touching H2, so the apex is unchanged — H2/OQ-5 stands
> alone.

The trace also surfaced **stale-binary
verification hygiene** (a shipped fix that did not run), but that is a
**project-local** wart (shared `CARGO_TARGET_DIR` in this repo's jail), **not** a
platform hazard — it does not bear on doctrine-as-a-platform correctness and is
deprioritised accordingly; see Tensions.

### Coherence with RFC-004 (the funnel is RFC-004's capture site) — 2026-06-26

The dispatch funnel is where **RFC-004's** path-intent selector primitive meets
this RFC. RFC-004's `slice conformance` consumer diffs *declared* design-target
selectors against the *actual* git delta; the actual side is the **conformance
registry** (`state/slice/NNN/boundaries.toml`), and one of its two writers is the
funnel — meant to deposit one boundary row per landed phase at `prepare-review`.
So a correctness link the H1–H4 chain did not name: **after `verify`, before
`integrate`, the funnel must record the slice's actual delta**, or RFC-004's
audit consumer fail-closes downstream.

That write was never firing (**ISS-052**) for the same root cause as **H3's
ISS-039**: the dispatch `boundaries.toml` was not committed to `dispatch/NNN`, so
the derive read an empty ledger. **ISS-039 was the shared joint** — H3 saw it as
lost per-phase review cuts, RFC-004 as an empty conformance registry. **SL-154
(done)** fixed both with one move (commit the ledger to the branch, SPEC-022
conformance) and made the funnel conformance-registry derive + completeness gate
an **enforced `prepare-review` beat** (was a documented-but-skippable
`record-delta` step). The funnel now carries a recorded-delta obligation, keyed
on **landing path (funnel vs solo)**, not the now-collapsed arm axis. The
solo-side twin (**ISS-051**, final-phase off-by-one) was the other leak SL-154
closed. (Codex/pi symmetric ledger deferred — **IMP-171**.) This joint is now
closed: `slice conformance` runs at audit without a manual bootstrap.

### What's structural vs incidental

- **Structural.** The shared-single-clone model. Git's repo-global locks mean
  concurrent worktree operations contend. The claude arm's `isolation: worktree`
  + `baseRef: "head"` architecture makes it inherently vulnerable to `main`
  movement. The integrate step's dependence on live checkout state (not a
  dedicated clean worktree) is a mechanism choice, not an inherent limitation.
- **Incidental.** Specific missing commits (ISS-039 boundaries.toml), specific
  missing gitignored paths (`web/map/dist/`), specific config gaps. These are
  fixable one-by-one but the accumulation suggests the funnel's provisioning and
  ledger models need a systematic review.

### Tensions

1. **Claude arm capability vs reliability.** The claude arm gives in-harness
   workers (higher quality) but has unique failure modes (isolation race,
   missing ledger commits). The subprocess arm is deterministic but uses
   codex/pi workers. Is the right posture: harden the claude arm to subprocess
   reliability, or accept the asymmetry and route by load?
   **[SL-152 — resolved: harden to parity.]** The creation seam is now shared;
   the claude arm keeps in-harness workers *with* subprocess-grade base
   determinism. Residual asymmetry is ISS-039 (ledger commit) only.

2. **Per-dep guards vs general provisioning.** CHR-017 chose skip-loud for
   `node_modules`. Two distinct gitignored targets now exist. Each new
   gitignored dep adds a guard. At what N does a general provisioning answer
   (`.worktreeinclude` symlink/copy) become cheaper than N guards?

3. **Post-run vs pre-run fail-closed.** SL-123's post-run belt catches wrong-base
   before import but after a wasted worker run. IMP-072's pre-run hook would
   abort before the first command. Is the wasted-run residual acceptable at
   current churn rates, or does it block throughput?
   **[SL-152 — resolved: pre-run.]** The hook aborts before the first worker
   command; the post-run belt is retained as rarely-fired defence-in-depth. No
   wasted-run residual.

4. **Integrate checkout independence.** ISS-038's tracked chain is closed
   (SL-121 M4); the open residual is the None-leg `RacedDesync` window (R1) plus
   the IMP-122 resync guards (R3/R4). The structural fix — integrate via a
   dedicated clean worktree or pure-ref CAS — dissolves all three at once but is
   a mechanism change, not a guard. Is it warranted for a low-frequency window,
   or do the proportionate hardenings (R2 recovery + IMP-122 + a None-leg
   pre-CAS gate) suffice?

5. **Stale-binary verification hygiene (project-local, low priority).** The
   SL-122 phantom's proximate cause was a stale binary running old integrate
   code while the fix sat committed but unbuilt (shared `CARGO_TARGET_DIR`). It
   is a real *verification-integrity* trap — any "is it fixed?" judgement is only
   as good as the binary under test — but it is **local to this repo's jail
   build setup**, not a property of the platform. It does not bear on
   doctrine-as-a-platform correctness, so it ranks well below the harness- and
   mechanism-level hazards. Cheap local mitigation only (`just rebuild-stale`
   reflex; optionally a build-freshness assertion in the close gate).
