# RFC-011: Dispatch Token Efficiency

## Context

Worktree & Dispatch execution are necessary for parallel throughput, but
- they impose their own overheads: orchestration (orientation, setup,
teardown, worker context preparation, review / correction). They also
increase the complexity (hence cost) of the audit, reconcitiation and
close stages.

This RFC seeks to find ways to minimise this cost, end to end - not just 
relative to solo execution, but holistically. Efficiency is one of the 
core design goals for doctrine - less important than e.g. correctness, 
but still a critical value driver.

## Discussion

### Cost model (corrected)

A slice's end-to-end token cost decomposes as:

```
total ≈ Σ(worker phase-work)  +  orchestration  +  back-end
```

- **Worker phase-work** — the actual implementation, in the worker's own
  context. **~Constant across drive types**: the same code gets written whether
  driven solo, lean-serial, or via dispatch. Switching drive type does not move
  this term.
- **Orchestration** — the driving agent's own context (verify, lifecycle flips,
  context prep, funnel ceremony). Measured **comparable across drive types**:
  - lean-serial: SL-166 P3–P5 = 148k orchestration / 3 phases ≈ **~49k/phase**
  - dispatch (estimate): ~120–200k / 3–5 phases ≈ **~40k/phase**
- **Back-end** (audit + reconcile + close) — **~115–141k/slice**, i.e. ≈ one
  extra session for a large slice (SL-166 audit 115k; SL-155 141k; SL-163 ~130k).

**First conclusion: the drive *mechanism* is roughly token-neutral.** Lean-serial
is not materially cheaper than dispatch on orchestration, and phase-work is fixed.
An earlier "2.5–3× cheaper" reading was an artefact of comparing lean
*orchestration-only* against dispatch *orchestration-plus-worker-sessions* —
retracted. Drive-type choice is a **rigour / autonomy / parallelism / human-cost**
decision, not a token lever.

### Where tokens actually bleed: orientation

First instrumented run (SL-166 lean-C drive; full log in `case-notes.md`) shows
the cost is not execution but **orientation friction**, recurring across 4
sessions. The worker's own verdict: *"the expensive parts were all in
orientation, not execution."* The recurring cluster (call it **L0**):

| Friction | Recurs | Fix |
|---|---|---|
| [done] CLI id-form split (`slice status/conformance/phases` want bare `166`; `show`/`review` want `SL-166`) | 4+ sessions | accept both / clear error — **IMP-189** |
| `slice status` reader/writer overload (no bare query form) | 4+ | `slice status <ID>` prints state + phase rollup |
| [done] Stale boot signpost `doctrine paths` (no such verb) | 2+ | fix `boot.md` "useful commands" |
| Solo/fork runtime phase-state never reaches primary → close mis-reads X/X | 3+ | close/audit recipe for stale-rollup; suppress spurious binding-capture warning |
| [transient] Design premise vs moved target (SL-146 relocated config the same day; P5's biggest single spend) | 1 (canonical) | **design-lock re-greps touched constants/paths at lock time** |
| `/audit` silent that review verbs refuse a worktree fork (constraint only in `review-ledger.md` §6) | 2+ | one-line `/audit` pointer: land worktree slice on parent before opening the RV |
| [done] Bin-crate test invisibility (`cargo test --lib` empty; logic in the bin) | 1 | `just test` recipe + AGENTS.md note |

These are cheap, independent, and hit **every orchestrator and every worker
context, under every drive type**. This is the genuine token lever.

### The reframe: cold-small contexts + compact durable handoff, not warm-large

The obvious "saving" — keep a worker warm and reuse it (`SendMessage`) to skip
re-orientation — is usually a **trap**:

- **Dumb zone** — answer quality decays as the context fills; a fresh small
  context often reasons better than a warm bloated one.
- **Cost curve** — beyond ~120k tokens, per-turn cost climbs steadily; prompt
  caching blunts but does not flatten it.
- **Hard to heuristic** — teaching a model *when* to reuse vs refresh is itself
  non-trivial and error-prone.

So the efficient shape is the **opposite of warm reuse**: many **cold, small**
contexts, each re-orienting *cheaply* off **compact, durable, stage-targeted
state**. Doctrine already embodies this principle — the progressively-built RV
ledger, notes, handover, and phase sheets *are* the durable carry between stages.
The lever is to make that state **tight, single-copy, freshness-marked, and
accretive**, and to make re-orientation off it cheap (which is exactly L0).

This sharpens the priority rather than softening it: because cold re-orientation
is paid *often and by design*, minimising its **unit cost** (clean CLI, accurate
signposts, compact handoff artefacts) is the whole game. It also re-frames the
old "warm reuse" lever: agent-teams / `SendMessage` is now available
(project-local, via `.envrc`), so warm reuse becomes **testable** — but the prior
is against it; treat as an experiment, not a default.

### Drive types (reclassified — token-neutral, choose by rigour/autonomy)

| Drive | Parallel | Rigour / audit | Human | Wins when |
|---|---|---|---|---|
| **Solo** | no | high (human-in-loop) | babysits + restarts | tiny / exploratory, human wants the wheel |
| **Lean serial** (shared tree, no `isolation: worktree`) | no | medium — self-prep, no funnel, no per-phase RV/candidate | fire-and-forget | serial-dependent phases, moderate rigour bar |
| **Dispatch** | yes | high — funnel + RV ledger + per-phase isolation | orchestrates | parallelizable phases, high rigour, large slice |

Lean-serial is validated empirically (SL-166 P3–P5 autonomous, clean delegation,
2-call verify per phase) and, by dropping `isolation: worktree`, sheds the
bucket-3 deadweight (fresh-fork artifact provisioning / IMP-187) entirely. For
serial-dependent phases, dispatch's parallelism buys nothing — so lean-serial is
the right rung there, paid for in auditability.

### Levers (re-ranked under the corrected model)

**Token levers (the real efficiency play):**

1. **L0 — orientation hygiene.** The recurring cluster above. Cheap, independent,
   every-context, every-drive. Highest aggregate payoff. (IMP-189 + siblings.)
2. **L2′ — compact durable handoff state.** Tighten the cold-start surface
   (ledger / notes / handover): single-copy, freshness-marked, progressively
   accreted, scoped to "what the next stage needs." Ties to **IMP-059**
   (unify the cross-corpus harvest) and **IMP-060** (relocate `/handover` into
   core). *Replaces* the old warm-reuse framing.
3. **L0e — design-lock re-grep.** Kill the doc-vs-moved-target spike: at design
   lock, re-grep the touched subsystem's constants/paths rather than trusting the
   author-time mental model. Highest *per-incident* worker cost.

**Capability / cost-profile levers (token-neutral — do for ergonomics & rigour):**

4. **L5 — promote lean-serial to a first-class drive type.** Named mode with
   defined rigour guarantees (what audit trail it *does* keep), so it is
   discoverable and selectable. Validated; cheap; autonomous.
5. **L3 — drive-type selector.** Route each slice to the cheapest drive that
   clears its rigour × parallelism bar. Gated on more runs.

**Demoted:**

- **L2 (warm reuse)** — experiment only; the prior is against warm-large in favour
  of cold-small + compact state.
- **L1 (fork provisioning)** — moot on the lean path; matters only for the
  dispatch/candidate worktree path, where the candidate creator
  (`add_candidate_worktree`) currently skips `run_provision` (one-call fix). Cheap
  hygiene, small token line.

### Low-hanging fruit (ship soon, mostly one-liners)

- id-form: accept both prefixed + bare, or a clear "expected bare slice number"
  error (IMP-189).
- `slice status <ID>` read-only query form (state + phase rollup).
- fix the stale `doctrine paths` signpost in `boot.md`.
- close/audit recipe for solo/fork stale runtime-rollup; silence the spurious
  `phase-binding capture skipped` warning on that legitimate path.
- `/audit` one-line pointer: land a worktree-built slice on the parent tree before
  opening the RV (review verbs refuse forks).
- `just test` recipe targeting the bin crate + AGENTS.md note.
- route candidate-worktree creation through `run_provision` (L1).

### Open questions

- **OQ-1** design-lock re-grep — mechanism (skill step? lint?) and scope (which
  constants/paths count as "touched").
- **OQ-2** compact-durable-state — the minimal "next-stage brief" shape, and its
  relation to the existing ledger/notes/handover unification (IMP-059/060).
- **OQ-3** L3 selector heuristic — needs more instrumented runs and a measured
  worker phase-work baseline before a threshold can be drawn.
