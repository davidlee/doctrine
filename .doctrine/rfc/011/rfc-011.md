# RFC-011: Dispatch Token Efficiency

## Context

Worktree & Dispatch execution are necessary for parallel throughput, but
- they impose their own overheads: orchestration (orientation, setup,
teardown, worker context preparation, review / correction). They also
increase the complexity (hence cost) of the audit, reconcitiation and
close stages.

This RFC seeks to find ways to minimise this cost, end to end - not just 
relative to solo execution, but holistically. Efficiency is one of the 
core design goals for doctrine - less important than e.g. correctness, 
but still a critical value driver.

## Discussion

### Cost taxonomy Бқ” where the tokens go

End-to-end dispatch cost decomposes into four buckets:

1. **Orchestrator fixed cost** Бқ” orientation (boot/route/canon/retrieve-memory),
   fork + provisioning, per-worker context-packet authoring, the
   importБ†’verifyБ†’branch-pointБ†’commitБ†’record funnel, review/correction. Amortised
   across phases; SL-085 already pushed the drive loop into the CLI to shrink the
   skill body.
2. **Worker variable cost ц— N** Бқ” each worker *re-pays* orientation (boot + route +
   memory retrieval + design/plan/phase-sheet read) before doing any work. The
   multiplied tax: N workers = N near-identical orientations.
3. **Environmental waste (pure deadweight)** Бқ” fresh forks lack gitignored
   build/embed artifacts Б†’ spurious compile/gate failures unrelated to the change,
   re-discovered each run. Evidence: IMP-187, `memБқ¦eec32`, `memБқ¦eac33`,
   `memБқ¦eeac33` (coord-tree variant), `memБқ¦f0715` (reflink pre-warm). Same class,
   paid repeatedly, on both worker and coordination trees.
4. **Back-end amplification** Бқ” audit/reconcile/close cost rises *because* dispatch
   multiplied branches/candidates/GC-timing (`memБқ¦ed653` coord branch GC'd before
   audit; `memБқ¦ee369` fold fix-now before close).

### What the corpus already settles

- Dispatch only pays above a throughput threshold; authoring entities cannot be
  fanned (`memБқ¦eb5eb`). A dispatch-worth-it gate is itself a token lever.
- Bucket 3 is solved in principle, unshipped: provisioning copies generated
  embed/build dirs (IMP-187 opt 1) or pre-warms the fork via reflink (`memБқ¦f0715`).
  Highest leverage, lowest risk Бқ” deadweight, no judgment tradeoff.

### Candidate levers (leverage / risk)

| # | Lever | Targets | Risk | Decidable now? |
|---|---|---|---|---|
| L1 | Fork provisioning of generated artifacts (IMP-187) | bucket 3 | low | yes Бқ” corpus unanimous |

| L4 | Back-end simplification (branch/candidate juggling) | bucket 4 | med | needs evidence |

### Meta-question Бқ” decide now vs instrument first

instrumented run yet Б†’ no empirical data). L1 is decidable now from the corpus.
L2Бқ“L4 carry judgment tradeoffs and benefit from accumulated case-notes evidence.
