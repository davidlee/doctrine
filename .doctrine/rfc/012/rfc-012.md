# RFC-012: Parallel Fan-out Safety: Containment, Disjointness, and Shared Responsibility

## Context

Dispatch fans a slice's phases out to **N worker instances** running
concurrently against one shared clone. The confinement question — *can a
malicious worker escape its worktree and write the host?* — is owned by RSK-014
and being graduated by SL-182 (Linux/bwrap Claude arm). This RFC addresses the
other failure mode, the one that survives a perfect jail:

> Multiple **honest** workers, every one doing permitted work, collide on shared
> resources that were safe or invisible in a single-instance workflow.

Two workers bind `:3000`. Two write `/tmp/app.pid`. Two share a dev DB schema, a
Redis namespace, a named volume, a durable daemon socket reachable outside the
mount namespace. One worker's branch-specific cache contaminates another's run.
None of these is an escape; all are direct consequences of fan-out itself.

Doctrine is the **only layer that knows N workers are about to run**. The project
chose its runtimes and resources blind to concurrency; the operator reasons about
one branch at a time. So fan-out safety cannot be wholly delegated downward — and
it cannot be wholly absorbed either, because Doctrine does not own the semantics
of a project's database. The contribution of this RFC is the **cut**: which
hazards Doctrine owns by construction, which it must detect-and-warn, and which
remain the project's to decide.

**Standing of this RFC.** This is a *leading position* — a vision and a
responsibility contract — not governance. It asserts no canon. ADRs and standards
**precipitate downstream**, as the relevant pieces are adversarially tested and
locked in slice design and emitted at reconciliation (the same way SL-182 is
hardening the confinement floor that RSK-014 first proved). The policy schema and
capability ladder below are the **target shape**, deliberately subordinate to
what is materialised today.

## Discussion

### 1. The governing principle: the ownership cut

Apply one test to every resource:

> Did the hazard arise because **Doctrine created fan-out**, or because the
> **project chose a runtime/resource** with particular semantics?

Doctrine owns hazards introduced by fan-out itself. The project/operator owns
semantic choices about resources the project runs or reaches. Doctrine still owes
**detection and warning** for project-owned resources when fan-out changes their
risk profile — because only Doctrine knows the fan-out is happening.

This yields **three Doctrine duties and one explicit non-duty**.

#### Duty 1 — make disjoint

Make framework-created resources disjoint *by construction*: worker identity,
worktree path, worker scratch, in-jail `/tmp`, redirected XDG cache/state,
allocated port windows, per-instance runtime names where an adapter exists.

#### Duty 2 — contain

Provide a write-containment floor where a supported backend exists. On the
Linux/bwrap Claude arm that backend is **SL-182** (consumed, §3; not re-specified
here).

#### Duty 3 — detect and warn

Surface likely fan-out collisions on project-owned surfaces: absolute host binds,
named/external volumes, hardcoded ports, shared mutable external URLs,
DB/Redis/S3 namespace reuse, privileged socket reachability, writes needing
`extra_rw`. Some detections are advisory; some are **fail-closed until granted**.

These detections matter **because the default jail leaves the network up**
(SL-182: `network = true` by default). A worker that cannot reach the network
cannot collide on a shared remote DB; the detect-and-warn duty exists precisely
in the gap that network-on leaves open. Network egress as a *hard wall* remains
out of scope — this is collision detection, not a firewall.

#### Non-duty — the project owns semantics

Whether a DB is intentionally shared, whether a branch gets its own schema,
whether a socket grant is acceptable, whether a service is one-per-worker or
shared, whether credentials are per-branch, whether test data is destructive,
whether a build step is trusted, whether a SaaS sandbox tolerates parallel
mutation — these are the project's. Doctrine makes the choice **explicit**, never
decides it by hidden convention.

### 2. Resource taxonomy by responsibility

The reusable artifact of this RFC is the matrix. Each row is classified by the
cut above; the default posture is the leading position, not yet canon.

| Resource / hazard | Doctrine duty | Project/operator duty | Default posture |
| --- | --- | --- | --- |
| Worktree identity | Create and track | Don't hand-edit runtime state | Framework-owned |
| Per-arming profile | Declare / provision / consume | Choose safe batch-wide profile | Framework-owned |
| Distinct concurrent sibling profiles (Claude arm) | Not currently guaranteed | Use serial / pi / future token | Out of scope for now |
| Bash write boundary | Enforce via active backend | Request explicit grants | SL-182 on Linux/bwrap |
| Edit/Write boundary | Pathcheck (`realpath ⊆ cwd`) | Request explicit grants | SL-182 |
| `.git` writes | Deny (hard-ro, not tunable) | Use orchestrator import funnel | Hard deny |
| `/tmp` | Private tmpfs where supported | Don't rely on host `/tmp` | Private by default |
| XDG cache/state | Redirect if enabled | Make tooling respect env | Private by default (future) |
| Host ports | Allocate or warn | Consume assigned ports | Registry recommended (future) |
| Absolute binds | Detect / warn | Decide intended sharing | Warn; grant if write |
| Named / external volumes | Detect / warn | Namespace if needed | Warn |
| Docker socket | Detect and grant-gate | Accept daemon risk | Deny unless granted |
| Postgres socket | Detect, grant-gate if privileged | Least-privilege service user | Strong warn / grant |
| nix-daemon socket | Treat as privileged peer | Decide build requirement | Explicit grant / documented exception |
| Shared remote DB/Redis/S3 | Detect when visible; document pattern | Namespace / provision | Warn |
| Secrets / env | Plumbing only | Own semantics | Materialise explicitly |
| Build trust | Contain if local; gate daemon socket | Own build semantics | Peer-class handling |

Two framings collapse most of the table:

1. **Project runtime resources** that may collide under fan-out (ports, volumes,
   shared stores).
2. **Durable socket-reachable peers** living outside the worker's mount namespace
   (Docker, Postgres, nix-daemon, buildkit).

Docker/Compose are **instances** of these two classes, never the organising
frame. A malicious *build instruction* is not a parallelism problem — it is
either contained (runs in the jail) or a privileged-peer problem (reaches a
daemon outside it). Fold buildkit/nix-daemon into peer reachability.

The **Nix store** is a deliberate exception worth naming: content-addressed,
shared, reproducible — a *strength* under fan-out, not a collision. But the
**nix-daemon socket** is still a privileged peer. Treat the store as shared
infrastructure and the daemon socket as a grant.

### 3. Relationship to SL-182 (consume, don't re-derive)

The parallelism model assumes a confinement floor supplied by the active backend.
For the Claude Linux/bwrap arm, that backend is **SL-182**, which graduates the
RSK-014 probe-h1 apparatus into installed Rust hook machinery. This RFC consumes
SL-182's invariants and does **not** restate its hook design.

> *Caveat — SL-182 is in `design` (RV-201, the second adversarial inquisition, is
> active at time of writing). The invariants below are cited as the reconciled
> floor; treat them as in-flight until SL-182 closes.*

Invariants this RFC inherits:

- **Per-arming policy, not per-agent.** The orchestrator cannot pre-key by
  `agent_id` (the Claude harness assigns it at spawn). The policy unit is a
  **fan-out arming**: a serial arming may effectively target one worker; a
  parallel arming declares **one batch-wide profile** all siblings share —
  intentional sharing, not a leak. Truly distinct concurrent sibling profiles
  require the pi/subprocess arm or a future per-spawn token. This RFC uses
  per-arming as the **current contract**, not a fantasy per-agent map.
- **`.git` is hard read-only and not a knob.** Loosening `.git/objects` would let
  a worker write arbitrary blobs into the shared store — defeating the jail. The
  consequence: **workers do not self-commit** on the confined Claude arm; the
  orchestrator imports a captured working-tree delta. Parallelism safety inherits
  that funnel rather than changing it. (`.git` posture, worker self-commit, and
  Path C are explicitly *not* re-opened here.)
- **Degradation is never silent.** A platform lacking the declared backend means
  *deny*, not unwrapped pass-through. Doctrine may still offer non-confinement
  utilities, but must not label such a run as carrying the same guarantee.

### 4. Policy model

One conceptual policy object per arming, provisioned to worktrees as the active
arm permits. **Today's reality (SL-182):** the orchestrator pre-declares intent
beside the `base` file; the `create-fork` hook provisions it to
`<main>/.doctrine/state/dispatch/jail/<worktree-name>.toml` (runtime state,
outside every worktree, ro to the worker); the `pretooluse` hook resolves it by
`cwd → basename`. The materialised schema is **minimal — `extra_rw` + `network`
only**. Absence ⇒ the strictest floor (rw worktree, ro everything else).

**Target shape (vision, subordinate to the floor above):**

```toml
[arming]
id = "2026-07-01T..."
mode = "parallel"          # or "serial"
profile = "default"

[confinement]
extra_rw = []              # SL-182 implements this today
network = true             # SL-182 implements this today
allow_sockets = []         # future: privileged-peer grants

[scratch]                  # future
tmp = "private"
xdg_cache_home = "private"
xdg_state_home = "private"

[ports]                    # future
strategy = "registry"
range_start = 43000
range_size = 1000
per_worker_window = 20

[warnings]                 # future
absolute_binds = "warn"
external_volumes = "warn"
privileged_sockets = "deny_unless_granted"
shared_external_resources = "warn"
```

The `[arming]`/`[scratch]`/`[ports]`/`[warnings]` tables are **not** the SL-182
file format; they are the direction later slices may grow toward, each landing
its own ADR-altitude decision at reconciliation. Nothing here asks SL-182 to
change its locked minimal schema. Invariant across all of it:

- The policy authority is the **orchestrator** (ADR-006 sole-writer). Workers
  never author their own policy. Policy state lives outside every
  worker-writable tree.
- Absence of optional policy ⇒ **conservative default**, never unconfined
  execution.
- `extra_rw` is footgun-validated: no root, no main-root ancestor, no `.git`;
  violations **deny**.

**Port allocation** (a pure fan-out collision, Duty 1) is the clearest near-term
extension. Recommended default: an **orchestrator-owned allocation registry**
outside every worktree — stateless hashing into a configured range is attractive,
but a registry gives explicit collision handling and clearer diagnostics. The
range, window size, allocator/service-port sharing, and surfacing mechanism are
slice-level, not decided here.

### 5. Capability levels (degraded mode)

Doctrine must never silently weaken a guarantee, and must separate **"parallelism
*utility* is available"** from **"parallelism *safety guarantee* is active."**
Proposed ladder:

| Level | Meaning |
| --- | --- |
| `none` | No fan-out safety guarantee; do not call it safe parallel execution |
| `disjointness-only` | Ports/scratch/names distinct, but no hard write containment |
| `contained-writes` | Write-containment floor active |
| `contained-writes-plus-detection` | Containment plus spawn-time hazard warnings |
| `full-declared-profile` | Containment, disjointness, warnings, grants, adapter support |

Expected current matrix:

| Platform / arm | Expected level |
| --- | --- |
| Linux Claude, SL-182 bwrap | `contained-writes` → later `contained-writes-plus-detection` |
| Linux pi subprocess bwrap | already confined; can adopt the broader policy later |
| macOS Claude before a backend exists | not equivalent; **do not** claim a containment floor |
| macOS with a future `sandbox-exec` backend | likely `contained-writes` once proven |
| No supported jail backend | refuse confined fan-out, or run **explicitly** degraded |

This ladder is the *fan-out safety* axis of the existing **ADR-011 per-harness
capability altitude**; it should ride that concept, not fork a parallel one.
Naming/placement is a downstream ADR concern.

> **macOS is wider-open than it looks.** IMP-045 scopes a `sandbox-exec`/Seatbelt
> backend for the **subprocess** jail seam (shared profile generation). The
> **Claude-arm** macOS equivalent — a PreToolUse hook wrapping `sandbox-exec`
> instead of `bwrap` — is **not** currently scoped by any backlog item. The
> matrix row above assumes work that does not yet exist; track it as a distinct
> follow-up, not a near-term given.

### 6. Non-goals

This RFC references but must not become: adversarial write-containment design
(RSK-014 / SL-182); reads and exfiltration; network egress as a hard wall;
Doctrine as a deployment/config/orchestration system; project-specific data
semantics; Docker/Compose as the organising model; automatic namespacing of
cloud resources; re-opening `.git` posture, worker self-commit, or Path C.

## Outcome

**Open.** This RFC takes the following positions as its leading vision; each
hardens into governance (ADR/standard) or implementation (slice) downstream, as
adversarial slice design locks it and reconciliation emits it.

1. Doctrine owns **identity-derived disjointness**.
2. Doctrine owns the **active write-containment floor** where a backend exists.
3. Doctrine owns **per-instance private scratch** as both jail allowance and
   collision prevention.
4. Doctrine owns **port allocation**, or at minimum hardcoded-port detection.
5. Doctrine owns **detection/warning** for project-owned fan-out hazards.
6. Projects own the **semantic correctness** of external mutable resources.
7. **Privileged socket reachability** is explicit-grant territory.
8. The current Claude-arm policy unit is **per-arming, not per-worker**, for
   parallel fan-out.
9. The model exposes **degraded capability levels** and never silently downgrades.
10. Docker/Compose/build daemons are **examples** of generic resource classes,
    not the organising frame.

**Follow-ups — ordinary slices/backlog under this model, not new RFCs:** port
allocator/registry; scratch + XDG redirection; spawn-time hazard scanner;
privileged-socket grant model; a Compose adapter *if actually needed*; macOS
`sandbox-exec` backend (subprocess via IMP-045, **plus** an unscoped Claude-arm
hook variant); pi/subprocess policy convergence; a future per-spawn token if
distinct concurrent Claude sibling profiles become necessary; Path C / IDE-024 if
the confined linked-worktree funnel cost becomes unacceptable.

SL-182 remains the slice for the bwrap Claude-arm confinement floor. The rest
lands as ordinary slices under this responsibility model.
