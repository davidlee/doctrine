# RFC-002: Consumption surfaces program: estimate, value, and scoring outward pipeline

## Context

RFC-001 establishes the thesis: doctrine's graph value is gated on **consumption
surfaces**, not on deeper internal modelling. The estimate/value authoring CLI
(SL-118) shipped the capture side; this RFC tracks the program to build the outward
surfaces that make that data matter.

**Why now.** SL-118 shipped `doctrine estimate set/clear` and `doctrine value
set/clear` CLI verbs. The facets (`[estimate]`, `[value]`, risk `[facet]`) exist in
the TOML model but nothing consumes them:
- No read path surfaces them (you can't *see* an item's estimate without opening its
  TOML)
- No scoring engine uses them to influence priority ordering (`survey`/`next`
  currently sort by raw inbound-reference count)
- No history tracking makes estimate drift observable (every `set` silently
  overwrites)

This is exactly the pattern RFC-001 names: the model exists, the capture exists,
but the **consumption surfaces** are missing. The cheapest, highest-leverage moves
are all outward.

Six items were surfaced during the 2026-06-21 session, scoped into slices, and
sequenced. This RFC is the program-level coordination artifact — it holds the
overall context, tracks progress, and maintains the big picture so neither context
drift nor parallel work loses coherence.

### Session origin

The items emerged from:
1. Re-reading RFC-001 and identifying concrete consumption-surface gaps
2. Noticing that IDE-013 (estimate/value history) and IMP-112 (show-path display)
   were created same day (2026-06-19) but unlinked and unconnected to the thesis
3. Identifying IMP-118 as the "real value" item — it makes estimate/value facets
   actually matter by feeding them into the priority scoring engine
4. Discovering gaps while dependency-mapping IMP-118: no risk facet CLI verb
   (IMP-132), tagging only on backlog items (IMP-134), CLI UX inconsistencies
   (IMP-133 / ISS-040)

A brief detour through ISS-008 (shared-target pollution flake) confirmed the
pattern: ISS-008 is a concrete motivating example of why estimate/value surfaces
matter for triage, but IMP-118 is the systemic fix.

### Stream C — The tentpole

| Item | Slice | What |
|------|-------|------|
| **IMP-118** | SL-133 | Multi-dimensional priority scoring for survey/next |

Consumes `[value]`, `[estimate]`, and risk `[facet]` from the authored items;
applies kind weights and tag coefficients from `doctrine.toml`; computes a
composite score via two-pass (base + consequence). Replaces the current raw
inbound-ref-count consequence with the weighted score. `after` SL-132 and SL-134.

## Environment

- **Bwrap jail**: `/workspace` is the repo root, `~/.cargo/doctrine-target-jail/`
  is the shared CARGO_TARGET_DIR — per-worktree target dirs for dispatch workers.
- **Gate**: `just check` (fast, root package only), `just gate` (full, includes
  cordage workspace). Both run clean before committing.
- **Build**: `cargo build` writes to `~/.cargo/doctrine-target-jail/debug/`;
  `./target/debug/doctrine` is STALE.
- **Lint**: `cargo clippy` — no `--all-targets` (would trigger `unwrap_used`/
  `expect_used` denials in tests). `just gate` runs plain clippy.
- **Test**: `cargo test` (root package), `cargo test --workspace` (incl. cordage).
- **Format**: `cargo fmt`
- **Conventions**: 2-space indent, conventional commits (`feat(SL-NNN): …`,
  `fix(SL-NNN): …`). Commit to `main`. No stashing. No branch discards.

## Risk

- **Scope creep.** IMP-118's body is ambitious (two-pass computation, kind weights,
  tag coefficients, consequence post-pass). The MVP could ship with value dimension
  only and defer risk dimension + tag coefficients to a follow-up.
- **Tagging dependency.** If IMP-134 is slow, IMP-118 can ship without tag
  coefficients (coefficients default to 1.0 / neutral when absent). Not a hard
  blocker.
- **IMP-112 has no design body.** The empty body is a hard prerequisite — a
  design-less slice is a red flag per project conventions. Fill the body first.
