# RFC-002: Consumption surfaces program: estimate, value, and scoring outward pipeline

## Context

RFC-001 establishes the thesis: doctrine's graph value is gated on **consumption
surfaces**, not on deeper internal modelling. The estimate/value authoring CLI
(SL-118) shipped the capture side; this RFC tracks the program to build the outward
surfaces that make that data matter.

**Why now.** SL-118 shipped `doctrine estimate set/clear` and `doctrine value
set/clear` CLI verbs. The facets (`[estimate]`, `[value]`, risk `[facet]`) exist in
the TOML model but nothing consumes them:
- No read path surfaces them (you can't *see* an item's estimate without opening its
  TOML)
- No scoring engine uses them to influence priority ordering (`survey`/`next`
  currently sort by raw inbound-reference count)
- No history tracking makes estimate drift observable (every `set` silently
  overwrites)

This is exactly the pattern RFC-001 names: the model exists, the capture exists,
but the **consumption surfaces** are missing. The cheapest, highest-leverage moves
are all outward.

Six items were surfaced during the 2026-06-21 session, scoped into slices, and
sequenced. This RFC is the program-level coordination artifact — it holds the
overall context, tracks progress, and maintains the big picture so neither context
drift nor parallel work loses coherence.

### Session origin

The items emerged from:
1. Re-reading RFC-001 and identifying concrete consumption-surface gaps
2. Noticing that IDE-013 (estimate/value history) and IMP-112 (show-path display)
   were created same day (2026-06-19) but unlinked and unconnected to the thesis
3. Identifying IMP-118 as the "real value" item — it makes estimate/value facets
   actually matter by feeding them into the priority scoring engine
4. Discovering gaps while dependency-mapping IMP-118: no risk facet CLI verb
   (IMP-132), tagging only on backlog items (IMP-134), CLI UX inconsistencies
   (IMP-133 / ISS-040)

A brief detour through ISS-008 (shared-target pollution flake) confirmed the
pattern: ISS-008 is a concrete motivating example of why estimate/value surfaces
matter for triage, but IMP-118 is the systemic fix.

## Next actions

### A (highest leverage) — wire tag coefficients into scoring

The formula in `base_score()` (src/priority/graph.rs:72) computes
`cfg.coefficients.value × v.value × kw / est_mid` but the ADR-015
`× Σtag_coeffs` term is **missing**. The config infrastructure and tag
storage exist — only the seam is absent.

**Seam to close:**
- Add `tags: Vec<String>` to `EntityFacets` (src/facet.rs)
- Return tags from `scan::read_facets` (or read via TOML table)
- Add `tags` field to `ScannedEntity` (src/catalog/scan.rs)
- Call `cfg.tag_coeff(tag)` + sum in `base_score()` and multiply into `value_dim`
- Tests: tags change score and ordering in `survey`/`next`

**Files:** `src/facet.rs`, `src/catalog/scan.rs`, `src/priority/graph.rs`, `src/priority/config.rs` (just ungating)

**Slice shape:** small — one or two phases. All dependencies done (SL-136, SL-133).

### B — seed tag_coefficients in doctrine.toml

Once the seam is live, add project-meaningful `[priority.tag_coefficients]` to
`doctrine.toml` and verify `survey` ordering reflects them. The config schema
(`BTreeMap<String, f64>`) already exists and clamps — this is data authoring,
not code.

### C — `doctrine priority` CLI verb

After coefficients are live in scoring, add a CLI surface to set/unset/show them
without hand-editing `doctrine.toml`. This completes the consumption-surface
loop for config: data fields exist, scoring consumes them, and a write verb
makes them editable.

```bash
doctrine priority show                        # display current [priority] config
doctrine priority set coefficient.value 2.0   # set a scalar coefficient
doctrine priority set coefficient.risk 1.5
doctrine priority set consequence.dep_coeff 0.8
doctrine priority set kind_weights.SL 2.0     # set a map entry
doctrine priority set tag_coefficients."area:priority" 3.0
doctrine priority unset kind_weights.SL       # remove entry, back to default
doctrine priority reset                       # clear whole section to defaults
```

**Design questions to decide before slicing:**
- Dotted path syntax for map entries (`.key` suffix) vs typed subcommands (`--kind SL --weight 2.0`)?
- Is `priority show` a verb, or does `config show --section priority` cover it?
- Should `set` validate against the clamp ranges (`COEFF_MAX`, `dep_coeff ∈ (0, 1]`) or let the load-time clamp silently correct?
- Should `unset` restore the field to its serde default or remove it from TOML entirely?
- Update workflow: does `doctrine priority set` require a regenerate (`doctrine boot` or `debug reload`) or is scoring always live-read?

**Slice shape:** medium — new subcommand module (`src/commands/priority.rs`),
TOML write leaf (reuse `toml_edit` pattern from `facet_write.rs`),
and the `priority show` display.

## Risk

- **Tag coefficients change all scores.** Once wired, any entity with tags
  will see a multiplier shift. If `tag_coefficients` defaults are absent
  (all 1.0), the term is neutral; but if the project seeds aggressive
  coefficients, ordering may shift non-obviously. Mitigation: land the
  code (A) before configuring meaningful values (B), so the delta is
  visible at each step.
- **`doctrine priority` CLI is TOML-editing.** It mutates `doctrine.toml`,
  which is a project-root config file. Corrupting it (e.g. concurrent write,
  encoding error) would break the project's boot and build. Mitigation: reuse
  the `toml_edit` pattern from `facet_write.rs` which is edit-preserving;
  validate before write; keep a single-writer lock (no concurrent priority
  CLI from dispatch workers).
- **IMP-021 is scope-creep adjacent.** It's a legitimate feature but
  orthogonal to the scoring-config track. Don't let it pull focus from
  closing the tag-coefficient seam.
- **IMP-147 is a genuine correctness gap.** A wrong-type value in `[priority]`
  today silently nukes the whole section. Small fix, easy to defer, but it's
  the kind of sharp edge that erodes trust in the config surface.
