# RFC-010: Skill improvement sweep

## Context

A looping sweep to improve the agent skills doctrine ships under
`plugins/doctrine/skills/` (30 canonical `SKILL.md` files; the
`doctrine-partner` and `doctrine-memory` plugin dirs are **symlinks** back into
the canonical domain per SPEC-010, so there is one source, not three). The
product anchor is **PRD-003 (Skills)**: skills exist so the agent in front of a
codebase actually *carries* doctrine's conventions instead of improvising —
"routes by instinct, skips the gates, rediscovers the process every session."
Related governance: ADR-005 (skills route, reference docs explain), PRD-007
(Boot & Governance), SPEC-010 (Skills distribution — canonical tree + embed).

Note: PRD-003 owns *distribution*, not skill *bodies*. This RFC judges bodies
against PRD-003's **intent** (skills must land and take hold), not against a
requirement it disclaims.

This RFC is the single durable notebook. It is the source of truth for what is
done — resume from the ledger, not from memory.

## Lenses

Every skill is judged against four lenses, in priority order:

1. **ADHERENCE** (highest value) — does the skill make the agent reliably DO the
   right thing? Hunt weak MUST/SHOULD, fuzzy routing, escape hatches, ambiguous
   gates, missing "do NOT" boundaries. A skill that reads nicely but gets skipped
   or misapplied is worse than a blunt one that lands.
2. **TOKEN EFFICIENCY** — SKILL.md is paid on every load. Hunt redundancy,
   restatement of boot/routing context already in the agent's context, prose that
   could be a table, examples that don't earn their tokens. Quantify (line/token
   counts) before/after any cut.
3. **OUTCOME QUALITY** — does following the skill produce the right artifact/state?
   Gaps, stale command shapes (verify against `doctrine <kind> --help`, don't
   trust memory), missing failure handling, drift from how the CLI actually works.
4. **COHERENCE** — cross-skill: duplication, inconsistent terminology, routing
   handoffs that don't line up, the same guidance phrased N different ways.

Disposition per skill: **SUGGESTION-ONLY** (concrete recommendation, quote
offending lines, expected effect) or **PATCH** (behaviour-preserving sharpening,
edited directly with before/after token delta). Anything that changes what a
skill is FOR is a suggestion, never a silent patch. Prefer fewer high-confidence
changes over churn.

## Cross-cutting themes

- **T1 — canonical source is `plugins/doctrine/skills/` only.** `doctrine-partner`
  and `doctrine-memory` skill dirs are symlinks (SPEC-010). Patches edit the
  canonical `doctrine/` copy; the symlinks follow. No duplication violation.
  Status: RESOLVED (informational; governs where every patch lands).
- **T2 — boot-freshen ritual duplicated across skills.** The "edited governance →
  run `doctrine boot` THEN `/clear`/restart; boot alone can't refresh an inlined
  prefix" ritual is restated verbatim-ish in `route` (§"Governance edited this
  session?", ~6 lines) and `canon` (lines 34-39, ~6 lines). The boot digest
  itself already carries the freshen rule. Candidate: collapse to a one-line
  pointer in each skill, or to a single reference doc, once the full skill set
  confirms how many carry it. Watch for more copies (preflight, design likely).
  Status: OPEN — tally all occurrences before deciding patch vs reference-doc.
- **T4 — conditional dispatch/worktree mechanics inline on hot-path skills.** Large
  blocks of mechanics that apply only to a *conditional* case are paid on every load
  of a frequently-used skill: `/execute` solo-isolation block (l.65-112, ~48 lines,
  in-tree is default) and `/close` dispatch-integrate block (l.62-108, ~47 lines,
  dispatched slices only). ~95 lines / ~800 tok loaded unconditionally for cases that
  fire a minority of the time. Candidate: factor the deep mechanics into the
  `/worktree` and `/dispatch` skills (or a reference doc) and leave a tight pointer +
  the load-bearing safety one-liners (never-squash, --integrate-only-here) inline.
  Caveat: the land/gc/integrate safety warnings must survive the move. NOT patched —
  cross-skill refactor needing author sign-off. Status: OPEN.
- **T5 — stale flag→positional CLI drift (PATTERN).** Two skills documented a `--flag`
  for an argument the CLI takes **positionally**, which would error verbatim:
  `/reconcile` `revision new --title "…"` (TITLE is positional) and `/backlog`
  `backlog tag <ID> --add <tag>` (adds are positional `[TAGS]...`). Both PATCHED in
  their rows. Root signal: skill examples drift when a CLI moves a flag to a
  positional arg. Recommendation in final summary: a CI check that extracts fenced
  `doctrine …` invocations from SKILL.md and dry-run/`--help`-validates them would
  catch this class. Status: both instances fixed; pattern noted for prevention.
