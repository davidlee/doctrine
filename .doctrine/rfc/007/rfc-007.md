# RFC-007: Actionability: legibility and correctness

## Context

RFC-001's thesis: doctrine's graph value is gated on **consumption surfaces**.
RFC-002 built the *data-in* half of one surface — the priority/scoring pipeline
(estimate, value, risk, tag/kind/risk coefficients, config CLI). That work is
largely done.

But the **Actionability Graph** — score-ordered "what should I pick up, and what's
blocking what" via `next` / `survey` — is the actual killer feature, and it does
not yet land. Authoring this RFC after a working session that diagnosed *why*.

The data-in layer is **not** the bottleneck. Two other things are.

### Problem 1 — legibility (both ends are dark)

The graph is **correct but invisible**, and the verbs are **visible but opaque**.

- The cordage actionability calc runs, but the only rendering of it is one *view*
  in the map web UI — a particular projection, not a faithful, inspectable
  representation. There is no textual surface that shows the true graph or how a
  coefficient feeds a score.
- `next` and `survey` — which *should* be among the most-reached-for verbs in the
  whole CLI — present as a large, flatly-ordered, opaque list. The order is real
  (score-driven) but its derivation is hidden, so it reads as arbitrary.
- `explain` already decomposes a single entity's order-key contributors +
  transitive blocker chain + consequence. **It is the truth-renderer.** But it is
  a separate verb buried in a wide CLI surface — discoverable only if you already
  know it exists. (Working-session evidence: a heavy user had never run it.)

These compound: you cannot tune coefficients you cannot observe, and you cannot
trust an order you cannot explain. Polishing data-in further does not move this.

### Problem 2 — correctness hole (epistemic records are graph-inert)

The priority engine partitions status binary: `Workable | Terminal`
(`priority/partition.rs`). Knowledge / governance records (PRD-010 / SPEC-019) are
never work, so SPEC-019 D7 parks every record state as `Terminal` — which also
means a record can **never block**.

So the single highest-value thing the graph could say —

> "blocked by an unanswered **question**, an unverified **assumption**, an active
> **constraint**, or a pending **decision**"

— it **structurally cannot express**. `next` is therefore not merely opaque, it is
*incomplete*: epistemic gating is invisible and the order is wrong, not just ugly.

IMP-047 (trinary actionability) is the keystone fix: a third status-class —
provisionally `Gating` — that **blocks dependents but is never itself workable**.
For records, no state is ever `Workable`: unsettled → `Gating`, settled →
`Terminal`. Plus a gating edge into the `dep` overlay so a record gates what it
affects (PRD-010 §3). This is a **model change** — PRD-011 / SPEC-001 canon move
first.

## Discussion

### The unifying lever

`explain` is already the per-entity truth-renderer. The feature lands when that
decomposition becomes the **native texture of `next` / `survey`** — not a separate
verb you must know to run. "Hard to feel how the coefficients feed in" is a direct
symptom of `explain` not being woven into the list, plus the absence of any
**what-if / trace** (perturb a coefficient → observe the reorder).

So legibility is *not* "add another verb." It is: bring the truth-renderer to where
attention already is, and make the calc inspectable as itself.

### Three workstreams, sequenced

**(1) Correct — make the graph true.** IMP-047 trinary actionability + epistemic
gating edges. Canon (PRD-011 / SPEC-001) moves first: a D-decision + requirement
for the third class and the `eligible` vs `blocks` split. Then
`priority/partition.rs` (third class + generalised cover invariant), `channels.rs`
(`eligible` = `Workable`; `blocks` = `Workable ∪ Gating`), `graph.rs` (overlay
allocation), new gating `RelationLabel`(s) + `RELATION_RULES` rows. Overlaps
IMP-033 (cross-kind dep capture), IMP-026 (actionability mask, SPEC-001 D6),
IMP-099 (graph admission across kinds) — shares the "non-backlog kinds in the dep
overlay" machinery. **Keystone — sequence first.**

**(2) Legible — make it feel right.** Fold `explain` into the consumption verbs:
`next` / `survey` rows carry their score decomposition inline or one keystroke away
(`--why`); sectioned / grouped order, not an undifferentiated flat list; a faithful
textual rendering of the actionability graph; and a coefficient **trace / what-if**
so tuning is observable (which term moved this item, what reorders if I bump a
coefficient). Fix the cordage `explain()` foreign-node bug (ISS-003) en route. New
backlog items to mint for the fold + what-if; IDE-019 (backlog verbose/explain
flag) is an adjacent small affordance.

**(3) Populate — make gating bite.** Epistemic-record authoring + lifecycle
engagement, so the gating edges (1) depends on actually exist and get settled.
Without records in the graph, trinary actionability has nothing to gate on. Spans
PRD-010 / SPEC-019 authoring ergonomics; relates to IMP-158 (survey held claims).

### Sequencing rationale

(1) before (2): legibility over a *wrong* order just renders the wrongness
faithfully. (1) and (3) are mutually reinforcing — (1) is the mechanism, (3) is the
data; (3) can start in parallel since record authoring is independent of the
partition change. (2) is the payoff and should follow (1)'s correctness.

### Out / deferred

- **Risk-modelling expansion** — acknowledged (risk facet likely wants richer
  modelling) but **not blocking**; defer.
- **`doctrine doctor`** (IMP-121, RFC-002 Stream E2) — a real corpus-health gate
  but **orthogonal** to this critical path; do not let it jump the queue.
- **Static graph file-export** (GraphML/Cypher/DOT-file) — RFC-002 already demoted
  this to on-demand; the navigable surface ships via map/web.

## Outcome

Open. The diagnosis is accepted (working session 2026-06-26); the sequencing above
is the proposed program. First concrete entry: route **IMP-047** to `/slice` as the
correctness keystone (canon-moves-first). This RFC umbrellas the three workstreams;
RFC-002 closes on its original scope (surface (d), data-in) once SL-138 lands.

Open questions:
- **OQ-1** — Naming the third status-class (`Gating` / `Ambient` / `Pending`).
- **OQ-2** — Gating edge authored outbound-from-record vs as dependent's
  `needs → record` (IMP-047 flags both; outbound fits the relation seam / ADR-004).
- **OQ-3** — `--why` inline vs a richer `explain`-as-default: how much decomposition
  belongs in the default `next` / `survey` render before it becomes noise.
- **OQ-4** — What-if scope: read-only trace of current contributions vs an actual
  coefficient-perturbation preview (the latter is bigger).
