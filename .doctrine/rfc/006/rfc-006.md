# RFC-006: Auto-merge non-FF trunk at integrate: reversing the FF-only posture

## Context

`dispatch sync --integrate` advances the trunk ref fast-forward-only. A non-FF
trunk — i.e. trunk advanced (by another agent) between the dispatch run's fork
point and integrate — is **refused** at plan time (`plan_trunk_row`,
`dispatch.rs:1984`: `is_ancestor` ensure! → "trunk moved; re-anchor required, not
auto-resolved"). This is not an accident of implementation: **ADR-012 D2/D4 fixes
it as doctrine.** D2/D4 state trunk projection is *"fast-forward-only … never auto
non-ff; conflict/moved-target ⇒ report, never auto-resolve."* The legitimacy of
the whole integration topology (ADR-012 D6) rests partly on *"unreviewed code
never touches trunk"* — an auto-composed merge commit on trunk is code+code the
human never reviewed as a unit.

The cost of FF-only is the **H2 shared-trunk-race** (RFC-005;
[[mem.pattern.dispatch.close-integrate-shared-trunk-race]]): under concurrent
multi-agent load on a shared `main`, integrate refuses whenever someone else
landed first, and the close driver **cannot self-unblock** — the operator must
`create --supersedes → re-admit → re-integrate`, repeatedly under churn. The
documented workaround is chaining `create→admit→integrate` in one shell to shrink
the window. Friction, not corruption — but persistent.

SL-157 makes integrate **checkout-independent** (advance via object-DB CAS, no
live worktree) while **preserving** FF-only unchanged. It deliberately keeps its
land seam *policy-free* (a CAS to the journaled `planned_new_oid`, agnostic to how
that oid was derived) so this RFC's capability can extend it **without rework**.

**The question this RFC opens:** should integrate gain the power to *auto-merge* a
non-FF trunk — `clean non-FF → merge-tree → commit-tree → CAS`, with an ephemeral
private worktree for genuine conflict surgery (IMP-127) — thereby letting the
agent self-unblock the shared-trunk-race? This **reverses ADR-012 D2/D4's FF-only
posture**, so it merits structured, externally-reviewed deliberation before any
Revision.

## Discussion

### The capability

Extend the SL-157 object-DB lander so a *real advance* (CAS `expected_old`
matches, but `planned` does not fast-forward trunk) is handled by merging rather
than refusing:

```
plan_trunk_row(non-FF case):
  mb = merge-base(trunk_tip, phase_tip)
  merge-tree --write-tree --merge-base=mb  trunk_tip  phase_tip
    Clean{tree} → commit_tree_merge(tree, trunk_tip, phase_tip)  → record as planned_new_oid
    Conflict    → park ephemeral private worktree for hand-resolve (IMP-127); refuse this run,
                  ingest the resolved commit on re-run as planned_new_oid
land(trunk, planned_new_oid, expected_old)  → unchanged CAS  (SL-157 seam)
```

**The merge must happen at PLAN time, not advance time.** `commit-tree` stamps
committer-date = now, so the merge commit oid is **non-deterministic across
replay**. Recording it once at plan (journaled `planned_new_oid`) preserves D4's
CAS-replay idempotency (`current == planned_new_oid` → verified no-op). This
exactly mirrors how the candidate flow already works: `candidate_create`
(`dispatch.rs:946`) computes the merge oid early and records it; integrate then
FF-lands the recorded oid. So the building blocks (`merge_tree` `git.rs:844`,
`commit_tree_merge` `git.rs:875`) exist and are proven — this is a lift, not
greenfield.

### What it buys

- **Self-unblocking shared-trunk-race.** The agent merges the concurrently-landed
  trunk instead of re-superseding. The H2 friction (RFC-005) largely dissolves.
- **DRY with candidate.** The same object-DB merge seam serves both flows.
- **Absorbs IMP-127.** Integrate gains a first-class hand-resolved-conflict ingest
  path (none exists today).

### What it costs

- **Reverses ADR-012 D2/D4.** "Never auto non-ff" becomes "auto-merge clean
  non-ff." This is a conscious governance reversal, not a mechanism restatement.
- **Trunk honesty.** An auto-composed merge commit lands code+code on trunk that
  no human reviewed *as a combination*. A clean 3-way merge has no textual
  conflict, but semantic interactions (two slices touching adjacent logic) are not
  caught by "clean". ADR-012 D6's legitimacy claim weakens.
- **Conflict-surgery surface.** The ephemeral private worktree (IMP-127) is
  net-new machinery: where it materialises, how it is isolated from the shared
  buffer, how the resolved commit is ingested under CAS, how a half-finished
  surgery is discarded. Real design + lifecycle UX cost.
- **CAS contract still holds, but is now load-bearing for a richer operation.** A
  moved target (someone advances trunk *again* during surgery) must still refuse,
  never force — the merge does not relax D4's moved-target refusal, only the
  FF requirement.

### Alternatives

1. **Keep FF-only (status quo + SL-157).** The race stays operator-resolved
   (supersede chain). Lowest risk to trunk honesty; persistent friction.
2. **Auto-merge clean non-FF only; never surgery.** A non-FF *conflict* still
   refuses (no IMP-127); only clean divergence auto-merges. Smaller surface,
   captures most of the ergonomic win (most concurrent lands are textually
   disjoint), defers the surgery-worktree cost. Partial D4 reversal.
3. **Full capability (clean merge + conflict surgery / IMP-127).** Maximum
   self-unblock; maximum surface and trunk-honesty cost.
4. **Gate the reversal behind config.** `intent_target = "trunk_automerge"` as an
   opt-in alt to `trunk_ff_only` (ADR-012 already has a config seam for
   `intent_target`). Preserves FF-only as the safe default; lets a
   high-churn/solo-trusted project opt in. Likely the most defensible shape — the
   reversal becomes a *project posture*, not a framework default.

## Outcome

**Open.** Authored as the home for B (the non-FF auto-merge capability split out
of SL-157). Routed here for external adversarial review *before* any ADR-012
Revision, because it reverses an accepted ADR. SL-157 ships the checkout-independent
lander (FF-only preserved) independently; its policy-free land seam is the
forward-compat anchor this capability extends without rework.

Decision needed: (a) reverse FF-only at all; (b) if so, alternative 2 vs 3 (clean
only vs + surgery); (c) framework default vs config-gated opt-in (alternative 4).
