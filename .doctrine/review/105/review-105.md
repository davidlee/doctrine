## Synthesis

All four phases ship cleanly against design.md and plan.toml. The implementation
matches the design exactly, with one refinement: `plan_append_system` uses
`symlink_metadata` to detect the symlink entry regardless of target existence,
handling the dangling-symlink window between install and first boot correctly.
This is a positive deviation.

**Standing risks:**
- The dangling symlink window is harmless — pi reads empty content, sentinel
  absent, LLM falls back to reading the `@` ref from AGENTS.md. The extension's
  `session_start` handler closes the window on first session start.
- The extension runs `doctrine boot` on every session start. Boot is cheap
  (write-if-changed) but touches disk. Acceptable; throttlable if needed.
- `execSync` timeout of 5000ms prevents a hung boot from blocking pi startup.
- The foreign-file ownership marker (`// Generated by ...`) is a heuristic — a
  user could write a file starting with that line. Acceptable tradeoff.

**Tradeoffs accepted:**
- Symlink over copy — follows live boot.md, no stale copy problem.
- `--agent codex` → `--agent pi` rename is a user-facing break. Intentional;
  the rename is mechanical and the old name was never released.
- Extension `stdio: "inherit"` — if pi captures stdout, boot output is invisible.
  Harmless (boot is write-if-changed, no useful stdout).
