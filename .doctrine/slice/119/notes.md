# Notes SL-119: Wire boot snapshot to pi via APPEND_SYSTEM.md

Durable per-slice scratchpad — tracked in git. The place to lift anything from a
disposable phase sheet (`.doctrine/state/.../phase-NN.md`) that must survive
`rm -rf` before the slice close-out audit harvests it.

## Implementation notes

### PHASE-01 — Sentinel, guard, rename

- `BOOT_SENTINEL` is a module-level const so the AGENTS.md guard never rots.
- `boot_guard_block()` assembles the full guard: `@` line + sentinel-aware prose.
- `upgrade_bare_boot_ref()` converts old-style bare `@` lines to the full guard block
  via string replacement on the exact matching line.
- `plan_boot_import` idempotency: matches on `@` line (unchanged), then checks
  `content.contains(BOOT_SENTINEL)` to distinguish old-style from new-style.
- Rename is mechanical: `s/Codex/Pi/g` across enum, parse, label, targets, resolve,
  install_refresh, all tests. `--agent codex` → `--agent pi`.

### PHASE-02 — APPEND_SYSTEM.md symlink

- `plan_append_system` uses `symlink_metadata` not `Path::exists()` to detect the
  symlink entry regardless of target existence. This correctly handles the
  dangling-symlink window between install creation and first boot.
- Symlink target is relative: `../../.doctrine/state/boot.md` — portable within
  the project tree.

### PHASE-03 — Pi extension

- Ownership marker: `// Generated by \`doctrine boot install\` — do not edit.`
  as first line. Used as heuristic for "did we generate this?"
- `execSync` timeout: 5000ms.
- Extension module format: CommonJS (`require`/`module.exports`), matching pi's
  extension format.

### PHASE-04 — Wiring

- `install_refresh` Pi arm: calls `install_append_system` then `install_pi_extension`.
- Claude arm: `AppendSystemOutcome::NotApplicable` and `ExtOutcome::NotApplicable`.
- `wire()` reporting: separate match blocks for append_system and extension
  outcomes, with merged arms for silent states.
