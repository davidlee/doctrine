# IMP-148: MCP memory tool inline help ŌĆö document response fields, expose validate, add examples

The four MCP memory tools (`memory_find`, `memory_retrieve`, `memory_show`,
`memory_list`) defined in `src/mcp_server/tools.rs` L195ŌĆō272 have solid safety
guidance and workflow chaining. But response fields are opaque to agents, two
parameter ambiguities exist, and `memory validate` isn't exposed.

**Context budget constraint.** MCP tool descriptions ride every turn ŌĆö
`tools/list` re-sends them on every response. So inline help must be terse;
reference material belongs in on-demand retrievals. This reassessment
(2026-06-22) weights each gap against its per-turn token cost.

### Shipped-memory authorship flow (essential ŌĆö agents will lose work without this)

The shipped corpus is NOT editable via the memory tools:
- Mutation verbs (`edit`, `tag`, `status`) correctly refuse with "shipped/ is
  read-only (regenerated by 'doctrine memory sync')".
- Read verbs (`show`, `verify`) fail with "memory not found" for shipped-only
  uids ŌĆö `resolve_show` only looks in `items/`, not `shipped/`.

Editing `.doctrine/memory/shipped/` directly is EPHEMERAL ŌĆö overwritten on
next `doctrine memory sync` (or `doctrine claude install`, which calls it).

The canonical authorship flow mirrors skills:
1. Edit `memory/<key>/memory.md` (RustEmbed source, `src/corpus.rs` L44)
2. `touch src/corpus.rs && cargo build` (force re-embed ŌĆö RustEmbed has no rerun-if-changed)
3. `doctrine memory sync` (materialize into `.doctrine/memory/shipped/`)

Without this documented, an agent discovering a stale shipped fact will either
(a) edit `shipped/` directly and lose it on next install, or (b) hit
tool-level "not found" errors and give up.

## Gap 8: `resolve_show` doesn't fall back to `shipped/` ŌĆö read verbs fail for shipped memories

`resolve_show` (`src/memory.rs` L2110) resolves only against
`MEMORY_ITEMS_DIR` (`.doctrine/memory/items/`). It has no fallback to
`shipped/`, unlike `collect_all` and `read_body` which both union items +
shipped. Consequence:

| Verb | Works for shipped? | Error |
|---|---|---|
| `find` / `retrieve` / `list` | Ō£ō | (uses `collect_all`) |
| `show` (CLI + MCP) | Ō£Ś | "memory not found" |
| `verify` | Ō£Ś | "memory not found" |
| `edit` / `tag` / `status` | Ō£Ś (correctly guarded) | "shipped/ is read-only" |

The `edit`-family verbs use `resolve_memory_toml_path` (L2309) which DOES
check `shipped/` as a fallback diagnostic ŌĆö they give the right error.
`resolve_show` needs the same pattern: try `items/`, fall back to `shipped/`.

This blocks `memory_show` from rendering shipped memories (both CLI and MCP).
`verify` on shipped memories is arguably correct to reject (no git anchor
means nothing to verify against), but `show` is broken.

**Fix**: add `shipped/` fallback to `resolve_show`, mirroring
`resolve_memory_toml_path`. Shipped memories are read-only so the returned
`dir` path must not be used for writes ŌĆö but `run_show` and `run_verify`
already only read the toml/body (verify writes via `stamp_verification` which
would fail on a shipped path, but verify on shipped is semantically wrong
anyway).
